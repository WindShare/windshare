- 分离密钥保持可选，默认仍是一条完整能力链接；起传和 ICE 继续受用户手势约束。
- 双模块发布先标记 `core/vX.Y.Z`，再升级根模块依赖并标记 `vX.Y.Z`；根模块禁用
  `replace`，`GOWORK=off` 构建始终是硬门禁。

## 3. M2 — 桌面 GUI 与签名套件

### 3.1 完成定义

M2 完成时，Windows 用户可以从资源管理器或桌面应用选择文件/文件夹，立即得到分享链接，
管理在线分享，也可以粘贴链接接收并续传。CLI 与 GUI 共享同一 engine。新分享使用签名套件
`0x02`，持链者即使掌握 `readSecret`，也不能伪造可被其他接收端接受的清单或块。

M2 不包含移动端、离线寄存、账户、链接撤销/次数限制、多中转同时注册、TURN、padding、
分片清单或公网正式运营。签名内容的原样重放仍是已知残余，由 M4 的签名 `expiresAt` 关闭。

### 3.2 签名套件 suite `0x02`

- fragment 为 `0x02 ‖ readSecret(16B) ‖ pkHash(16B)`；`pkHash = SHA-256(pubkey)[:16]`。
- 每次分享生成独立 Ed25519 密钥对。私钥只存在发送端，且不得从 `readSecret` 派生。
- 清单内嵌公钥并签名；接收端先校验 `pkHash`，再验证清单签名。
- 每个块签名绑定明文哈希、全局块号和本次分享的 manifest tag。只签清单不能关闭块替换漏洞。
- 块在第一次请求时惰性加密并签名，随后按字节缓存复用；生成链接仍不得读取文件内容。
- 构造 API 按发送者/接收者角色拆分 sealer 与 opener，禁止用布尔开关组合权限。
- 新分享默认使用 `0x02`。项目处于 pre-v1.0，不为 `0x01` 增加迁移旗标或兼容分支；仅在
  测试向量确有价值时保留最小解析夹具。

### 3.3 engine 与桌面边界

- `engine` 是分享、接收、续传、连接编排和生命周期的唯一应用层入口；CLI 与 Flutter 只是适配器。
- 续传模型从 CLI 私有实现下沉为共享领域能力，持久化通过可注入接口实现，不能让 GUI 再造 journal。
- FFI 只导出 `Start/Stop`。`Start` 返回 loopback 随机端口和每次启动随机 bearer token；token
  不落盘、不进入环境变量，认证失败立即断连。
- 本地 WebSocket RPC 使用带 `id` 的 JSON 请求/响应和事件推送；进度事件节流。文件数据面不经过
  FFI 或 RPC，避免复制大块数据和扩大内存所有权边界。
- engine 负责取消、正常关闭、错误分类和敏感信息脱敏；时钟、随机源、存储和网络均可注入测试。
- Windows shell 集成只传递用户选择并激活应用，不在 Explorer 扩展进程内启动分享或持有密钥。

### 3.5 M2 验收门

以下条件必须同时成立：

1. 持链攻击测试证明：攻击者拥有 `readSecret` 并能重算 AEAD，仍不能在原链接下伪造清单或块；
   截获并原样重放真实签名内容被明确记录为 M4 前残余，而不是误报为已修复。
2. 出链接路径只做 `stat`；块签名开销有新基线，且无未经批准的产品策略或分块常量改动。
3. CLI、Windows GUI 与浏览器完成 relay、P2P、回退、热切换、选择性下载、断线续传和并发接收回归。
4. `--relay-only` 与 GUI 同名选项都阻止 offer、answer、candidate 和 ICE 收集，同时 relay 下载可用。
5. 资源管理器单选/多选、应用已运行/未运行、安装/卸载四类 Windows 路径均有自动化或可复现实测。
6. `make ci` 全绿；Flutter analyze/test、Windows GUI E2E 和安装包 smoke test 已进入 CI，而非仅本机通过。
7. core 总覆盖率 ≥90%、根模块总覆盖率 ≥80%、每包 ≥70%；新增 Dart 领域/RPC 代码有单元测试。
8. `docs/acceptance/M2.md` 作为 tracked 的精简证据索引入库，链接到 CI、浏览器矩阵、性能基线和
   安装包校验；原始日志留在 CI artifact，不塞回本计划。

## 4. M3–M5 边界

### M3 — Android/iOS

- 复用 M2 engine 与同一 Dart RPC 客户端，仅 bootstrap 分平台：桌面 c-shared，移动端 gomobile。
- 接入系统分享面板、文件选择器、后台/前台切换、移动网络与存储权限；iOS 后台限制必须以真实设备验证。
- 密码档 suite 仍是可选决策。只有“可口述/可记忆”价值经用户验证成立才引入 Argon2id；不能以其
  替代 128-bit 分离密钥，也不能降低抗离线爆破参数。
- 验收门：Android/iOS 各至少一台真实设备完成分享、接收、暂停恢复、进程切换和互操作测试。

### M4 — 离线模式

- 中转支持有期限的密文寄存、账户/API Key、配额、保留策略与明确计费边界。
- 实装独立 `writeSecret` 所有权证明；接收能力不能获得修改、撤销或续期权限。
- 支持过期、撤销和下载次数限制；`expiresAt` 必须进入签名清单，关闭真实签名内容的原样复活。
- 验收门：发送端离线后可按策略下载；越权续期/撤销、过期重放、配额竞态和清理失败均有对抗测试。

### M5 — 生产扩展与上线

- 多中转采用“链接即路由、节点零共享”：发送端就近注册，接收端并行赛跑，resume token 每节点独立。
- 区域内 L7 按 WS 路径中的 shareId 一致性哈希；支持 drain、容量指标和零停机重注册/续传。
- 评估多 PeerConnection、TURN、padding、请求隐藏、逐块哈希/去重与百万级条目的分片清单；没有明确
  威胁或容量收益的功能不进入协议。
- `windshare.top` 上线前必须完成举报/禁用 shareId、速率与配额、ToS、监控告警和 egress 成本模型。
- 验收门：容量、故障切换、安全、滥用响应和成本预算都有可复现演练，才能声明 production-ready。

## 5. 持续门禁与维护规则

- 日常入口为 `make ci`；另按根 `AGENTS.md` 保持 gopls、lint、SLOC、Web 与本地 OS-network 覆盖门禁。
- 协议变化必须同一变更内更新 `协议规范.md`、`威胁模型.md`、Go↔TS 向量和跨实现测试。
- 新 UI 不得直接依赖 CLI internal 包；重复业务流程视为 engine 边界缺失，优先重构根因。
- 本文不记录逐次评审对话和版本流水账。完成一个里程碑后，只保留一行结果和 tracked 验收索引，
  详细材料归档；下一里程碑才展开为任务表。
- 状态只能在验收门全部满足后改为 `✅`，不能用“代码已合并”替代“端到端已验收”。
