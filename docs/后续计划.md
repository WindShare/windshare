M1 已交付 CLI 分享/接收、浏览器接收、中转回退、WebRTC、热切换、选择性下载、续传、
路径安全、并发双接收端与 Go↔TS 黄金向量。后续实现不得破坏以下基线：

- 出链接只验证所选根，不枚举后代，不预读内容或预哈希。
- 中转不持有内容密钥，不获取文件名或明文内容。
- catalog/content 协议只依赖 `FrameChannel`；传输适配器不拥有身份、调度或 UI 策略。
- `core` 不依赖根模块；CLI、各桌面端和移动端必须复用同一应用能力，不能复制业务流程。
- 分离密钥保持可选，默认仍是一条完整能力链接；起传和 ICE 继续受用户手势约束。
- 双模块发布先标记 `core/vX.Y.Z`，再升级根模块依赖并标记 `vX.Y.Z`；根模块禁用
  `replace`，`GOWORK=off` 构建始终是硬门禁。

## 3. M2 — Windows 桌面

### 3.1 完成定义

M2 完成时，Windows 用户可以从资源管理器或桌面应用选择文件/文件夹，立即得到分享链接，
管理在线分享，也可以粘贴链接接收并续传。CLI 与 GUI 共享同一 engine，并直接复用 R8 已验收的
descriptor、catalog、ProtocolSession、file-local content、OutputSession checkpoint 和 suite `0x02`，不得在 UI 重定义协议。

M2 不包含 macOS/Linux 客户端、移动端、离线寄存、账户、链接撤销/次数限制、多中转同时注册、
TURN、padding 或公网正式运营。
签名内容的原样重放仍是已知残余，由 M4 的签名 `expiresAt` 关闭。

### 3.2 engine 与桌面边界

- `engine` 是分享、接收、续传、连接编排和生命周期的唯一应用层入口；CLI 与 Flutter 只是适配器。
- M2 在线分享随 engine 进程存活；明确 `Stop` 或退出必须拒绝新 join、终止活动下载并注销 relay，
  只有意外断线可进入具名恢复宽限。
- OutputSession 统一恢复验证和 checkpoint store；CLI/GUI 不得自行推断 durable ranges 或另造 journal。
- FFI 只导出 `Start/Stop`。`Start` 返回 loopback 随机端口和每次启动随机 bearer token；token
  不落盘、不进入环境变量，认证失败立即断连。
- 本地 WebSocket RPC 使用带 `id` 的 JSON 请求/响应和事件推送；进度事件节流。文件数据面不经过
  FFI 或 RPC，避免复制大块数据和扩大内存所有权边界。
- engine 负责取消、正常关闭、错误分类和敏感信息脱敏；时钟、随机源、存储和网络均可注入测试。
- engine/RPC 契约保持平台中立；路径、shell 激活、打包和凭据存储由各平台适配器负责，不能把
  Windows 语义泄漏进共享领域模型。
- Windows shell 集成只传递用户选择并激活应用，不在 Explorer 扩展进程内启动分享或持有密钥。

### 3.4 M2 验收门

以下条件必须同时成立：

1. 持链攻击回归证明：攻击者拥有 `readSecret` 仍不能伪造 descriptor、page、block、error 或 terminal，
   也不能在诚实 relay 注册发送端路由；真实签名内容的过期重放明确记录为 M4 前残余。
2. 出链接路径不枚举后代或读取内容；descriptor/catalog/block 开销保持 R8 基线。
3. CLI、Windows GUI 与浏览器完成 relay、P2P、回退、热切换、选择性下载、断线续传和并发接收回归。
   明确停止/退出另须证明活动下载被终止、路由不可恢复且资源全部清理。
4. `--relay-only` 与 GUI 同名选项都阻止 offer、answer、candidate 和 ICE 收集，同时 relay 下载可用。
5. 资源管理器单选/多选、应用已运行/未运行、安装/卸载四类 Windows 路径均有自动化或可复现实测。
6. `make ci` 全绿；Flutter analyze/test、Windows GUI E2E 和安装包 smoke test 已进入 CI，而非仅本机通过。
7. core 总覆盖率 ≥90%、根模块总覆盖率 ≥80%、每包 ≥70%；新增 Dart 领域/RPC 代码有单元测试。
8. `docs/acceptance/M2.md` 作为 tracked 的精简证据索引入库，链接到 CI、浏览器矩阵、性能基线和
   安装包校验；原始日志留在 CI artifact，不塞回本计划。

## 4. M3–M5 边界

### M3 — 跨平台客户端

- M3 分为桌面移植与移动端两条可并行交付线；都复用 M2 engine 和同一 Dart RPC 客户端，仅 bootstrap
  分平台：桌面使用 c-shared，移动端使用 gomobile。
- macOS/Linux 桌面端复用 Windows 已验收的分享、接收和续传流程；平台适配器只负责文件选择、
  应用激活、系统集成与打包，不持有业务规则或内容密钥。
- macOS 覆盖 Finder/系统分享入口、签名、公证和沙箱文件访问；Linux 覆盖 XDG portal、主流桌面环境
  集成及明确的软件包/发行版支持矩阵，不能笼统宣称所有 Linux 环境均受支持。
- Android/iOS 接入系统分享面板、文件选择器、后台/前台切换、移动网络与存储权限；iOS 后台限制
  必须以真实设备验证。
- 密码档 suite 仍是可选决策。只有“可口述/可记忆”价值经用户验证成立才引入 Argon2id；不能以其
  替代 128-bit 分离密钥，也不能降低抗离线爆破参数。
- 验收门：macOS/Linux 的声明支持矩阵均完成安装、卸载、分享、接收、续传和 CLI/浏览器互操作；
  Android/iOS 各至少一台真实设备完成分享、接收、暂停恢复、进程切换和互操作测试。

### M4 — 离线模式

- 中转支持有期限的密文寄存、账户/API Key、配额、保留策略与明确计费边界。
- 实装独立 `writeSecret` 所有权证明；接收能力不能获得修改、撤销或续期权限。
- 生命周期为 `LiveOnly → OfflineUploading → OfflineCommitted`。上传期间链接继续由在线发送端服务；
  未提交对象不对 receiver 可见，失败或退出会取消上传、清理 staging 并终止在线分享。
- 上传必须物化 transport-neutral 的 sender-signed descriptor、CatalogPage、revision、block 及签名 inventory；
  寄存服务不能取得解密密钥或 sender 私钥，也不保存 ProtocolSession/OperationID/Lane envelope。
- 在线下载与上传复用同一 CatalogStore、RevisionStore 与逻辑块身份；inventory 固定上传时物化的具体
  密封记录，无需与在线服务过的记录逐字节相同。commit 后切到寄存服务不得改变 revision 身份或废弃
  receiver 已验证的 DurableRanges。
- sender-signed `OfflineCommit` 绑定 `ShareInstance + inventory root + expiresAt + delivery identity`；receiver
  分别验证 delivery session 与 sender 对象，寄存服务的动态错误不得冒充 sender 内容或控制消息。
- 只有服务器验证 inventory 完整并返回原子 commit ACK 后才进入 `OfflineCommitted`；此后发送端退出不影响
  现有或后续下载。源文件在上传中漂移则 commit 失败，发送端不得创建内容临时副本或完整复制。
- 支持过期、撤销和下载次数限制；`expiresAt` 必须进入签名 descriptor，并约束 page/block，关闭真实签名内容的原样复活。
- 验收门：上传中同一链接可在线下载，commit 前退出会失效，commit 后退出及活动下载可切到寄存服务；
  越权续期/撤销、源漂移、部分提交、过期重放、配额竞态和清理失败均有对抗测试。

### M5 — 生产扩展与上线

- 多中转采用“链接即路由、节点零共享”：各节点以同一 `pkHash` 验证注册 challenge proof，resume token 每节点独立。
- 接收端竞速以 `ShareInstance + pkHash` 锚定 descriptor；LaneSet 可按策略启用多 relay/P2P 数据通道。
- 区域内 L7 按 WS 路径中的 shareId 一致性哈希；支持 drain、容量指标和零停机重注册/续传。
- 评估 TURN、padding、请求隐藏及逐块去重；没有明确威胁或容量收益的功能不进入协议。
- `windshare.top` 上线前必须完成举报/禁用 shareId、速率与配额、ToS、监控告警和 egress 成本模型。
- 验收门：容量、故障切换、安全、滥用响应和成本预算都有可复现演练，才能声明 production-ready。

## 5. 持续门禁与维护规则

- 日常入口为 `make ci`；另按根 `AGENTS.md` 保持 gopls、lint、SLOC、Web 与本地 OS-network 覆盖门禁。
- 协议变化必须同一变更内更新 `协议规范.md`、`威胁模型.md`、Go↔TS 向量和跨实现测试。
- 新 UI 不得直接依赖 CLI internal 包；重复业务流程视为 engine 边界缺失，优先重构根因。
- 本文不记录逐次评审对话和版本流水账。完成一个里程碑后，只保留一行结果和 tracked 验收索引，
  详细材料归档；下一里程碑才展开为任务表。
- 状态只能在验收门全部满足后改为 `✅`，不能用“代码已合并”替代“端到端已验收”。
