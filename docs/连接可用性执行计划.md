## 1. 完成边界

### 1.1 目标

- 公共 STUN 单点失效不影响 WebRTC 直连候选发现。
- 对称 NAT、UDP 受限等场景可以按需升级到短效凭据 TURN。
- 现有应用 relay 与用户自部署 relay 都可同时发布 DirectWSS/TunnelWSS 入口，并与 WebRTC
  direct/TURN 路径并存；任一入口或节点失效后仍能恢复同一传输。
- 公共运营首版允许匿名分享，同时用能力证明、并发/速率边界和 Cloudflare 账号预算
  限制付费 TURN 的滥用与账单风险。
- 所有路径继续复用同一 FrameChannel/会话调度，不把供应商、回退或费用策略下沉到 core。
- CLI、桌面和后续移动端复用 Go engine；Web 用 TypeScript 实现相同的协议与状态语义，
  并由 Go↔TypeScript 黄金向量锁步。

### 1.2 非目标

- 不让 cloudflared 承载 STUN、TURN、ICE UDP 或 WebRTC DataChannel。
- 不把多个 Cloudflare 账号描述成供应商级网络冗余；它们共享 Cloudflare API、DNS 和
  Anycast TURN 数据面。
- 不接受第三方 relay 任意下发 TURN 域名、内网地址或未审核端口。
- 不由 WindShare 创建 Cloudflare Tunnel、修改 DNS，或持有用于 Tunnel/DNS 管理的账号权限；
  TURN 签发与预算只使用各自最小权限的 secret。
- 不承诺 Cloudflare TURN 账单逐字节零超额；官方 Analytics 存在统计延迟，本文采用预算熔断。
- 不保留 v1/v2 双栈或协议兼容分支。
- 不改变“出链接不预读、不预哈希内容”的产品不变量。

### 1.3 专项完成判据

| 维度 | 必须达到的结果 |
|---|---|
| STUN 冗余 | 任一默认 STUN 不可达时仍能发现 server-reflexive candidate；两者都失败时应用 relay 不受影响 |
| TURN | 强制直连失败的确定性拓扑中，Go 与真实浏览器可经 TURN 完成逐字节一致的传输 |
| 多 relay | 任一活动入口或节点失效，仍可从其他路由验证同一 ShareDescriptor/ShareInstance 并完成传输 |
| 降级 | TURN API、账号预算、Tunnel 或全部 PeerConnection 失败都不能使健康的应用 relay 传输失败 |
| 费用 | Cloudflare 观测用量达到账号预算 80% 后，一个统计轮询周期内停止该账号的新签发与刷新 |
| 安全 | 只有持完整能力链接且经发送端批准的会话可申请 TURN；WindShare 客户端与 relay 不把长期或短效凭据写入链接、日志、trace 或持久化存储 |
| 隐私 | Application relay only 模式产生零 SDP、零 ICE candidate、零 STUN/TURN DNS 与网络请求 |
| 质量 | Go 双模块覆盖率门、Go↔TypeScript 向量、race、network、Web、浏览器和完整 make ci 全部通过 |

此处的持久化约束只覆盖 WindShare 控制的客户端与 relay；Cloudflare 作为 provider 会按其
服务行为保留 credential 标识和用量元数据，隐私与运维文档必须明确这一边界。

## 2. 当前基线与根因

- [connectivity/pion.go](../connectivity/pion.go) 与
  [web/src/connectivity/peer-offer.ts](../web/src/connectivity/peer-offer.ts)
  分别硬编码单一 Google STUN；两端构造器虽可注入完整 ICE 配置，生产入口仍冻结为默认值。
- [cmd/windshare/internal/cli/share.go](../cmd/windshare/internal/cli/share.go) 和
  [cmd/windshare/internal/cli/get.go](../cmd/windshare/internal/cli/get.go)
  在 CLI 边界直接创建默认 Pion 工厂；Web 默认 runtime 同样持有模块级静态工厂。
- [core/link/link.go](../core/link/link.go) 已能保存多个 relay hint，但 CLI 与 Web 只消费第一个；
  发送端也只向一个节点注册。
- 当前一个接收端固定建模为“一条应用 relay + 一条 PeerConnection”。该模型不能表达多节点、
  同节点多入口、动态 TURN 升级或旧凭据到期前的 PeerConnection 替换。
- [relay/httpapi/handler.go](../relay/httpapi/handler.go) 已预留 SourceIdentity 注入点，但默认只信
  TCP peer，且接口不能返回身份校验错误；经本机 cloudflared 后来源会坍缩成同一个 loopback 地址。
- [relay/protocol/config.go](../relay/protocol/config.go) 的公开配置只通告版本和静态限额，不适合
  承载可计费短效 TURN 凭据。
- 两端当前 candidate 上限均为 64。增加 ICE 服务前必须用真实多网卡、IPv4/IPv6 和
  UDP/TCP/TLS 组合测量，不能因功能扩展直接取消安全上限。

根因不是缺少几个 URL 或旗标，而是缺少分层的部署配置/公开路由/客户端策略、带代际的
Peer 生命周期、多路由所有权和可测试的凭据/预算策略。

## 3. 已确认的产品与架构决策

| ID | 决策 |
|---|---|
| D1 | 首版即具备公共运营能力；发送端暂不要求登录或 API Key，账户化授权后续增加 |
| D2 | 每个 Cloudflare 账号配置月度 egress 预算；达到 80% 后停止该账号的新签发和刷新 |
| D3 | 多账号按剩余可用预算加权分配；每个 peer attempt 只激活一个账号 |
| D4 | 最新观测达到 100% 或管理员紧急停用时触发活动凭据撤销；各节点尽力撤销内存中仍可定位的 lease，其余由短 TTL 收敛；正常达到 80% 不强断 |
| D5 | TURN 签发前执行 readSecret 派生的单次能力证明，并由已注册发送端批准 |
| D6 | 默认分层升级：直连从 0 秒开始，3 秒未 Open 或 ICE 失败时申请 TURN；非小分享最迟约 8 秒启用应用 relay |
| D7 | 小分享必须已完成所选子树发现，且文件数 `<30`、总大小 `<8 MiB`；证明为小后立即启用应用 relay，未知总量按非小分享计时，明确失败可跳过等待 |
| D8 | TURN 凭据默认 TTL 为 2 小时，在生命周期约 75% 时带抖动 make-before-break 刷新 |
| D9 | 默认每个分享最多 16 个并发 TURN 会话；更多接收者仍可使用直连或应用 relay |
| D10 | 首版客户端只接受严格验证的 Cloudflare STUN/TURN 主机、端口和 transport 参数 |
| D11 | 默认使用 Google + Cloudflare 两个公共 STUN；Application relay only 模式全程零 ICE |
| D12 | cloudflared 由用户安装；WindShare 接收 Tunnel Token 和公网 URL，启动并监督已安装的 connector |
| D13 | WindShare 不创建 Tunnel、不修改 DNS；GUI 用系统凭据库，服务器用 token file/部署 secret |
| D14 | 采用 RelaySet → RelayNode → IngressEndpoint；现有/用户中转的 DirectWSS、TunnelWSS 入口可与 WebRTC direct/TURN 路径同时存在 |
| D15 | 至少一个 RelayNode 注册成功即发布链接，其余节点后台补齐并报告 degraded redundancy |
| D16 | receiver 以 250 ms 级联、最多三个在途拨号竞速；稳定后只保留一个活动控制 relay，数据 lane 是否并存由 LaneSet 策略决定 |
| D17 | 活动路由失效后重新竞速；发送端发布的 RelayNodeID 去重同节点入口，receiverInstanceId 归并竞速窗口中的重复 join |
| D18 | 线协议直接升级 v2，offer、answer、candidate 与动态配置均绑定 attemptId；不保留 v1 兼容 |
| D19 | connectivity 只提供 FrameChannel；core/session 的 role-neutral pump/writer 独占收发，role router 分发 |

这些决策是本提案的内部基线。负责人接受整份提案并同步 [后续计划.md](后续计划.md) 后，
才成为实施承诺。

## 4. 目标架构

    连接编排模型（Go 与 TypeScript 各自实现）
    └── LogicalReceiverConnectivity
        ├── RelaySet
        │   ├── RelayNode A
        │   │   ├── DirectWSS ingress
        │   │   └── TunnelWSS ingress
        │   └── RelayNode B
        │       └── DirectWSS ingress
        └── PeerSession
            ├── direct attempt
            ├── TURN attempt
            └── replacement attempt

### 4.1 模块边界

| 模块 | 责任 |
|---|---|
| core/session | ProtocolSession 身份、role-neutral pump/router/writer、reassembly、终局和 FrameChannel 合同 |
| core/transfer | BlockBroker、LaneSet、块调度、取消和 OutputSession checkpoint 合同 |
| engine | 分享/接收应用入口，拥有 ClientConnectivityPolicy 和跨 CLI/GUI/移动端的生命周期 |
| connectivity | RelaySet、PeerSession、FrameChannel 供给、路径升级、恢复、观测与故障隔离 |
| transport/relay | 单一 relay 连接和 FrameChannel 适配，不拥有多节点选择策略 |
| transport/webrtc | 单一 PeerConnection/DataChannel 适配，不拥有供应商和升级策略 |
| relay/server | 可信部署配置、可测试服务组装、listener、来源身份与 Tunnel connector 监督 |
| relay/turnauth | 通用 issuer、Cloudflare adapter、账号池、预算、撤销和凭据敏感类型 |
| relay/protocol | v2 控制消息、大小边界和 Go 线类型 |
| web/src/connectivity | 独立实现 Go 的公开投影、attempt 和路径升级语义，不接收部署私有配置 |

### 4.2 核心领域结构

**DeploymentConnectivityProfile（仅可信部署侧）**

- RelayNode 的发送端 DialURL、listener/source identity policy、TURN/Analytics/Tunnel secret
  reference、账号预算和 provider alias。
- 不序列化到能力链接、浏览器配置或公开 API，只生成下面三个最小投影。

**PublishedRelayRoutes（能力链接中的公开投影）**

- RelayNodeID、对接收端发布的 WSS IngressEndpoint、优先级和入口类型。
- 只包含路由所需的非秘密字段；RelayNodeID 由 DeploymentConnectivityProfile 分配并随发送端
  生成的 PublishedRelayRoutes 发布，不能接受 endpoint 自报。

**ClientConnectivityPolicy（Go/TypeScript 同语义、各自实现）**

- Mode：Adaptive 或 ApplicationRelayOnly；后者在 UI 和现有路线图中显示为 relay-only。
- 默认 STUN、TURN 升级条件、PathEscalationPolicy、刷新与 candidate 边界。

**PeerConfiguration（每端、每 attempt 的短效投影）**

- attemptId、严格验证的 ICE URL、username、credential 和 expiresAt。
- 仅在能力证明与发送端批准后下发，不进入链接或持久化；Go↔TypeScript 向量锁定其公开编码。

**RelayNode**

- DeploymentConnectivityProfile 分配的稳定 RelayNodeID 和仅投影给发送端使用的 DialURL。
- 一个或多个对接收端发布的 IngressEndpoint。
- 独立注册 challenge proof、resume token、退避和 `ShareInstance + pkHash` 身份。

**IngressEndpoint**

- 公网 WSS URL、优先级，以及 DirectWSS 或 TunnelWSS 类型。
- 两种入口使用不同 listener/source identity policy。
- 入口健康只影响自身，不等同于 RelayNode 进程健康。

**PeerSession**

- 一个逻辑 receiver 最多一个 active peer。
- 同时最多一个 gathering replacement。
- 每个 attempt 有单调 attemptId 和独立 candidate 预算。
- active peer 与 replacement 可以短暂重叠；replacement Open 后再关闭旧 peer。

**TurnIssuerAccount**

- 管理员 alias、Cloudflare TURN key ID、签发 secret reference、Analytics secret reference。
- 月度 egress budget、当前观测用量、权重和 breaker 状态。
- secret 值不可打印；错误和 metrics 只使用低基数 alias。

### 4.3 路径分类

core 调度仍只接收 FrameChannel。connectivity 与运维观测必须区分：

- application-relay
- webrtc-direct
- webrtc-turn

selected candidate pair 只报告候选类型、transport 和 provider alias，不记录候选 IP、
TURN username、shareId、RelaySessionID 或 ProtocolSessionID。

## 5. 关键流程

### 5.1 分享发布

1. 发送端复用 R8 产生的同一 sealed ShareDescriptor，不等待 catalog 枚举。
2. 按 RelayNode 并发注册相同 descriptor；challenge proof 绑定节点身份，各节点持独立 resume token，
   同节点多个入口不重复注册。
3. 第一个节点确认后立即打印包含全部配置路由的链接；尚未注册的路由可以暂时返回 typed
   `not_found`，接收端必须继续尝试其他路由。
4. 未就绪节点后台按有界退避继续注册；永久失败时对应 hint 保留为 degraded，且发送端必须
   明确看到该冗余缺口。
5. 需要严格高可用的部署可配置 MinReadyNodes，大于 1 时必须满足该值才发布；默认值为 1。

### 5.2 接收与 relay 恢复

1. `ReceiverInstanceID`（wire: `receiverInstanceId`）在一次接收生命周期内稳定，但不作为授权身份。
2. 首入口立即拨号，未成功时按约 250 ms 启动下一入口，同时最多三个。
3. 第一个通过链接 `pkHash` 验证的 descriptor 建立 `ShareInstance` 锚；停止多余控制竞速。
4. 后续节点返回不同 descriptor 身份或字节时隔离该节点并明确报告身份不一致。
5. 活动控制入口断开时优先尝试同节点其他入口，再竞速其他节点；已交给 LaneSet 的健康数据 lane
   按独立生命周期继续服务。
6. 已 Open 的 PeerConnection 不依赖旧 relay 控制连接继续传输；需要新协商时使用最新活动 relay。

`RelaySessionID` 是节点本地路由句柄；`ProtocolSessionID` 是 R8 建立的 E2E 身份，认证 reattach 才可延续；
每个新 FrameChannel 使用新 `LaneEpoch`。RelayNodeID 只采用发送端发布的分组，endpoint 自报值不是信任根。

### 5.3 分层 Peer 升级

1. 用户手势启动计时和增量 `SelectionMeasure`；ApplicationRelayOnly 立即启用应用 relay 并跳过 PeerSession。
2. 0 秒使用 Google + Cloudflare STUN 创建 direct attempt。
3. ICE failed 或约 3 秒仍未 Open 时取消仍在 gathering 的 direct attempt，并发起 TURN 授权。
4. 所选子树 terminal 且文件数 `<30`、总大小 `<8 MiB` 时立即把应用 relay 加入调度；任一阈值达到可提前判为非小分享。
5. 扫描未完成或失败不延后计时，应用 relay 最迟约 8 秒加入；任一连接阶段明确失败立即进入下一阶段。
6. TURN 全部不可用只产生 typed degraded event；candidate 上限按 attempt、按方向计算，首版保持 64。

### 5.4 匿名能力证明与 TURN 签发

1. receiver 请求 peer escalation，携 `RelaySessionID + ProtocolSessionID + AttemptID`。
2. relay 返回一次性 nonce。
3. capability proof 绑定协议版本、`ShareInstance`、RelayNodeID、RelaySessionID、ProtocolSessionID、
   AttemptID、nonce 和请求类型；派生 key 不离开 core。
4. relay 把证明转交已注册 sender；sender 使用对应 verifier 验证后批准。
5. relay 只在批准后选择 issuer，为发送端与接收端分别生成短效凭据并下发 peer config。
6. nonce 与 attempt 单次使用；错 share、route/protocol session、旧 attempt 和重放均拒绝。

证明只能阻止仅持裸 shareId、日志或入口元数据的一方触发费用，不能区分合法接收者与拿到
完整 fragment 后模拟客户端的扫描器。普通链接预览主要由 fragment 不随 HTTP 请求发送、用户
手势前零 ICE/零签发来缓解。完整能力持有者仍可能在 TTL 内滥用短效凭据；并发、限速、预算、
审计和撤销共同限制该残余风险。

### 5.5 凭据刷新

1. 默认 TTL 2 小时；到约 75% 生命周期时加入随机抖动开始刷新。
2. 新 attempt 使用当前可用账号生成凭据并建立 replacement PeerConnection。
3. replacement Open 后加入调度并关闭旧 peer；失败时旧 peer 保持服务并退避重试。
4. 原账号达到 80% 后不再用于刷新，replacement 选择其他账号。
5. 所有 issuer 不可用时保留旧 peer 至其自然失效，并确保应用 relay 已进入调度。

## 6. Cloudflare TURN 账号池与预算

### 6.1 账号选择

- 每个 attempt 只选择一个账号。
- 选择权重由账号预算与剩余可用字节决定，并对同一 ProtocolSession 保持稳定。
- 401/403 将账号标记为配置故障；429 尊重 Retry-After；超时和 5xx 进入短期 breaker。
- 单次签发最多尝试两个账号，避免外部 API 故障形成请求风暴。
- 所有 Cloudflare 账号共享数据面故障时，不因账号切换反复创建无效 PeerConnection。
- 账号池只允许纳入部署方合法拥有且获准共同用于该部署的账号，不得用于规避供应商条款、
  账户限制或计费规则。

### 6.2 预算状态

| 状态 | 行为 |
|---|---|
| Available | 可参与新签发和刷新 |
| SoftExhausted | 观测用量达到预算 80%；停止新签发/刷新，已有连接继续 |
| Emergency | 观测用量达到预算 100%；触发当前节点活动 lease 的尽力撤销并降级，其余等待 TTL 到期 |
| Disabled | 管理员或永久配置错误禁用；不自动恢复 |

Cloudflare Analytics 只作为运营熔断的观测来源；Cloudflare 账户结算与账单才是费用的最终
权威。WindShare 定期查询账号 egress，不能用客户端上报或本地估算作为安全边界。Analytics
约有 30 秒延迟并采用 adaptive sampling，因此 80% 与 100% 都是可能超调的软阈值。运营 UI
必须同时显示观测时间、已用量、预算、状态、数据延迟和采样性质，不得显示“绝不会超额”。

### 6.3 匿名运营边界

- 每个分享最多 16 个 active TURN sessions；超额接收者只禁用 TURN。
- 签发在能力证明、sender approval、route/protocol-session/source/share admission 之后发生。
- 每来源、每分享、每节点另设签发速率和 burst；初值在 N0 压测后冻结。
- 每个凭据使用独立、低敏的 customIdentifier 做 Analytics 会话归因，不把内部 shareId 写入该字段；
  customIdentifier 不用于撤销。
- 当前进程只在内存 lease registry 中保存撤销所需的 username、账号 alias 和 expiresAt；进程
  崩溃、重启或不可达节点遗失的 lease 不能保证提前撤销，最终由 2 小时 TTL 收敛。
- relay readiness 不依赖 Cloudflare API；TURN 故障只暴露 turn_degraded 指标。
- 后续账户/API Key 体系加入时，TURN 权益和用量归属发送者账户，不改变接收者匿名能力。

## 7. cloudflared 与多入口

### 7.1 用户流程

1. 用户在 Cloudflare 控制台创建 remotely-managed named tunnel 和公网 hostname。
2. 用户自行安装 cloudflared。
3. 用户向 WindShare 提供 Tunnel Token 的 secret reference 与 AdvertisedURL。
4. WindShare 检测 cloudflared，使用 token file 启动 connector，监督 readiness、退出与有界退避。
5. WindShare 不调用账号 API，不创建/删除 Tunnel，不修改 DNS。

GUI 粘贴 token 后写入系统凭据库；headless/server 使用权限受限的 token file、Docker secret
或等价 secret mount。token 不允许出现在命令行、普通配置导出、日志或错误文本中。

### 7.2 listener 与来源身份

- 同一个 RelayNode 可以同时开放公网 DirectWSS listener 和仅供 cloudflared 使用的 loopback origin listener，
  两者共享同一 Hub 状态。
- DirectWSS listener 只使用 TCP peer 身份，忽略所有转发头。
- Tunnel listener 只信显式配置的 immediate proxy 地址，默认仅 127.0.0.1。
- Tunnel listener 只接受恰好一个合法 CF-Connecting-IP；缺失、重复、逗号列表、端口、
  zone 或非法 IP 在 WebSocket upgrade 前拒绝。
- Docker bridge 等非 loopback 拓扑必须显式配置精确 trusted proxy CIDR。
- SourceIdentityPolicy 返回 identity 或 error，不能把坏头静默折叠成 loopback/unknown。

Quick Tunnel、Cloudflare Access 交互登录和任意 TCP Tunnel 不进入首版。Docker 用户仍可选择
独立 cloudflared 容器；该模式与 WindShare 监督本机二进制使用相同的 TunnelWSS 入口契约。

## 8. v2 协议与 Peer 状态机

### 8.1 必需语义

v2 至少需要表达：

- RelayNodeID、RelaySessionID、ProtocolSessionID、receiverInstanceId、LaneEpoch 与 AttemptID 的独立作用域。
- peer start、一次性 challenge、capability proof、sender approval。
- 两端独立的短效 peer config 与 expiresAt。
- attemptId 绑定的 offer、answer、candidate、结束和失败。
- 新活动 relay 接管后继续同一逻辑 receiver 的信令。

字段布局、HKDF label、HMAC preimage、消息大小和错误码在 N2 落地时冻结到
[协议规范.md](协议规范.md) 与 Go↔TypeScript 黄金向量。本文只冻结状态语义，不充当线协议规范。

### 8.2 状态不变量

| 当前状态 | 事件/结果 | 下一状态 |
|---|---|---|
| PeerDisabled | Adaptive 模式且用户手势后会话就绪 | ReadyOnRelay |
| ReadyOnRelay | 启动 direct attempt | NegotiatingDirect(attempt) |
| NegotiatingDirect | PeerConnection Open | PeerActive(attempt) |
| NegotiatingDirect | 明确失败或约 3 秒未 Open | AwaitingTurnAuthorization(attempt) |
| AwaitingTurnAuthorization | 证明获批并取得 PeerConfiguration | NegotiatingTurn(attempt) |
| AwaitingTurnAuthorization | 拒绝、预算停止或 issuer 不可用 | ReadyOnRelay(backoff) |
| NegotiatingTurn | PeerConnection Open | PeerActive(attempt) |
| NegotiatingTurn | 建连失败 | ReadyOnRelay(backoff) |
| PeerActive | 到达刷新窗口 | Refreshing(active, replacement) |
| Refreshing | replacement Open | PeerActive(replacement) |
| Refreshing | replacement 失败且 active 未过期 | PeerActive(active) |
| PeerActive / Refreshing | active 断开或过期 | ReadyOnRelay(backoff) |
| 任一非终局状态 | 取消或传输终局 | Closed |

- receiver 是唯一 offer/replacement 发起者，避免 glare。
- 同时最多一个 gathering attempt；刷新时允许一个 active 与一个 replacement。
- 旧 attempt 的 signal 一律丢弃，不能污染当前 PeerConnection。
- signaling 丢失不关闭已 Open peer。
- 应用 relay 的可用性独立于 peer 状态；刷新失败不关闭旧 peer 或应用 relay。
- ApplicationRelayOnly 始终保持 PeerDisabled，不发起 peer 状态转换，也不签发凭据。
- Closed 是 ProtocolSession 终局；receiverInstanceId 不能授权其复活。

## 9. 工作分解

| ID | 状态 | 工作包 | 依赖 | 验收结果 |
|---|---|---|---|---|
| N0 | ⬜ | 契约与受控 spike | M1 基线 | 配置 schema、候选预算、Cloudflare API/Analytics、coturn 与浏览器矩阵有可复现结论 |
| N1 | ⬜ | 私有配置投影与连接框架 | N0、M2 engine 边界 | 私有字段不外泄；Go/TS 公开投影同语义；core/transport 不拥有策略 |
| N2 | ⬜ | v2 信令、attempt 与能力证明 | N1 | Go↔TS 向量、重放/错代/身份混淆对抗测试通过 |
| N3 | ⬜ | RelaySet、多节点注册与有界竞速 | N1、N2 | 节点故障与 ShareInstance/ProtocolSession 身份锚定 E2E 通过 |
| N4 | ⬜ | 双 STUN 与隐私模式 | N1 | 单 STUN 故障、候选预算和零 ICE 测试通过 |
| N5 | ⬜ | cloudflared supervisor 与可信入口 | N1 | DirectWSS+TunnelWSS 双入口、来源身份和 connector 恢复通过 |
| N6 | ⬜ | 通用 issuer、Cloudflare adapter、账号池与预算 | N0、N2 | 80%/100% 状态、加权选择、尽力撤销/TTL 残余和秘密脱敏通过 |

| N8 | ⬜ | 系统验收、运维、文档与发布门 | N3、N5、N7 | 故障/容量/费用/浏览器演练与 make ci 全绿 |

### N0 — 契约与 spike

- 分别冻结 DeploymentConnectivityProfile、PublishedRelayRoutes、ClientConnectivityPolicy、
  PeerConfiguration 及 secret reference，不能用重复 CLI flags 表达账号、节点和入口集合。
- 冻结 RelayNode、IngressEndpoint、链接总长度和并行拨号的有界上限。
- 使用本地 STUN/coturn 测量两端合法 candidate 最大值，首轮保持每 attempt 64。
- 验证 Cloudflare credential、revoke 与 Analytics 的错误分类、延迟和字段上限。
- 冻结账号预算周期、轮询周期、初始 issuance rate/burst 和 receiver fan-out 压测基线。
- 冻结 `SmallTransferFileLimit = 30`、`SmallTransferByteLimit = 8 MiB` 的排他边界，以及
  `SelectionMeasure` terminal/unknown/scan-error 对连接计时的投影。
- 确认 Chromium、Firefox、WebKit 与 Pion 对 UDP、TCP、TLS TURN 的实际行为。

### N1 — 连接框架

- 建立 DeploymentConnectivityProfile 到 PublishedRelayRoutes、ClientConnectivityPolicy 和
  PeerConfiguration 的显式最小投影，并用负向测试证明 secret/admin 字段不能进入客户端。
- 在 Go engine 与 Web TypeScript runtime 中分别实现同构的公开路由、策略与 PeerSession
  语义，并用黄金向量锁步；不共享包含部署字段的 schema。
- 将各客户端的静态默认工厂移出业务流程；CLI/桌面/移动端由 Go engine 注入，Web 由
  TypeScript runtime 注入。
- 分离 DialURL 与 AdvertisedURL。
- 增加低基数路径事件和 selected candidate type 采集，不把 IP/secret 进入观测。

### N2 — v2 与能力授权

- 定义 v2 codec、大小上限及 RelayNode/RelaySession/ProtocolSession/ReceiverInstance/LaneEpoch/Attempt 身份。
- 在 core 内部 key hierarchy 增加独立 connectivity authorization 派生用途及黄金向量，
  由 core/share 返回角色化 prover/verifier 结构；根模块不得导入 core/internal 或取得派生 key。
- 实现 nonce、proof、sender approval、重放缓存和 typed error。
- v1 端点明确拒绝/提示升级，不维护兼容逻辑。

### N3 — 多 relay

- 发送端按节点独立完成注册 proof、恢复和 resume token 管理。
- 接收端实现有界 Happy Eyeballs、ShareInstance/ProtocolSession 锚、同节点入口去重与控制路由恢复。
- 控制连接与 LaneSet 数据 lane 生命周期分离；数据 lane 并存不能产生重复 receiver 或 PeerSession。
- 归并竞速窗口中的同一 receiver，禁止重复 PeerSession 和重复用户事件。
- 覆盖节点发布前、发布后、传输中故障，以及节点返回不一致 descriptor。

### N4 — STUN 与隐私

- Go/TS 使用同一规范化/去重向量维护 Google + Cloudflare 默认 STUN。
- ApplicationRelayOnly 增加 DNS/网络级零 ICE 验收。
- 用多网卡、IPv4/IPv6 和失败 STUN 组合验证每代 candidate 预算。

### N5 — Tunnel 入口

- 抽取 relay/server 深模块与可返回错误的 SourceIdentityPolicy。
- 实现 cloudflared binary discovery、token file、supervisor、readiness、退避和 graceful stop。
- DirectWSS/TunnelWSS listener 共享 Hub 但使用互斥身份策略；多 RelayNode 竞速仍由 N3 集成。

### N6 — TURN 控制面

- 定义 TurnCredentialIssuer 消费侧接口，返回具体 CredentialLease/Pair。
- Cloudflare adapter 限制响应体、URL 数、host、port、transport、字符串长度和 TTL。
- Cloudflare adapter 保留 UDP 3478、TCP 3478/80 与 TLS 5349/443，过滤浏览器常阻止且
  官方不建议单独依赖的 53 端口。
- 两端凭据分别签发；任一侧失败时撤销已签发的另一侧凭据，再尝试下一个账号。
- 实现账号加权选择、错误分类、breaker、Analytics 轮询、80% soft stop 与 100% emergency。
- 用进程内 lease registry 保留 username、账号 alias 和 expiresAt，尽力撤销可定位凭据；重启
  后不可定位凭据的测试必须证明只存活到 TTL。
- 凭据类型强制 redacted String/GoString；Authorization 和响应 body 不进入日志。
- TURN 签发失败只返回 degraded outcome，不改变 relay health/readiness。

### N7 — Peer 生命周期

- 把一次性协商器重构为 generation-aware PeerSession。
- 实现 job-scoped 小分享判定、0/3/8 秒分层策略、明确失败跳转和每代 candidate 边界；不得读取 descriptor 总量。
- 实现 2 小时 lease、75% 带抖动刷新和 replacement PeerConnection。
- 新 peer Open 前旧 peer 不退出；所有 peer 失败时应用 relay 保持服务。
- Web/Go 都从 selected pair 分类 direct/turn，不能把 TURN 误报为直接 P2P。

### N8 — 收敛

- 完成真实浏览器、真实本地 TURN、模拟 Tunnel、故障注入和负载/race 测试。
- 建立隔离 Cloudflare staging 的 scheduled/manual smoke；普通 PR CI 不依赖公共服务或付费账号。
- 完成部署诊断、预算告警、秘密轮换、紧急停用和降级演练。
- 同步权威协议、安全、路线图、测试向量和简洁部署文档。

## 11. 安全、隐私与运维不变量

1. readSecret、TURN 签发 secret、Analytics token、短效 credential 和 Tunnel Token 是五个独立秘密边界。
2. DeploymentConnectivityProfile 只能生成最小公开投影；链接只承载既有能力秘密和非秘密
   PublishedRelayRoutes，任何 Cloudflare secret 或管理员字段都不进入链接/Web schema。
3. 用户手势仍同时约束起传、ICE、能力证明和 TURN 签发。
4. relay 对内容机密性仍是不可信转发方，但 relay/turnauth 主机对部署方账单与供应商 secret
   是高权限控制面；主机失陷虽不能解密 E2EE 内容，却能盗用额度、窃取 provider token 或拒绝服务，
   因而必须最小权限、隔离部署并支持轮换。
5. 恶意/故障 TURN provider 只能破坏 peer 路径，不能获得内容明文或终止健康应用 relay。
6. 默认双 STUN 会让 Google 与 Cloudflare 分别看到双方公网 IP、时间和必要网络元数据，扩大
   第三方观察集合；relay-only/ApplicationRelayOnly 是零 STUN、零 TURN 的隐私模式。
7. Cloudflare TURN 只见 DTLS 密文及必要网络元数据；Tunnel 边缘仍可见 shareId、时序、
   信令和应用 relay 的既有流量模式泄漏，落地时必须同步 [威胁模型.md](威胁模型.md)。
8. 100% 或紧急停用时只承诺尽力撤销各在线节点内存 registry 可定位的 username；不得宣称
   已全局撤销，遗失 lease 的残余窗口由短 TTL 限制。
9. 多账号扩大的是潜在总账单，不是网络冗余；供应商级承诺必须增加非 Cloudflare TURN。
10. 同一 Tunnel 的多个 connector 只能提升 connector 可用性；多个独立 wsrelay origin 不能在
    没有共享/路由状态的情况下随意放到同一负载均衡入口后。
11. 运维日志和 metrics 不使用原始 IP、shareId、任何 session ID、credential 或高基数错误文本作标签。

## 12. 降级与回滚

- 关闭 TURN provider 后，Adaptive 自动退化为 STUN + application relay，无需协议降级。
- 关闭 Tunnel supervisor 后，同 RelayNode 的 DirectWSS ingress 继续工作。
- RelaySet 配置一个节点时保留完整框架语义，不维护单节点专用实现。
- 账号达到预算或 provider 故障只改变新 attempt；已有分享链接和 relay 传输继续有效。
- v2 上线后不回退 v1；紧急回滚通过禁用新路径能力完成，不重新引入旧协议。
