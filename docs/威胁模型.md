# WindShare 威胁模型

> 状态:**当前安全边界的权威规范**。档位:**防恶意中转**。
> 后续修复范围与验收门见 [`后续计划.md`](后续计划.md)；安全语义变化必须同步回写本文。

## 角色与信任假设

| 角色 | 信任假设 | 能看到什么 |
|---|---|---|
| 中转(官方/第三方) | **不可信**(诚实好奇 / 可能恶意) | `shareId`、密文帧、密文总字节数(≈大小)、连接时序、IP |
| 前端(官方/自部署) | **由使用者自行选择并信任**(持解密密钥,理论上可魔改窃密钥) | 全部明文——密钥只交给接收者亲自选定的前端;官方前端开源可自部署、可审计 |
| 持链接者 | **被授权**(链接即能力) | 全部内容(链接可多人复用,符合产品预期) |
| 发送端(清单提供方) | **不可信于接收端落盘** | 接收端不信任清单路径,落盘前自行校验(见协议规范 §3) |
| 输出根中的既有对象/竞态替换 | **不可信** | 路径检查不能授权覆盖或根外访问;真实操作受 retained `os.Root` 与精确 ownership 约束 |
| 网络中间人 | 不可信 | P2P 由 DTLS 加密;中转回退由应用层 AEAD 加密;均无明文 |

## 保护目标与已知泄漏

**保护目标**:中转拿不到密钥、文件名、文件内容;恶意清单不能以几何诱导越界分配,
不能用路径穿越/reparse point 或宽泛 resume 权限覆盖根外/无关文件;会话终止错误不能被
普通数据积压或紧随其后的 close 吞掉,迟到帧不能复活已终止 session。

**已知泄漏(MVP-1 接受)**:中转可见密文总字节数≈总大小、时序、`shareId`、IP、
**回退转发时明文 `REQUEST` 帧里的块号(访问模式)**。彻底隐藏需 padding/请求加密,
列为后续可选。

## 流转假设(一等公民)

链接预期经**群聊等半公开渠道**流转。推论:

1. 聊天平台服务器可见完整链接——fragment 是消息文本的一部分,"不发往服务器"只约束
   web/中转;平台在密码学上**等价于持链者**。敏感内容对策 = **分离密钥**(可选操作,
   默认单链):密钥不落单渠道,且 128-bit 随机密钥免疫离线爆破。
2. 持链者集合不可控扩大(转发/截图),**"中转拿不到链接"的前提系统性弱化**。
3. 链接预览/杀毒扫描机器人会自动访问链接 → 页面起传**与 WebRTC 信令/ICE** 均需
   用户手势(§6.10),防机器人消耗发送端带宽或经 ICE 收割发送端 IP。

## 残余风险(MVP-1 文档化接受,M2 修复)

完整性全部建立在 `readSecret` 派生的**对称**密钥上,而每个持链者都持有 `readSecret`
——"接收端能验 = 接收端能造"。两个攻击变体:

- **合谋伪造**:持有链接的中转(或与持链者合谋的中转)可伪造通过全部四层校验的清单
  与块,对其他诚实接收端无痕替换内容。P2P 不救:DTLS 指纹经中转信令交换,持链中转
  可自扮发送端完成握手与供块。
- **shareId 劫持(前提更弱,无需恶意中转)**:M1 无 `writeSecret` 所有权证明,且
  shareId 会话随发送端断开而回收 → 任意持链者可在发送端下线后(或注册竞态中)向
  **诚实**中转 register 同一 `shareId`,以伪造内容"复活"链接。宽限窗内的抢注 M1 即由
  `resumeToken` 封堵(§6.8);本残余指期满回收后的复活与首注册竞态。

结构性修复 = **签名套件 suite `0x02`(M2，见
[`后续计划.md` §3.2](后续计划.md#32-签名套件-suite-0x02))**:非对称签名打断
"能验即能造",两个变体同死,且中转零改动。**仅签清单不够**——清单无逐块哈希,
不签块则攻击者仍可在真实几何内替换内容;须逐块签名。

## 安全关键不变量

1. 链接中除 `readSecret` 外皆非秘密;
2. `readSecret` 永不离开浏览器/host;
3. **每块加密用全新随机 nonce**,单 `segKey` 的 Seal 次数由 `MaxSealsPerSegKey` 熔断
   封顶,nonce 碰撞概率 ≤ 2⁻³²(见协议规范 §1.3);
4. 解密清单后、任何块状态/缓冲分配前验证统一几何:`chunkSize ∈ [1 KiB,4 MiB]`
   且为 2 次幂,`streamLen ≤ 2⁴⁸`,`chunkCount ≤ 2²⁶`,稠密状态 ≤8 MiB。需求用
   半开区间 `ChunkSet` 保持紧凑,不能由攻击者迫使展开完整块号列表;
5. 所有路径按 `windshare/path/v1-unicode-15.0.0` 做 NFC、Unicode 15 full fold、Cc/Cf
   和设备名校验;所有文件系统操作经 retained `os.Root`。未被当前 journal 的
   `OwnershipLedger` 精确记录的 canonical path 一律 exclusive-create,不存在目录级
   resume/overwrite 开关;
6. selection 只由一个 `TransferPlan` 解释。journal v2 同时绑定完整 manifest 指纹与
   确定性 `PlanID`,且 have-state 是 plan-local;改变 `--only` 不得把共享边界块的旧
   have 位当成新选择已落盘;
7. terminal ERROR 只能经 `FrameChannel.SendTerminal`;接受即保证在关闭前交付并丢弃
   后续 session 流量。relay 以 `0x03 ‖ sessionId(8B) ‖ inner ERROR` 明示该边界,
   写完成后 tombstone SID;
8. relay 的控制/数据容量按 session 隔离。单 session 的 backlog、overflow、未知 SID
   或 hostile post-terminal 注入不得占用连接级优先 lane、关闭其他 session、panic 或复活。

## 非目标(MVP-1)

前向保密(PFS)、抗流量分析、同一操作系统账户对 journal/输出数据的任意本地篡改
(journal 绑定完整清单指纹与 `PlanID`、精确 ownership 为必做;**可选的 journal MAC**
是文档化的后续加固项,不在 M1)、抗恶意持链者**读取/再分发**(链接本就是授权凭证;
其**伪造**属上述残余风险,M2 修复)、供应链(以可复现构建 + SRI 另行应对)。
