# WindShare 威胁模型

> 状态：**suite `0x02` 当前生产安全边界**。目标是抵抗恶意 relay、网络与恶意持链者伪造；
> 可用性拒绝、读取能力转发和同账户本地篡改仍有明确边界。安全语义须与[协议规范](协议规范.md)、
> [跨运行时向量](../core/testvectors/README.md)和[验收索引](验收索引.md)同步。

## 1. 能力链接流转

能力链接预期经群聊等半公开渠道流转；fragment 是消息文本的一部分，因此聊天平台、预览机器人或截图
取得完整链接后，在密码学上就是持链者。用户可把 `#` 后密钥与不含密钥的 URL 分渠道发送；接收页只在
本地拼接，fragment 必须在网络、日志和持久化工作开始前擦除。持链者被授权读取和再分发全部分享内容，
但不得因此获得 sender 私钥、relay route 所有权或其他 receiver 的 session key。浏览不启动 ICE；preview
或 download 的显式用户手势才可启动 P2P，从而避免自动预览机器人触发地址暴露与带宽消耗。

## 2. 信任边界与可见信息

| 角色 | 信任/能力 | 允许观察或改变 |
|---|---|---|
| relay | 不可信；可延迟、丢弃、重排外层流量 | `shareId/pkHash/ShareInstance`、descriptor digest/resume-token hash 注册头、sender 公钥、RelaySessionID、WS2O 密文长度/时序/总量、IP；不得见内层 kind/direction/terminal/lane/FileID/名称/精确大小/local block/OperationID/ProtocolSessionID |
| 持链者 | 被授权读取；可能恶意且可与 relay 合谋 | 持 `readSecret + pkHash`，能解密 sender objects、创建自己的 session；不得伪造 sender object/control/terminal、注册/恢复/停止 sender route、派生另一 receiver 的 traffic key |
| sender | 内容权威；不受 receiver 信任用于输出路径 | 持 Ed25519 私钥、readSecret、源根；可签名真实对象，不能授权 receiver 越过选择根或覆盖输出根外 |
| receiver 前端 | 使用者选择并信任；持 readSecret | 可读全部授权内容；UI/媒体解析仍按 hostile input 处理 |
| offline delivery | 当前在线模式不存在；未来不可信 | 只存/投递 sender-signed transport-neutral objects/inventory；无 readSecret/private key，不能保存 session envelope |
| 本地源变化/输出既有对象 | 不可信竞态输入 | 不能把 revision 静默换代，不能借 symlink/reparse/ownership 恢复访问根外或覆盖无关对象 |

`pkHash/ShareInstance` 对 relay 可见是注册所有权和多节点路由所需的最小公开元数据；文件身份、请求类型
和精确大小始终在 OperationEnvelope/transport-neutral object 的密文内。operation 类型可能由长度/时序
被概率推断；本阶段不做 padding、固定速率或 ORAM。

## 3. 保护目标与机制

### 3.1 Sender authenticity 与能力持有者隔离

- link `pkHash` 锚定 Ed25519 public key；descriptor 解密后必须先验证 key hash/shareId mapping，再验签。
- shareId 只截断映射 pkHash，故 ShareInstance 刷新不换链接；诚实 relay 保存完整 pkHash，截断碰撞必须
  fail closed，不能覆盖现有 route。
- descriptor/page/directory error/revision/block/offline commit 分属不同签名 domain；签名覆盖 context hash 与
  完整 sealed bytes。持链者能用 readSecret 造 AEAD，但不能产生 sender signature。
- 所有 sender session control 另签 `ShareInstance + ProtocolSessionID + LaneID/Epoch + direction + sequence +
  OperationID + payload hash`；真实对象签名不能剪贴成 terminal/error。
- relay REGISTER/RESUME/STOP 分别使用 purpose 0/1/2 的新鲜一次性 challenge；proof 绑定 RelayIdentity、
  ShareInstance、descriptor digest（注册/恢复）或 StopID（停止）、ChallengeID/nonce/排他 expiry 与 sender
  private key。RelayIdentity 来自运维配置的、无 query 的 canonical
  `ws(s)://authority/base/v2/ws`，拨号 query 只用于 admission，不能让代理 header 或每用户 credential 改写
  relay 密码身份。resume token 只是节点本地附加条件，不能成为替代 authority；已认证连接本身也不能替代
  STOP challenge proof。
- 恶意 relay 仍能拒绝服务或占用自己的路由；这是明确接受的可用性风险。诚实 relay 上持链者不能抢占。

### 3.2 Session confidentiality、PFS 与 replay

- ClientHello 用 readSecret 派生 HMAC proof，在资源 admission 前证明能力；sender-signed X25519 transcript
  防 relay/其他持链者中间人。
- traffic key 同时依赖 X25519 shared secret、readSecret-derived sessionAuthKey 与完整 transcript。
  仅观察 hello 的另一持链者没有 ephemeral private key，不能派生该 receiver 的 traffic key。
- 每方向/LaneEpoch sequence 从 0 严格递增，AAD 绑定 E2E session/lane/方向；RelaySessionID 不进 AAD。
  跨 receiver、跨 lane epoch、跨 session 或方向翻转的 ciphertext 全部失败。
- LaneEpoch 由 session 全局唯一计数器分配，接收时按 LaneID 要求严格增大并允许 gap；跨 lane 乱序不能
  被错误当 replay，一次性 signed grant/LaneHello 仍阻止旧 channel 复活。
- envelope 无法认证时只产生本地无 ID terminal，避免 attacker 让实现签名/反射伪 OperationID。
- offer、answer 与 candidate 只作为 OperationEnvelope 内的 E2E 消息传输；relay 只见 WS2O 密文。
  首个 offer 的非零 `PeerPathID + AttemptID` 绑定本次协商；替换任一身份或在同一 ProtocolSession
  以 fresh OperationID 注入旧 attempt 都必须失败。两者只属于 connectivity 协商；core/content 只在
  channel 建立后接收 `LaneID/LaneEpoch`。
- `OperationID` is correlation only. Receiver lifecycle authority comes from an opaque local binding/generation issued
  before offer creation. Delayed terminal or cleanup work must prove that exact binding; the first transition provenance
  stays stable, consequences only strengthen, and publication waits for in-flight receive plus exact cleanup.
- active peer binding 受 attempt 数量上限约束；结束后进入 30 秒且条目有界的 binding tombstone，
  同 pair 换 OperationID 重放仍拒绝。若 tombstone 容量不足以记录已拒绝 offer，则整段 retention window
  fail closed，不通过驱逐旧 binding 恢复 admission。
  A valid peer-scoped operation error remains attempt-local and cannot impersonate directory/revision/block failure.
  Sender signature proves identity, not cross-scope semantic authority: message kind, failure scope, exact operation,
  envelope lane/sequence, and `PeerPathID/AttemptID` binding are checked independently. Malformed/cross-scope signed
  failures and authenticated answer/final/binding/continuation conflicts seal a session-unsafe consequence.
- Residual error inspection is bounded and diagnostic-only. Only audited exact node types are traversed; unknown,
  hostile, typed-nil, or truncated shapes stay opaque, and an owner-compatible benign leaf cannot erase a genuine
  sibling. Published outcomes own immutable diagnostic snapshots; termination traces export only correlation,
  provenance, and classes—not raw error text or shared content—so telemetry cannot become lifecycle authority or a
  disclosure channel.
- 新 lane 的 HMAC 合法后才可返回 sender-signed WS2B/WS2N；畸形或 HMAC 无效 candidate 必须 silent-close。
  accept/reject 都绑定完整 LaneHello digest，一次性 grant 及 30 秒 consumed/expired tombstone 阻止 replay。
- 临时 X25519 key 擦除后 session control 获得前向保密；readSecret 加密的长期 sender object 不宣称 PFS。

### 3.3 Catalog、源对象与资源耗尽

- synthetic-root/selected-root NodeRecord/预算在 REGISTER 前原子提交；ready 前零后代枚举是安全与体验共同门。
- receiver 只持 opaque kind-safe NodeID；root slot + root-relative locator + object identity 留在 sender store。
- 每次目录 scan 在枚举前后复核对象身份；replacement/rename/type/reparse/collision 不发布 generation。
  page chain、连续 index、唯一 terminal、ShareInstance/signature 防 gap、splice、旧 instance 与 post-terminal append。
- page/NodeRecord/terminal 一次事务提交；spill 只含 catalog metadata。崩溃/磁盘满/取消不留下可见半 generation。
- protocol-session/share/process 的 scan、entry、memory、spill、lease、handle、lane/cache/reassembly预算在取得
  资源前 charge，重连不重置 share/process，活动 revision 不因新 admission 被逐出。
- 未认证长度通过前不分配 fragment buffer；单 operation、session、share、process 均有 byte/record 上限。
  相同 fragment 幂等，冲突拒绝，取消/完成 tombstone 吸收迟到片。
- relay 的 stopped route 永久占一个配置的 route slot；active 与 stopped 共用 `MaxRoutes`。活动与 60 秒
  ended-ID tombstone 同时计入全局 `MaxSessions` 与对应 `MaxSessionsPerShare`，使单 share churn 不能耗尽全局预算，
  delayed WS2O 也不能命中新复用的 RelaySessionID。意外断线只保留 60 秒 crash grace；显式 STOP 以不持有全局
  route lock 的 per-route transaction 先 durable tombstone、再按 Registry exact participant batch drop session，
  永不进入 grace，且 committed cleanup 不依赖 WS2Y 写成功。definite-not-committed 才可恢复；ambiguous I/O 或
  rollback failure 使 route fail-closed，只有相同 StopID 重新验证并 sync 后可解除。该 60 秒窗口内只有
  exact former sender/receiver 的 late WS2O 可 session-local 丢弃；expired、never-issued 或 non-owner ID 必须
  connection-fatal。
- relay 以固定 16B `WS2F SESSION_RETIRED` 精确回收 client channel；命中只关闭该 ID，本地不存在/重复 ID
  allocation-free 忽略，畸形控制 connection-fatal。无法排队或写出通知时 fail-close peer link；60 秒 tombstone
  仍吸收通知与在途 WS2O 的竞态，不能因通知成功提前回收。

### 3.4 FileRevision 与禁止 sender 内容副本

- catalog private candidate + ExpectedSize 必须在打开时匹配；descriptor 只能在稳定 source/handle/quota 建立后返回。
- 稳定性来自排除写入的句柄，或同一已打开 POSIX 对象的
  `device+inode+size+mtime(ns)+ctime(ns)` MutationToken 前后复核；缺少 object identity 的
  size/mtime/ctime 组合不够。
- sender content temp、完整复制、snapshot/staging 是结构性禁止项；无法稳定读取就返回
  `unsupported_stability`。Catalog spill 与 receiver output staging 不属于 sender 内容副本。
- drift 永久终止该 revision、全部 lease 与 cache；receiver 丢弃该 revision 的 DurableRanges/未提交 file
  transaction。其他 file/directory/receiver 不受无条件连带终止。
- file-local key/AAD/signature 绑定 FileID/revision/local index/derived length；BlockCount 不上 wire，块不跨文件。

### 3.5 Output recovery、Web 能力与本地权限

- OutputSession 是唯一 have-state authority；TransferJob、UI、broker 不得由“Write 返回 nil”推断 durable。
- checkpoint 必须 data flush → journal write/flush → atomic replace/目录 sync（按声明级别）→ reopen/identity verify，
  再返回 DurableRanges。任何 crash cut 最多回到上一个已验证 checkpoint。
- recovery 绑定 backend/output session/share/file/revision/size/path-or-persistent-handle/owned object identity；
  一个文件不匹配只废弃该文件状态。
- 所有 OS 写、mkdir、mtime、rename 继续经 retained root/handle 与精确 ownership。未 owned 路径 exclusive-create；
  不存在目录级 overwrite/resume 开关。
- direct stream 报 Durability None 且不承诺 rollback/file isolation。single-file 首字节后失败必须 abort job；
  ZIP 仅能跳过尚未输出 local-file-header 的 member，一旦 member started 后失败必须 abort 整个 ZIP。
- journal checksum 只防意外损坏。同一 OS 账户可同时篡改 journal 与输出数据仍是本阶段非目标。
- Web 的持久目录、OPFS staging、单文件流与 ZIP 只按实测 capability 声明恢复、随机写与失败隔离；
  preview cache 永远不是 DurableRanges。签名媒体仍按 hostile input 处理：有界头探测先于解码，
  尺寸/内存/codec 受限，对象 URL 在替换、失败、取消和关闭时释放。

### 3.6 当前在线生命周期与 M4 offline 边界

- 当前生产只实现 `LiveOnly`；Preparing 失败从未注册，`REGISTERED` 才是链接 ready。
- 显式 Stop/退出必须以独立的 `sender-stopped(0x1008)` 签名 terminal，并完成
  `WS2X -> WS2Q(purpose=2) -> WS2V -> WS2Y` 后永久拒绝新 join、取消活动 revision/scan/lane；
  `illegal-terminal(0x1007)` 只表示协议错误，显式停止不能进入 crash grace。
- 只有非预期网络/进程消失保留 60 秒 route grace，恢复仍须新 relay challenge + sender private-key proof。
- 当前没有 offline uploader、inventory commit 或寄存交付路径。若 M4 实现，未提交 inventory 不对 receiver
  可见，只有原子 `COMMIT_ACK` 后才允许 sender 退出而保持链接；只能存 descriptor/page/revision/block/
  OfflineCommit 等 transport-neutral sender objects 及 digest inventory，保存 OperationEnvelope 或取得
  sender key/readSecret 都违反边界。

## 4. 已知泄漏与残余风险

- relay 仍见 share 路由头、sender 公钥、IP、连接图、密文长度/时序/总量，可推断近似文件/操作模式。
- link 仍是读取能力；被转发、截图或聊天平台扫描后，持有者可读取和再分发。分离密钥仍是单渠道泄漏对策。
- sender 签名的真实对象在同一 ShareInstance 内可被原样重放；在线 session/lease/page chain 会限制其作用。
  M4 必须用 signed `expiresAt`/OfflineCommit 关闭过期后复活，当前在线实现不虚构尚未交付的离线保证。
- X25519/Ed25519、WebCrypto、OS locking、persistent handle 与 filesystem durability 的实现/供应链缺陷仍需平台测试。
- POSIX token 依赖 filesystem 时间精度与 metadata coherency；能在两次复核间完成并恢复同一
  device/inode/size/mtime/ctime 的原地改写可能漏检。该残余只对明确列入支持矩阵的本地 filesystem 接受。
- malicious sender 可提供恶意但签名的文件名/媒体/内容；path policy、root confinement、媒体 parser bounds 与
  OutputSession ownership 是 receiver 防线，签名不等于“内容安全”。
- 恶意 relay/网络可以永久 DoS；流量分析、padding、匿名性、receiver 身份认证、撤销已泄露读取能力、
  同账户本地攻击和供应链攻击仍是非目标。

## 5. 安全验收门

1. 持 readSecret 的攻击测试须无法伪造 descriptor/page/error/revision/block/control/terminal 或注册 route。
2. 修改 ShareInstance、DirectoryID/generation/page、FileID/revision/local index、ProtocolSessionID、lane epoch、
   direction、sequence、OperationID、PeerPathID、AttemptID、fragment total 任一轴都必须在 Go/TS 失败。
3. relay-view 测试只能取得公开注册头与密文长度/时序，不能取得 FileID/ExactSize/local index/OperationID。
4. 后代永久阻塞时 root transaction/descriptor 注册仍完成；注册前 descendant FS/content 操作为零。
5. drift、directory stale、fragment flood、lease/lane/cache admission、多个 receiver 取消均保持错误/资源域隔离。
6. 每个 OutputSession crash cut 都不得恢复未完成 checkpoint；无隔离 stream 不得报告可继续。
7. REGISTER/RESUME/STOP challenge 的 purpose、ID、nonce、expiry、RelayIdentity 或任一 route identity 替换及
   challenge replay必须失败；expiry 相等即过期，错误 proof 消耗 attempt，stopped code 的 retry 必须为 0。
8. WS2A/B/N 修改任一 hello axis、digest、code、retry、reserved 或 signature 均失败；同 grant/response replay
   只消费一次。WS2O 修改 magic/reserved/RelaySessionID/length、零长、65,537B 或非 owner connection 均拒绝。
