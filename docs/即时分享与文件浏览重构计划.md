# WindShare 即时分享与文件浏览重构执行计划

> 状态：**已评审，待 R0 冻结合同**。更新时间：2026-07-15。
>
> 本文定义从现行“单体完整 manifest + 全局打包流”迁移到“即时分享描述 +
> 渐进目录清单 + 文件内块”的执行顺序和验收门。切换前现行协议仍以
> [`协议规范.md`](协议规范.md) 为准；未来计划中冲突的 manifest/global-block 身份语义由本文取代。
> 已确认的产品解释见 [`clarifications/即时分享与文件浏览产品澄清.md`](clarifications/即时分享与文件浏览产品澄清.md)。

## 1. 重构授权与原则

WindShare 尚未到 v1.0，没有外部兼容负担。本次工作以正确领域模型和核心体验为最高优先级：

- **允许并鼓励删除代码、包、协议字段、测试向量和文档，或完全重写相关模块。**
- 不得因为改动范围大而保留全局打包流、单体 manifest、全局块号或全局稠密 bitfield。
- 不得增加 `legacy`、`fileAligned`、双协议栈、翻译适配器或运行时兼容开关。
- 旧抽象阻碍原生语义时直接删除；旧测试若只保护被废弃行为，应由新合同测试替换。
- packing 只能是批量帧或单次传输作业的优化，不能再定义文件身份、内容地址或续传身份。
- 发送端不得为稳定读取创建内容临时文件或完整副本；无法锁定或可靠检测变化时宁可使该文件失败。
  Catalog 元数据 spill 与接收端 OPFS 输出 staging 不属于内容副本。
- 短期开发分支内允许新旧代码暂时共存，但合入后的生产路径不得保留双重语义。

应保留并复用已经正确的能力：能力链接的密钥隔离、路径规范化、root-confined 文件系统、
FrameChannel 传输抽象、P2P/relay 编排、终止帧语义及跨实现黄金向量方法。

本计划是 M2 桌面工作的前置重构；Windows shell、移动端、离线寄存分别仍归 M2、M3、M4。
多 relay/多 P2P 的产品策略归 M5，但本次必须先提供不限制通道数量的调度边界。

## 2. 完成后的用户语义

### 2.1 分享就绪

- 发送端生成分享身份并打开用户选择的根；synthetic-root generation、NodeRecord 和预算必须先在
  CatalogStore 事务提交，再构造 `ShareDescriptor` 并注册 relay。失败不得发送 `REGISTER`。
- descriptor 只携带 synthetic-root ID，不内嵌根条目。根数量、名称总字节和碰撞检查有具名上限，
  该事务只依赖所选根，不枚举后代。
- relay 返回 `registered` 后链接即真正可用；该关键路径复杂度只能依赖所选根数量和一次网络握手，
  不能依赖任何后代条目数量或内容体积。
- 注册前不得递归 `ReadDir`，不得打开后代文件，不得读取、哈希、缩略或分析文件内容。
- 链接就绪后可以低优先级预取根目录；接收端请求的目录始终高于后台预取。
- `OfflineCommitted` 前，明确停止或退出必须拒绝新 join、终止活动下载、取消未完成的离线上传并注销 relay；
  进程崩溃只允许具名网络恢复宽限。离线上传期间仍由在线发送端服务，收到原子提交 ACK 后才允许退出而不使链接失效。

### 2.2 浏览与选择

- 浏览器 join 后先取得已认证的 `ShareDescriptor` 和 synthetic root，不等待完整目录清单。
- 展开目录时才请求其直接子项；未展开目录不进入浏览器内存，也不进入 DOM。
- 目录首次扫描后形成不可变 generation，同一分享的所有接收端复用相同结果。
- 目录选择是可继承的选择规则，不要求其后代已经发现；展开状态与选择状态完全独立。
- “下载目录/全部”触发增量递归发现；每个目录 generation 完整提交后，其中的已选文件立即加入传输，
  不等待其他目录扫描完成。
- 扫描终止前只显示“已发现”的文件数与字节数；不得把未知总量伪装成精确进度。

### 2.3 预览与下载

- 文件是独立可寻址资源，图片预览、视频 seek 和正式下载共享文件范围读取能力。
- 预览按用户动作惰性读取；链接生成阶段不生成缩略图、不探测 MIME、不分析媒体。
- 预览、下载和重连共享有界块缓存，但分别拥有优先级、取消、完成和错误状态。
- 一个文件漂移、删除或不可读，只使该文件 revision 失败，不得无条件终止其他文件。
- 下载点击立即启动连接计时。只有所选子树发现完成且文件数小于 30、总字节小于 8 MiB 才属于小分享；
  未知总量按非小分享时序处理，不能延后约 8 秒的 relay 上限。

## 3. 已确定的领域模型

```text
Share
├── Capability / ShareInstance / lifecycle
├── sealed ShareDescriptor
├── Catalog -> DirectorySnapshot -> CatalogPage[]
└── FileObject -> FileRevision -> file-local Block[]
SessionRuntime(role-neutral)
├── ProtocolPump[] <- FrameChannel[]
├── RoleRouter
└── SessionWriter[] -> FrameChannel[]
SenderRuntime -> Catalog/Revision/Content services
ReceiverRuntime
├── CatalogClient
├── BlockBroker -> LaneSet
└── TransferJob -> SelectionRules + OutputSession
```

### 3.1 ShareDescriptor

`ShareDescriptor` 是 relay 注册和 receiver join 的唯一初始对象。它包含 wire/suite 版本、
`ShareInstance`、synthetic root、块参数、能力位和 suite `0x02` 公钥信息；
不包含完整 entries、总字节、总块数或全局布局。描述必须有独立具名大小上限并 seal-once 复用。
suite `0x02` 链接中的 `pkHash` 锚定发送端公钥；descriptor、directory page 和 file block 使用
互不混淆的签名域，不再引用 manifest tag 或全局块号。

`shareId` 必须按具名映射绑定 `pkHash`；`REGISTER/RESUME` 对 relay challenge 签名，并绑定
`shareId + ShareInstance + relay identity`。持链者没有发送端私钥，不能在诚实 relay 抢占路由；
恶意 relay 拒绝服务仍是明确接受的可用性风险。

能力 URL 的身份仍来自密码学随机材料；产品上的“可用”以小型 descriptor 获得 relay ACK 为准。
若接收端先于 ACK 打开链接，join 可以在有界窗口内显示“分享正在启动”并重试，但不能假装已经就绪。

### 3.2 逐目录冻结 Catalog

- `NodeID` 是分享内稳定且对 relay 无语义的 opaque 标识；`DirectoryID` 与 `FileID` 是其
  kind-safe 形态，receiver 不提交本地路径字符串。
- 第一次 `ListChildren(NodeID)` 对该目录的**直接子项**执行一次 root-confined 枚举、规范化、
  sibling collision 检查和确定排序，然后发布不可变 `DirectoryGeneration`。
- `DirectoryID` 绑定父 generation 发现的对象身份，而非路径。枚举前打开并验证身份，枚举后再次复核；
  重命名、同路径替换、类型变化或 reparse 均返回 directory stale，不提交 generation。
- 目录条目超过一页时拆成具名协议上限约束的 `CatalogPage`；UI 的“每次显示多少项”不得成为线协议常量。
- catalog 文件条目记录冻结的 `ExpectedSize` 和 sender-private 版本候选；后续打开只接受该候选，
  不把扫描后的新文件内容静默更新到旧 generation。
- directory generation 与条目携带签名的可选 `ModifiedTime + Precision`；输出端只恢复存在且后端支持的 mtime。
- 页身份绑定 `ShareInstance + DirectoryID + Generation + PageIndex + Terminal`，并以链式 digest
  或等价 commitment 检测缺页、乱序、拼接和伪造终止页。
- 同一页的逐字节重传必须幂等；同一身份对应不同内容、终止页后的追加或旧 ShareInstance 重放必须拒绝。
- 多接收端并发首次展开由 share-scoped singleflight 合并；页面 seal-once 并由 CatalogStore 复用。
- 已提交 generation 的页面必须保留到分享结束。CatalogStore 以有界内存配合发送端本地临时 spill
  实现，不能通过逐出语义状态后重扫磁盘来伪造同一 generation；临时数据在分享结束时清理。
- CatalogStore 同时保存 sender-private NodeRecord（所选 root、root-relative 定位、类型和身份基线）；
  receiver 只持 opaque ID，不能借请求路径越过已选择的根。NodeRecord 索引也必须支持有界内存和 spill。
- 预算同时约束 ProtocolSession、share 和进程的扫描工作、已提交条目、内存及 spill 字节；重连不能重置
  share 预算。超限发布已认证的 directory error，不得发布半个 generation。
- generation 先完整写入临时事务，页面、NodeRecord 和 terminal 全部持久化成功后才原子提交；磁盘满、
  取消或崩溃只留下可清理的未提交状态。
- 扫描失败不产生 generation。永久错误在分享期内复用；瞬时错误返回已认证的
  `ScanAttemptID + retryable + retryAfter`，冷却后的显式请求才能开始新 attempt，并发请求仍由 singleflight 合并。
- 初版不提供 live view 或 refresh。文件候选在扫描后变化时，打开 revision 返回 stale，用户须重新分享。

该模型有一个诚实边界：若单个目录直接包含百万子项，并且要求完整碰撞检查和稳定排序，首次展开
仍需扫描这些直接子项。它不再阻塞链接，也不会递归扫描子目录。若将来需要 provisional pages，
必须另行定义 commit/abort 语义，不能让临时页冒充已冻结快照。

### 3.3 文件 revision 与文件内块

receiver 先取得 revision，再按文件内块寻址：

```text
OPEN_REVISIONS(FileID[], optional initial ranges)
  -> RevisionOpenResult[] =
       { FileRevisionDescriptor { FileID, FileRevision, ExactSize },
         RevisionLease { LeaseID, FileID, FileRevision, ExpiresAt } }
       | { FileID, RevisionError }
REQUEST_BLOCKS(LeaseID, BlockRef[])
RELEASE_REVISION(LeaseID)
BlockRef = (FileID, FileRevision, LocalBlockIndex)
```

以上请求和响应由 `OperationEnvelope` 关联；descriptor 本体不包含 `OperationID`。`BlockCount`
只由 `ceil(ExactSize / chunkSize)` 派生，不上线。

- 打开可以批量执行或与首批 range 融合，但每个成功 revision 返回独立 lease；批量只是传输优化，
  续期和释放均按 revision 执行，一个文件完成后不得被同批慢文件继续占用容量。
- `RevisionSource` 必须验证 catalog 的版本候选和 `ExpectedSize`，再取得写入排除句柄，或可在每次读取
  前后校验的可靠单调 `MutationToken`。候选已变化则返回 stale。
- WindShare **不得在发送端创建内容临时文件、复制完整文件或以本地 staging 换取稳定 revision**。
  无法可靠锁定或检测变化时拒绝打开；打开后发现变化则永久失败该 revision、取消其 lease 并废弃缓存，
  receiver 必须中止对应 file transaction 并丢弃该 revision 的 DurableRanges。
- `RevisionStore` 在成功返回 descriptor 前完成句柄、变化跟踪和并发建立的 admission。
  活跃 lease 及最后 lease 后的具名 resume grace 内不得驱逐；过期且无 lease 的 revision 才可回收。
- lease 续期、释放、最长生命周期及 protocol-session/share/process 配额在 R0 冻结。预算不足只拒绝新打开，
  不驱逐活跃 revision；回收后再次打开会验证原候选并产生新 `FileRevision`，旧 resume 明确失效。
- 同一 `FileRevision` 的所有块必须来自 RevisionStore 中同一对象；无法建立稳定读取或可靠变化检测时
  不得返回 descriptor。变化检测失败只终止该文件；进程退出统一释放全部 revision。
- 一个块只能覆盖一个文件；`ReadRange` 只映射到该文件的本地块，首尾允许有明确上限的块边界冗余。
- key derivation 至少区分 catalog、file 和 file segment；descriptor、块 AAD/签名绑定 wire version、
  `ShareInstance`、`FileID`、`FileRevision`、本地块号和明文长度。
- 加密块缓存可逐出；重新读取必须在变化校验内来自 RevisionStore，并使用全新随机 nonce 重 seal，
  遵守每 key Seal 次数上限。

### 3.4 增量选择、传输与续传

- 删除“完整 entries -> immutable TransferPlan -> 全局 ChunkSet”的前提。
- `SelectionRules` 表达默认选择及目录/文件 override；未知后代继承最近祖先规则。
- 点击下载时冻结本次作业的选择规则；之后的 UI 选择只影响下一次作业。
- `TransferJob` 持续消费已冻结目录，产生文件级 range operation，并对每个文件独立完成或失败。
  已选目录扫描失败时记录 typed directory failure、跳过未知子树并继续独立分支；`JobOutcome` 必须为
  `Succeeded/CompletedWithErrors/Aborted` 之一，不得把未发现子树算作成功。
- `SelectionMeasure` 单调报告已发现文件数、字节数和 discovery terminal。`SmallTransferFileLimit = 30`
  与 `SmallTransferByteLimit = 8 MiB` 都是排他上限；只有 terminal 且同时低于两者才为 `Small`。
  任一阈值达到即可提前判为 `Large`；扫描未终止或失败为 `Unknown`，连接层按非小分享时序处理。
- 输出选择器必须在浏览器点击处理器中同步启动，随后才异步发现目录，避免丢失 user activation。
- preview cache 只绑定源身份 `ShareInstance + FileID + FileRevision`；落盘 resume 另绑定
  `OutputBackend + OutputSessionID + canonical path/persistent handle + ExpectedSize + owned file identity`。
- OutputSession 独占恢复验证和 checkpoint；只有它返回的 `DurableRanges` 才能进入 have-state。
  一个文件 revision 变化只废弃该文件状态，TransferJob 不得自行推断写入已持久。
- 全局 manifest fingerprint、基于完整路径集合的 PlanID 和 share-wide 稠密 bitfield 全部删除。

### 3.5 OutputSession

`OutputSession` 是 `TransferJob` 消费侧的窄接口，不实现传输。`DurabilityLevel` 明确为
`None/ProcessRestart/PowerLoss`；它同时报告随机写、file failure isolation 和 mtime 能力，并提供
`EnsureDirectory/FinalizeDirectory/BeginFile/FinishJob(JobOutcome)/AbortJob`。
`BeginFile` 先验证输出身份、大小和 journal，返回 file transaction 及 `VerifiedDurableRanges`；transaction
提供 `WriteRange/Checkpoint/Commit/Abort`。`Checkpoint` 必须按声明的持久级别先提交数据、再提交 journal，
并返回实际 durable ranges；恢复时不得信任未验证或未完成的 checkpoint。

`Abort` 只有在后端声明 file failure isolation 时才保证丢弃未提交文件并继续作业。未 staging 的流式
单文件/ZIP 一旦输出字节就不能回滚，后续 file 失败必须升级为 `AbortJob`；staging 后才能声明隔离。
空目录和空文件是正式语义；CLI 由 `osfs` 实现，Web 只展示后端实际能力。

Web 后端固定按能力分层：File System Access 持久化 handle 与 journal，经重新授权和验证后可报告
`ProcessRestart`；接收端 OPFS staging 用于大文件/多文件 fallback，只有持久存储和重开验证成立时才
报告 `ProcessRestart`，完成后再导出；直接单文件/ZIP 流报告 `None`。页面刷新建立新 ProtocolSession，
不得依赖 Service Worker 维持传输。OPFS 仅属于接收端输出，不得成为发送端内容副本。

### 3.6 SessionRuntime、BlockBroker 与 LaneSet

每个 FrameChannel 由 role-neutral `ProtocolPump` 独占读取、`SessionWriter` 独占写入。pump 打开 AEAD、
验证 sender 签名后交给 role router：receiver 侧分发 catalog/content/error，sender 侧分发 list/open/block/
lease/cancel。业务组件不得直接 `Recv/Send`。writer 使用有界 control/data 队列和公平调度；control/terminal
优先，数据背压不得阻塞 pump 接收 cancel，terminal 接受后丢弃后续流量。

身份轴必须正交：

- `RelaySessionID` 仅供 relay 外层路由，不可信且可随节点/拨号变化；
- `ProtocolSessionID` 与方向独立 traffic keys 来自 sender-authenticated transcript，约束 `OperationID`
  和 terminal；只有认证的 lane reattach 才能延续，新 ProtocolSession 使旧 operation 全部失效；
- `ReceiverInstanceID` 只归并一次接收生命周期中的拨号和 UI 事件，不参与授权、AAD 或输出 ownership；
- `LaneID + LaneEpoch` 标识逻辑 lane 及每次 FrameChannel 实例，方向序号和重放窗口按 epoch 隔离。

`OperationEnvelope` 以 `ProtocolSessionID` 内唯一的 `OperationID` 关联请求、响应、`OPERATION_ERROR` 和
`CANCEL`；AAD 绑定 wire version、`ShareInstance`、`ProtocolSessionID`、`LaneID`、`LaneEpoch`、方向和
重放序号。seal-once 对象不包含 `OperationID`，可跨 receiver 复用。所有 sender 控制消息另行签名；
operation 控制签名绑定 `OperationID`，terminal 不含 `OperationID`，二者都绑定 ProtocolSession 和 LaneEpoch。

`ShareDescriptor/CatalogPage/FileRevisionDescriptor/BLOCK_RECORD` 是 transport-neutral 的 sender-signed
密封对象，不绑定 ProtocolSession、OperationID 或 Lane。M4 只寄存这些对象及签名 inventory；离线服务以
独立 delivery 身份包装传输，不能解密、修改或新建 sender 对象，也不能取得 sender 私钥。

收到合法 `SESSION_TERMINAL` 时终止整个 ProtocolSession；envelope 无法认证时只产生本地 terminal。
BlockBroker/LaneSet 为 receiver + ProtocolSession scoped；多 receiver 仅共享 sender share-scoped store、
源读取 singleflight 和 sealed-block cache，不共享 OperationID、取消、窗口或接收状态。

目录浏览默认使用已建立的 relay lane，不触发 SDP/ICE；预览或下载点击才启动 P2P 与 0/3/8 秒计时。
应用 relay 的入池时机消费 job-scoped `SelectionMeasure`，不得依赖 descriptor 中不存在的分享总量。
实际多 relay/多 P2P 策略可由 M5 启用，但接口不得假设只有一个活动通道。认证/线协议/分享身份错误是
ProtocolSession 级；目录扫描、file revision 和 block/range 错误分别留在各自错误域。

### 3.7 BLOCK_RECORD 分帧

`BLOCK_RECORD` 是逻辑上的签名密封对象；线上拆为不超过 `MaxFrameSize` 的 `BLOCK_FRAGMENT`，
每片各自使用 `OperationEnvelope` 认证。片内字段至少包含 record ID、片序号/终止标记、认证后的总长和
payload 长度；未验证 AEAD 与具名长度上限前不得分配重组缓冲。

重组按 `ProtocolSessionID + OperationID + record ID` 隔离，并受 protocol-session/share/process 预算约束。
相同片可幂等重传，冲突片拒绝；取消后保留有界 tombstone 并丢弃迟到片。R0 冻结乱序、缺片、超时、
认证失败和取消的错误域；可复用的是逻辑 `BLOCK_RECORD`，不是 receiver-specific fragment envelope。

## 4. 目标模块边界与删除范围

| 现有区域 | 目标 | 处置 |
|---|---|---|
| `core/manifest` | 路径策略迁入 `catalog`，descriptor/page 使用新编码 | 删除整个单体 manifest 包及 `Manifest{Entries}`、单一 fingerprint |
| `core/layout` | file-local geometry/range set 迁入 `content` | 删除整个 packed layout 包、全局 starts、`ChunkToRanges`、全局 `ChunkSet` |
| `core/share` | share identity 与应用门面 | 重写 `NewSharer(files...)`、完整快照 Receiver、`TransferPlan/PlanID/TransferSink` |
| `core/session` | session transcript/identity、role-neutral pump/router/writer、fragment 与终局 | 单点收发；控制优先、签名验证和有界重组 |
| 新 `core/catalog` | 节点、目录 generation、页面 codec/验证 | 作为目录身份与浏览语义唯一权威，不依赖网络或 OS 实现 |
| 新 `core/content` | revision descriptor、RevisionStore/Lease、file-local geometry/key/signature | 变化即失败且禁止 sender 内容副本，提供 share-scoped 读取与密封缓存 |
| 新 `core/transfer` | SelectionRules、SelectionMeasure、TransferJob、BlockBroker/LaneSet、OutputSession、sparse resume | 统一 CLI/Web 传输语义，不让业务规则回流 UI |
| `core/chunk`、`internal/keyderiv` | 能力迁入 `content` 或按新职责重写 | 删除 global stream index 对密码身份的控制 |
| `core/osfs` | 无内容副本 RevisionSource/Store 与可恢复 OutputSession | 删除 eager `Walk`；实现变化检测、输出身份验证、durable checkpoint 和根约束 |
| connectivity/transport | 向 LaneSet 提供 relay/P2P FrameChannel | 不拥有块身份、选择、去重或输出策略 |
| `relay/admission`、signaling hub | descriptor、注册所有权证明与 route/connection 预算 | 删除 manifest blob 存储/总量预算，relay 不持有 catalog 页面 |
| CLI journal | OutputSession-owned file-local checkpoint store | 仅持久化经输出端确认的 DurableRanges |
| `web/src/manifest` | catalog client、file geometry、range reader | 删除完整 manifest 解码后才能 join 的门槛和 packed layout |
| `web/src/session`、`web/src/ui` | SessionRouter、BlockBroker、文件树、增量作业和 FSA/OPFS/流式输出矩阵 | 删除竞争读取、一次性 plan/session 与平面 `SelectionPageWindow` |
| `testvectors`、协议文档 | 新 wire/suite 权威合同 | 替换旧向量和规范，不保留生产兼容解析分支 |

包名以提供的能力命名，优先形成 `catalog`、`content`、`transfer` 等深模块；不得把新逻辑堆入
`util`、`common`、`misc` 或 CLI/UI 边界。

## 5. 线协议工作集

R0 必须冻结下列语义后才允许实现，不在旧帧上叠加临时字段：

- synthetic-root/NodeRecord 事务先于 `REGISTER`，以及 `ShareDescriptor`、`shareId/pkHash` 和注册证明；
- sender-authenticated session transcript、方向 keys、`RelaySessionID/ProtocolSessionID/ReceiverInstanceID/LaneEpoch`；
- AEAD `OperationEnvelope`、方向/重放绑定、`OperationID`、`OPERATION_ERROR`、`CANCEL` 与无 operation 的 `SESSION_TERMINAL`；
- 所有 sender 控制消息的签名 preimage，以及无法认证时只产生本地 terminal 的规则；
- `LIST_CHILDREN`、目录身份复核/stale、带可选 mtime 的 seal-once `CATALOG_PAGE` 和 scan error；
- 可批量/融合但逐 revision lease 的 `OPEN_REVISIONS`、`FILE_REVISION`、renew/release、
  `REQUEST_BLOCKS`、`BLOCK_RECORD` 与 `BLOCK_FRAGMENT`；
- fragment 的认证、乱序重组、取消、迟到片、错误域和三级在途预算；
- role-neutral pump/router/writer 的单点收发、有界背压、公平调度、cancel 和 terminal 优先级；
- directory/file/block/ProtocolSession 四级错误分类；
- `OperationID`、`NodeID`、`FileRevision`、`RevisionLease`、`DirectoryGeneration` 编码；
- descriptor、catalog page、file revision 和 block 的签名 preimage、随机 nonce、重放与 seal-once 语义；
- transport-neutral sender 对象与 session envelope/delivery 身份的分离，保证 M4 可寄存而无需 sender 私钥；
- suite `0x02` 的 capability-holder 对抗：持有 `readSecret` 仍不能伪造 sender 内容或控制消息；
- `RevisionSource/RevisionStore` 的无内容副本建立、变化检测、lease、过期回收、resume grace、重连和 admission；
- protocol-session/share/process 三级扫描、revision lease、句柄、Catalog 元数据 spill、接收端输出 staging、
  lane、缓存和在途字节上限；发送端内容临时文件必须由结构门禁禁止；
- OutputSession-owned recovery/checkpoint、`DurabilityLevel`、`DurableRanges` 和崩溃提交顺序；
- directory failure 的 `JobOutcome`，以及 OutputSession/Web capability matrix；
- `SelectionMeasure` 的 30 文件/8 MiB 排他边界、Unknown 语义和 0/3/8 秒连接计时；
- `OfflineCommitted` 前明确停止/退出的 terminal、注销、上传取消及 crash grace；
- receiver-scoped `LaneSet/BlockBroker` 与 sender share-scoped cache 的身份、取消和资源边界。

除 relay 外层 share/RelaySessionID 路由外，catalog、open/revision、content request/response 和 operation error
全部使用 AEAD envelope；`FILE_REVISION` 的 `ExactSize` 不得明文暴露。威胁模型只接受 relay 观察
密文长度、操作时序和传输量，不再接受逐文件标识或精确大小泄漏。

## 7. 结构性验收与测试矩阵

### 7.1 “立即可用”硬门

不使用容易受 CI 机器波动影响的绝对毫秒阈值作为唯一判据。注入可观测文件系统和假 relay：

- 将所有后代枚举永久阻塞，仍必须先完成 descriptor 注册并产生可用链接；
- ready 前后代 `ReadDir/Open/Stat` 次数必须为零，内容读取和哈希次数必须为零；
- 0、1k、10k、100k、1m 后代虚拟树的 ready 前操作数、注册字节和核心分配不得随后代增长；
- 真实磁盘基准记录 cold/warm p50/p95、`ns/op`、`allocs/op`、`B/op`，只用于趋势和回归归因；
- 进程 E2E 必须证明“释放枚举闸门前，CLI 已输出链接且浏览器已打开 descriptor”；
- synthetic-root/NodeRecord/admission 任一步失败都不得发送 `REGISTER`；ACK 后根 generation 无需再次准入即可读取；
- `shareId/pkHash` 映射和 challenge proof 必须使持链者无法在诚实 relay 注册或恢复发送端路由。
- `OfflineCommitted` 前明确停止/退出必须令新 join 失败、活动下载收到 terminal、未提交上传清理；
  只有网络中断可进入具名 crash grace，显式停止不得恢复。

### 7.2 Catalog

- canonical codec、路径策略、页/条目/深度/整数上限的单测与 fuzz；
- 篡改 share、parent、generation、page index、terminal 或 entries 均认证失败；
- gap、乱序、跨目录拼页、冲突重传、旧 generation、terminal 后追加全部拒绝；
- 不同 receiver/OperationID 的 envelope 可变化，但其中 seal-once CatalogPage 保持逐字节相同；
- 永久扫描错误复用；瞬时错误在冷却后由显式请求产生新 attempt，并发 waiter 得到同一 attempt 结果；
- 篡改 ScanAttemptID/retryable/retryAfter 认证失败；取消、超时和最后 waiter 离开释放未提交资源；
- 极宽目录在内存预算内完成外部排序/spill，已提交页面即使从内存逐出仍可逐字节重放；
- share/process 预算不能被重连重置；超限、磁盘满或崩溃不发布半个 generation，临时状态可恢复清理；
- 展开目录 A 不得扫描其兄弟目录 B；持有 `readSecret` 的 receiver 仍不能伪造目录页或扫描错误；
- 子目录在首次展开前被重命名、同路径替换、改类型或变为 reparse 时返回 stale，枚举竞态不提交 generation；
- 篡改或越权恢复 mtime 必须失败；冻结后文件变化不改变旧页面，无法建立 revision 时只失败该文件。

### 7.3 Content、传输与续传

- OPEN_REVISIONS 可批量/融合但逐项返回 lease/error；完成一个 revision 即可独立释放，候选变化返回 stale；
- 打开后在读取前、读取中或读取后改变源文件，必须永久失败该 revision、取消全部 lease，并使 receiver
  丢弃对应 DurableRanges；测试文件系统同时证明发送端没有内容临时文件、完整复制或隐藏 staging。
- BlockCount 只从 ExactSize/chunkSize 派生；wire descriptor 出现冗余块数字段必须拒绝；
- `BlockRef` 边界、EOF、溢出和 range 属性测试；任何块不得跨文件；
- 修改 `FileID/FileRevision/LocalBlockIndex` 任一字段都必须导致认证或签名失败；
- 活跃 lease 不被新 admission 驱逐；最后 lease 到期并经过 grace 后可释放，顺序传输能继续打开新文件；
- 缓存逐出和 receiver 在 grace 内重连仍读取同一 revision；回收后旧 lease/resume 明确失败；
- 每 lane 只有一个 pump reader 和 writer arbiter；sender 的 block 背压不得饿死 list/open/cancel/terminal；
- RelaySessionID 变化不改变 E2E 身份；ReceiverInstanceID 不授权，ProtocolSessionID/LaneEpoch 越域重放拒绝；
- sender 控制签名绑定 operation、ProtocolSessionID、LaneEpoch、方向和重放状态；持链者不能伪造终局；
- OPERATION_ERROR 只结束对应操作；畸形/未认证 envelope 只产生本地无 ID terminal；
- 最大块跨多个 ≤64 KiB fragment 完成 Go↔TS 向量；乱序、重复、冲突、缺片、超时、取消和迟到片受预算约束；
- `ReadRange` 只读取覆盖范围所需本地块，除具名首尾块冗余外不得退化为完整文件读取；
- preview 与 download 的重叠范围只上游读取一次；单 consumer 取消不破坏共享请求；
- 多 lane 抢跑只提交一个 winner，迟到块幂等丢弃，取消和窗口不串扰其他 lane；
- relay 视角测试证明 OPEN/FILE_REVISION/REQUEST/BLOCK 不暴露 FileID、ExactSize 或本地块号；
- BeginFile 只返回验证过的 durable ranges；在 data write/flush、journal write/flush 和 commit 各点注入崩溃，
  重启后未完成 checkpoint 的 range 必须重取；
- checkpoint 绑定 backend/OutputSession/path-or-handle/size/file identity；改变一个文件只废弃该文件状态；
- `SelectionMeasure` 覆盖 29/30 个文件、8 MiB 前一字节/恰好 8 MiB、提前越界、空目录、扫描失败和
  未完成发现；只有 terminal 且同时低于两个排他上限才立即启用小分享 relay。
- 已选目录扫描失败产生 CompletedWithErrors 并继续独立分支；无隔离能力的流失败必须终止 job；
- OutputSession 覆盖根约束、空目录/文件、mtime、file/job abort 和声明的 DurabilityLevel；
- receiver broker 状态互相隔离；sender cache 可共享读取，但慢、恶意或取消的 receiver 不阻塞其他 receiver。

### 7.4 Web 与媒体

- 乱序条目、隐式目录、深树、宽树、未加载后代继承选择规则；
- 展开/折叠/搜索不编译完整传输计划，DOM 行数受全局预算约束；
- output picker 在用户点击同步启动，随后目录发现不丢失 activation；
- File System Access 持久 handle、OPFS 接收端 staging、单文件流和 ZIP 流分别验证重授权、配额、
  checkpoint、刷新恢复、导出与失败隔离；流式后端不得宣称可回滚或续传；
- 页面刷新必须关闭旧 ProtocolSession，从已验证的 DurableRanges 新建会话恢复；Service Worker 退出不能影响正确性；
- 图片头嗅探、内存/尺寸上限、对象 URL 释放和恶意媒体输入；
- 视频头尾读取、快速连续 seek、取消、多个预览及 preview 后正式下载缓存命中；
- Chromium、Firefox、WebKit 与真实 relay→P2P 热切换路径通过。

### 7.5 工程门禁

迭代期间按影响运行 `make vectors`、`make race`、`make coverage`、`make network`、`make web`、
`make browser`；最终必须 `make ci`。新增一个与 CI 同构的确定性 experience gate，检查 ready 前零递归
扫描、descriptor/注册字节有界及阻塞枚举仍先出链接。继续满足 core 总覆盖率 ≥90%、根模块 ≥80%、
每个 Go 包 ≥70%，不得用 exclude 隐藏新模块。

## 8. 已知代价与边界

- 完整下载、精确总量和全局搜索最终仍需发现所有相关条目，只是不再阻塞分享就绪和首屏。
- 逐目录冻结不是“点击分享时的全树原子快照”；不同目录可能在不同时间冻结，UI 和文档必须直说。
- 稳定 FileRevision 在 lease/grace 内占用锁定句柄或变化跟踪资源；源漂移宁可失败，
  发送端也不得创建内容副本。
- 每个非空小文件至少有一个认证单元和一次 revision 打开；批量/融合只降低往返、调度和帧开销。
- 极宽目录的原子冻结仍是 `O(direct children)`；后台预取只能隐藏部分等待，不能消除文件系统下界。
- Web 的 FSA/持久 OPFS 后端可在重新验证后跨刷新续传；直接单文件/ZIP 流刷新后重下，UI 必须按能力说明。
- relay 虽看不到名称和内容，仍可观察访问时序与流量形状；padding/请求隐藏不属于本阶段。
- 注册所有权只防诚实 relay 被持链者抢占；恶意 relay 或网络直接拒绝服务不在可用性保证内。
- `OfflineCommitted` 前发送端退出会终止整个分享；M4 必须原子物化 transport-neutral descriptor、CatalogPage、
  revision 和 block 后才允许发送端退出，不得重新引入单体 manifest 或保存 session envelope。

## 9. 非目标

- 每文件独立链接、WebRTC 连接或 relay 会话；
- 在本阶段启用多 relay/多 P2P 产品策略；本阶段只保证 LaneSet 不阻塞 M5 实现；
- 链接生成阶段的内容读取、哈希、缩略图、MIME 探测或媒体转码；
- 发送端内容临时文件、完整文件复制或静默 staging；
- 为旧 suite、旧 manifest 或旧全局块号提供运行时兼容；
- 未索引完整 catalog 时承诺精确全局搜索、总条目数或总字节数；
- 用扫描并发、manifest 压缩或更大的单体上限伪装成根因修复。

## 10. 总完成定义

只有以下条件全部满足，本计划才可标记完成：

1. synthetic-root/NodeRecord/预算在注册前提交；ready 前不递归扫描，注册证明阻止持链者抢占诚实 relay。
2. receiver 不以完整 manifest 作为 join 门槛；目录按首次访问冻结，绑定对象身份并携带可信 mtime。
3. 生产数据面不存在全局 packed-stream 地址；所有内容以 file-local block/range 寻址。
4. OPEN_REVISIONS 逐项返回 descriptor/lease/error；候选变化 stale，完成文件可独立释放 lease。
5. 同一 FileRevision 只从已 admission 的 RevisionStore 交付；源漂移使 revision 失败，发送端不创建内容副本。
6. RelaySessionID、ProtocolSessionID、ReceiverInstanceID 和 LaneEpoch 正交；envelope/签名使用明确的 E2E 身份。
7. 每个 FrameChannel 只有一个 role-neutral pump 和 writer；控制优先，分片重组、取消和预算跨实现一致。
8. FILE_REVISION/ExactSize 与所有 operation payload E2EE，relay 只观察密文长度、时序和传输量。
9. BlockBroker/LaneSet 为 receiver scoped；多 receiver 仅共享 sender store/cache，取消和错误域不串扰。
10. OutputSession 独占恢复验证和 checkpoint；只有返回的 DurableRanges 能进入 have-state。
11. OutputSession 按 capability 承诺持久级别与失败隔离；FSA/OPFS/流式刷新语义准确，目录失败进入明确 JobOutcome。
12. CatalogStore 的预算、事务提交、目录 stale、扫描错误复用/重试和崩溃清理均通过验收。
13. 续传身份为 file-local sparse state，单文件变化不会废弃整次分享；小分享分类不依赖 descriptor 总量。
14. 旧 manifest/layout/global PlanID/global bitfield、relay manifest storage/admission 和旧行为测试已删除，
    CI 拒绝重新引入这些 import、协议字段和生产符号。
15. [`协议规范.md`](协议规范.md)、[`威胁模型.md`](威胁模型.md)、Go↔TypeScript 向量和跨实现测试
    与实现同一变更收口，不存在“代码先行、规范以后补”。
16. `make ci` 全绿，覆盖率门禁满足，并有 tracked 验收索引记录体验测试、浏览器矩阵和性能趋势。
