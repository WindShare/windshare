# WindShare 协议规范

> 状态：**suite `0x02` 当前生产协议的权威规范**。生产入口仅接受本文模型；协议变化必须同一变更更新
> [威胁模型](威胁模型.md)、[跨运行时向量](../core/testvectors/README.md)与实现测试。
> 真实浏览器、Windows 进程网络门和最终 CI 的证据状态见[验收索引](验收索引.md)，不能由确定性单测替代。

## 1. 适用范围与编码规则

本文是当前 suite `0x02` 生产实现的权威线合同。任何未列出的字段、标志位、消息类型和枚举值都必须拒绝；实现不得从已退役语义推断缺失字段。

### 1.1 基础密码档

suite `0x02` 固定为：

- AES-256-GCM，96-bit 随机 nonce，128-bit tag；每次 Seal 取得全新 nonce；
- HKDF-SHA256；
- Ed25519 sender 签名（public key 32B、signature 64B）；
- X25519 临时 ECDH（public/private value 32B）；
- SHA-256、HMAC-SHA256；
- `readSecret` 16B；sender Ed25519 私钥与 relay resume token 永不进入能力链接。

Ed25519 公钥只接受 RFC 8032 的 32B 压缩编码；X25519 只接受 RFC 7748 的 32B little-endian
u-coordinate。全零 X25519 shared secret 必须拒绝。私钥和临时 shared secret 在派生完成后尽快清零。

### 1.2 整数、文本与 canonical CBOR

- 固定二进制头中的整数全部 big-endian；`u32/u64` 溢出在算术或分配前拒绝。
- 结构对象使用 RFC 8949 Core Deterministic CBOR：definite length、最短整数/长度、map key 按确定性顺序。
- schema map 只使用本文列出的非负整数 key；重复/未知 key、tag、float、undefined、indefinite length、
  非最短整数、非确定性 key 顺序和尾随字节全部拒绝。
- schema 版本字段值为 `1`；未知版本返回 upgrade-required，不做宽容字段解析。
- text 是有效 UTF-8 与 NFC；路径另受 `windshare/path/v1-unicode-15.0.0` 约束。
- 超过 JavaScript safe integer 的线值仍可用 CBOR `u64`，但 `ExactSize` 和所有本地文件 offset 上限
  明确为 `2^53-1`，使浏览器文件 API 能逐字节精确表达。

### 1.3 固定宽度身份

| 身份 | 编码 | 生成/语义 |
|---|---:|---|
| `ShareInstance` | 16B | `crypto/rand`；同一能力链接刷新时换代 |
| `NodeID` / `DirectoryID` / `FileID` | 16B | `crypto/rand`；Go/TS 类型必须 kind-safe；线上不编码 kind bit |
| `DirectoryGeneration` / `ScanAttemptID` | 16B | `crypto/rand`；目录提交/扫描 attempt 身份 |
| `FileRevision` / `LeaseID` | 16B | `crypto/rand`；逐 revision 独立 |
| `ProtocolSessionID` | 16B | transcript hash 截断，见 §6 |
| `ReceiverInstanceID` | 16B | receiver 随机诊断身份；不授权、不进 AAD/输出 ownership |
| `PeerPathID` / `AttemptID` | 16B | connectivity 的非零随机协商身份；首个 offer 绑定本次 attempt |
| `OperationID` / `RecordID` | 16B | operation 随机唯一；record 为完整 sender object SHA-256 前 16B |
| `OutputSessionID` | 16B | receiver 随机本地恢复身份 |
| `RelaySessionID` | 8B | relay 外层路由；不可信、不进 E2E 身份 |
| `LaneID` / `LaneEpoch` | u32 | LaneID 是 session 内唯一的逻辑 lane（可跨 channel instance 复用）；epoch 是 ProtocolSession 全局计数，初始为 0，之后每个 FrameChannel 严格递增且跨 lane 不复用 |
| page/local block/replay sequence | u32/u64/u64 | page u32；file-local block 与方向 sequence u64 |

比较身份时使用常数时间比较。全零的随机 16B 身份保留并拒绝；`DirectoryID`/`FileID` 只由父
generation 中认证的 entry 建立，receiver 不能把任意 `NodeID` 改解释为另一 kind。
ShareInstance 在一次 root transaction 前生成，随后锚定 descriptor、catalog、revision、session 与 route，
不得在局部失败后悄悄换值。同一 capability 的 refresh 必须生成从未使用过的新值；relay/receiver 对旧值
只允许既有 session 收尾，不允许旧 descriptor/page/open 在新 instance 下重放。

`PeerPathID/AttemptID` 只属于 connectivity signaling。WebRTC channel 建立后，connectivity 只把
`FrameChannel` 及 `LaneID/LaneEpoch` 交给 session；provider、TURN、RelayNode、路径成本和 peer identity
不得进入 BlockRef、cache、OutputSession 或 catalog 身份。

### 1.4 portable 名称与输出路径

路径策略标识固定为 `windshare/path/v1-unicode-15.0.0`。每个 catalog 名称必须是有效 UTF-8、NFC 的单一
segment；拒绝空段、`.`、`..`、盘符/UNC、Cc/Cf、Windows 非法字符、尾随空格/点、设备名和 ADS 形态。
collision key 固定为 `NFC(fullFold(name))`，Unicode 版本升级不得静默改变结果。完整 root-relative 路径由
已认证的父子 NodeID 链派生，最大 256 segments、32 KiB UTF-8；receiver 不向 sender 提交本地路径。

sender 不跟随 symlink、junction、reparse 或 irregular object。receiver 在输出前重复校验同一策略，所有
mkdir/open/write/mtime/publish 都经 retained root/handle；未由当前 OutputSession 精确拥有的对象只允许
exclusive-create。词法检查用于明确错误，真正的 containment 与覆盖权限来自 retained authority，而非字符串拼接。

## 2. 能力链接、分享身份与 relay 注册

### 2.1 suite `0x02` 链接

```text
https://<frontend>/<shareId>?r=<relay-base>#<fragment>
```

`?r=` 可重复且只携非秘密 relay base hint。fragment 不进入 HTTP/WebSocket 请求；浏览器在任何 crypto、
storage 或 network 工作前捕获并从地址栏擦除。缺 fragment 时只允许本地输入拼接。

fragment 的原始字节固定为：

```text
suite(0x02,1B) || readSecret(16B) || pkHash(16B)
```

再以 strict base64url、无 padding 编码，结果必须恰为 44 ASCII chars；任何 padding、别名字母或非 33B
解码结果拒绝。`pkHash` 与 `shareId` 映射为：

```text
pkHash     = first16(SHA-256("windshare/v2 sender-key\x00" || senderPublicKey))
shareIdRaw = first12(SHA-256("windshare/v2 share-id\x00" || pkHash))
shareId    = base64url-no-pad(shareIdRaw) // 16 ASCII chars
```

因此 shareId 与 fragment 在 `ShareInstance` 换代时保持稳定；receiver 解封 descriptor 后必须同时验证
public key→`pkHash` 与 `pkHash`→URL `shareId`，再把 descriptor 的 ShareInstance 作为所有 v2 对象/session
身份锚。`?r=` 继续是多值、非秘密提示。

relay 以 shareId 查路由但必须保存并比较完整 pkHash：活跃 shareId 若映射到不同 pkHash，返回不泄露现有
身份的 `share_id_collision`，不得覆盖；sender 只能生成新 key/fragment。相同 pkHash 的 fresh 重注册仍须
通过 challenge 私钥证明，且 route 活跃时返回 `already_registered`，不能把截断碰撞或持链者当 owner。

### 2.2 relay endpoint、identity 与 challenge

v2 对每个 `?r=` relay base hint 使用 [v2 session vector](../core/testvectors/v2-session.json) 中
`v2-relay-endpoint-normalization` 的严格 authority、base-path、query、
百分号转义和边界空白规则，但有两个有意收紧：userinfo 与 fragment 一律拒绝。`http/https` 分别规范为
`ws/wss`，scheme/host/IP/default port 按向量规范化；base path 只移除一个字面结尾 `/`，再追加固定且
**不含 shareId** 的 `/v2/ws`。`REGISTER_INIT`/`JOIN` 内的 `shareIdRaw` 才是 v2 share 路由输入。

规范化产生两个不同用途的 ASCII 值：

```text
DialEndpointASCII          = ws(s)://authority/base/v2/ws[?originalCanonicalQuery]
RelayIdentityEndpointASCII = ws(s)://authority/base/v2/ws
RelayIdentity              = SHA-256("windshare/v2 relay-identity\x00" ||
                                    RelayIdentityEndpointASCII)
```

query 可承载 relay admission credential，拨号时逐字节保留，但不属于 relay 的长期密码身份；userinfo、
fragment 和 redirect 都不是 v2 凭据或发现机制。公开 relay 必须使用 `wss`（本地测试可 `ws`），并以同一
规则从运维配置的 public base URL 得到 `RelayIdentityEndpointASCII`；不得从不可信 `Host`/
`X-Forwarded-*` 临时猜测 identity。HTTP upgrade path 必须逐字节等于规范化 base path + `/v2/ws`。
`v2-relay-endpoint-normalization` 向量同时冻结 base hint、拨号值和 identity 输入，sender 与 relay 任一侧
不一致都在 challenge proof 前失败。

注册/恢复在 descriptor 上传前执行一次性 challenge：

1. sender 只在 synthetic-root/NodeRecord/预算事务提交后发送 `REGISTER_INIT`：
   `{shareId, ShareInstance, pkHash, descriptorDigest, resumeTokenHash, mode}`；
2. relay 验证 shareId 映射，生成 purpose=`register(0)` 的
   `ChallengeID(16B) + ChallengeNonce(32B) + ExpiresAt`，缓存 30 秒且只用一次；
3. sender 回 `REGISTER_PROOF`：`senderPublicKey + signature`，随后才发送 descriptor；
4. relay 验证 pkHash、签名、descriptor digest/大小及 descriptor 外层签名，原子发布 route 后回 `REGISTERED`；
5. ACK 前失败释放 route、descriptor 与 admission；客户端不得输出 ready 链接。

`mode`：fresh=`0x00`、resume=`0x01`；其他值当前均为 `unsupported_mode`。fresh 只发送 32B
`resumeTokenHash=SHA-256(token)`；resume 另在 TLS 连接内先出示 32B token，再取得
purpose=`resume(1)` 的新 challenge。token 只增加节点本地恢复约束，**不能替代私钥证明**。

注册/恢复签名 preimage：

```text
domain = mode==fresh ? "windshare/v2 relay-register\x00"
                     : "windshare/v2 relay-resume\x00"
domain || shareIdRaw(12) || ShareInstance(16) || pkHash(16) ||
descriptorDigest(32) || resumeTokenHash(32) || RelayIdentity(32) ||
ChallengeID(16) || ChallengeNonce(32) || u64_be(ExpiresAtUnixSeconds)
```

明确停止是独立的一次性 challenge：sender 先发 `STOP_INIT`，relay 绑定完整 init 签发
purpose=`stop(2)` challenge，再验 `STOP_PROOF`。签名 preimage 为：

```text
"windshare/v2 relay-stop\x00" || shareIdRaw(12) || ShareInstance(16) || pkHash(16) ||
RelayIdentity(32) || StopID(16) || ChallengeID(16) || ChallengeNonce(32) ||
u64_be(ExpiresAtUnixSeconds)
```

challenge expiry 是排他的：`now == ExpiresAt` 已过期；ledger 在验签前原子消费 challenge，因此错误签名不能
重试成 oracle。ChallengeID 不回显于 proof frame，因为当前连接 attempt 唯一拥有它，但 ID、nonce、purpose
与 expiry 全部受 ledger binding 和签名约束。现有已证明的 sender 连接仍必须完成该 exchange，不能把连接归属
当成协议所有权。

STOP 接受后 relay 以 per-route transaction 持久化 `(shareIdRaw,ShareInstance,pkHash,StopID)` tombstone；磁盘提交
不得持有全局 route lock，同 route 的新 join/resume/publish 在提交期间被 fenced，其他 route 与既有 session 的
结束不受阻塞。提交成功后 relay 必须先按 Registry 返回的精确 participant batch 删除活动 session/关闭 peer，再回
`STOPPED`；因此 WS2Y 写失败不能回滚已提交 STOP。同 StopID 并发重试等待同一 transaction，后续重试幂等；同
instance 的 fresh/resume 永久返回 stopped，且 retry=0。tombstone 永久占用
一个 `MaxRoutes` admission slot，直到未来经过认证的 refresh 协议原子替换；当前没有 refresh mode/frame，不能
用未实现的预留字节绕过 tombstone。sender 不得在明确停止后复用旧 ShareInstance。

durability store 必须显式返回 committed / definitely-not-committed / unknown，不能从普通 I/O error 猜测。只有
definitely-not-committed 可恢复原 route；unknown 立即 fail-close 该 route，且只能用相同 StopID 重试并重新验证、
sync 精确 tombstone 后转为 committed。两类 storage failure 都是 relay admission/durability error，不得伪装成 client
malformed。append 失败须 truncate 回原长度并 sync；rollback 任一步失败即 unknown。
启动时仅可在所有完整前缀 record 校验通过后截断最后一个不完整定长 record；完整 record checksum 错误必须拒绝启动。

descriptor 总长上限 16 KiB。relay 只保留 descriptor 与外层 route 状态，不保存 catalog page。
join 成功回 `RelaySessionID + descriptor`；ACK 前 join 可在 5 秒有界窗口得到 `starting`，之后 `not_found`。

### 2.3 relay v2 binary control

v2 注册控制只接受以下 big-endian binary frames；magic/长度/reserved/模式不符立即关闭注册 attempt，不回显
attacker 字段：

```text
REGISTER_INIT = "WS2R" || wire:u8(2) || mode:u8 || reserved:u16(0) ||
                 shareIdRaw(12) || ShareInstance(16) || pkHash(16) ||
                 descriptorDigest(32) || resumeTokenHash(32)                 // 116B
CHALLENGE = "WS2Q" || wire:u8(2) || purpose:u8 || reserved:u16(0) ||
             ChallengeID(16) || ChallengeNonce(32) || ExpiresAt:u64          // 64B
REGISTER_PROOF = "WS2P" || wire:u8(2) || mode:u8 || reserved:u16(0) ||
                  senderPublicKey(32) || signature(64)                       // 104B
REGISTERED = "WS2K" || wire:u8(2) || reserved(3B zero) || shareIdRaw(12) ||
               ShareInstance(16) || descriptorDigest(32)
RESUME_CREDENTIAL = "WS2T" || wire:u8(2) || reserved(3B zero) || resumeToken(32)
STOP_INIT = "WS2X" || wire:u8(2) || reserved(3B zero) || shareIdRaw(12) ||
            ShareInstance(16) || pkHash(16) || RelayIdentity(32) || StopID(16) // 100B
STOP_PROOF = "WS2V" || wire:u8(2) || reserved(3B zero) ||
             senderPublicKey(32) || signature(64)                             // 104B
STOPPED = "WS2Y" || wire:u8(2) || reserved(3B zero) || StopID(16)
JOIN = "WS2J" || wire:u8(2) || reserved(3B zero) || shareIdRaw(12)
DESCRIPTOR_UPLOAD = "WS2U" || wire:u8(2) || reserved(3B zero) ||
                     descriptorLength:u32 || descriptor sender-object bytes
DESCRIPTOR_DELIVERY = "WS2D" || wire:u8(2) || reserved(3B zero) || RelaySessionID(8) ||
                       descriptorLength:u32 || descriptor sender-object bytes
OPAQUE_ROUTE = "WS2O" || wire:u8(2) || reserved(3B zero) || RelaySessionID(8) ||
               ciphertextLength:u32 || ciphertext(1..65536)
SESSION_RETIRED = "WS2F" || wire:u8(2) || reserved(3B zero) || RelaySessionID(8) // 16B
```

`purpose` 固定为 register=`0`、resume=`1`、stop=`2`，并必须与当前 attempt 类型相同；fresh/resume mode
仍为 `0/1`。`descriptorDigest = SHA-256(descriptor sender-object bytes)`。REGISTER_INIT 长度固定，任何 mode
出现尾部或未知 mode 均拒绝。resume 的 RESUME_CREDENTIAL 必须紧跟 INIT 且在发 challenge 前验证；fresh
出现该 frame 拒绝。REGISTER_PROOF 后 sender 恰发送一个
`DESCRIPTOR_UPLOAD`；它没有尚未分配的 RelaySessionID，也不得用零值伪装 delivery。relay 只在成功 JOIN 后
以新分配的非零 RelaySessionID 发送 `DESCRIPTOR_DELIVERY`。两种 frame 的方向、magic 与长度不可互换；
digest、上限、外层 sender signature 全部验证后才发 REGISTERED。错误码固定：1 malformed、2 unsupported_mode、
3 share_id_collision、4 already_registered、5 challenge_expired、6 invalid_proof、7 descriptor_invalid、
8 not_found、9 starting、10 admission、11 stopped。错误 frame 为
`"WS2E"||wire:u8(2)||reserved:u8(0)||code:u16||retryAfterMilliseconds:u32`。只有 starting/admission 可带
`0..30000` ms retry；stopped 与其余 code 必须为 0。未知 code/尾字节拒绝。

JOIN 后 client→relay 只允许 `OPAQUE_ROUTE`；relay→client 允许 `OPAQUE_ROUTE` 或 `SESSION_RETIRED`。
WebSocket 已认证连接角色决定方向；外壳不得出现 kind、direction、
terminal、lane、operation、file 或 size 字段。ciphertext 可以是 ClientHello/ServerHello/LaneHello/LaneAccept/
LaneReject 或 OperationEnvelope 的不透明字节；relay 只校验非零 RelaySessionID、`1..65536` 长度与连接对该 route
的 sender/receiver ownership。活动与最近结束的 RelaySessionID 不得在同一 60 秒 tombstone 窗口复用。
An active owner is forwarded. During the 60-second ended-ID tombstone, frames from the exact former sender or
receiver are discarded session-locally; never-issued IDs, expired IDs, and non-owner connections are connection-level
protocol violations.
`SESSION_RETIRED` 是 relay 对已结束 session 的精确、幂等生命周期通知：命中本地 RelaySessionID 时只关闭对应
channel；本地不存在或已关闭的 ID 不得创建状态且直接忽略。畸形 frame 仍是 connection-level protocol violation；
若 relay 无法排队或写出该控制帧，必须 fail-close 该 peer link，避免已退休 channel 无限滞留。60 秒 tombstone 继续
吸收控制通知与已在途 `OPAQUE_ROUTE` 的竞态，不因通知成功而提前删除。

## 3. transport-neutral sender object

descriptor、CatalogPage、DirectoryError、FileRevisionDescriptor、BlockRecord 与 OfflineCommit 共用外壳：

```text
wireVersion:u8(2)
flags:u8(0)                    // bit0 reserved for M4 expiry-bound inventory objects
reserved:u16(0)
ciphertextLength:u32_be       // 包含 16B GCM tag，不含 nonce/signature
nonce:12B
ciphertextAndTag:ciphertextLength
signature:64B
```

固定开销 84B；`ciphertextLength >= 16`。未知 flags/reserved、长度不符、超对象上限或尾随字节拒绝。
当前在线对象的 bit0 必须为 0；M4 若启用，必须把 signed expiresAt 纳入 objectContext，且不得改变既有 domain/header
顺序。预留位已经在 AAD/signature 覆盖的 first8 header 内，因此未来无法被 offline delivery 静默翻转。
完整 object 上限固定为 descriptor/directory-error/revision/offline-commit 各 16,384B、CatalogPage 61,440B、
BlockRecord 4,194,816B；plaintext 必须非空。调用方只能通过对象类型构造带上述 ceiling 的 binding，不能传入
任意 domain/context/maxBytes 形成无界 envelope。

每类调用提供 ASCII `objectDomain` 与固定 `objectContext`：

```text
contextHash       = SHA-256(objectContext)
aad               = objectDomain || 0x00 || contextHash || first8BytesOfObjectHeader
signaturePreimage = objectDomain || 0x00 || contextHash ||
                    objectBytesThroughCiphertextAndTag
```

receiver 已知 context 后先验 Ed25519 签名再解密；descriptor 是唯一 bootstrap 例外：它先在 16 KiB
上限内解密，再以 link `pkHash` 验 public key，随后验签，任何一步失败都丢弃。持链者虽能造 AEAD，
但不能产生 sender 签名。

| 对象 | `objectDomain` | `objectContext` | key |
|---|---|---|---|
| descriptor | `windshare/v2 object/descriptor` | suite || pkHash || shareIdRaw | descriptorKey |
| catalog page | `windshare/v2 object/catalog-page` | ShareInstance || DirectoryID || u32(page) | catalogKey |
| directory error | `windshare/v2 object/directory-error` | ShareInstance || DirectoryID | catalogKey |
| revision descriptor | `windshare/v2 object/file-revision` | ShareInstance || FileID | fileObjectKey |
| block record | `windshare/v2 object/block-record` | ShareInstance || FileID || FileRevision || u64(localBlock) || u32(DataLength) | fileSegmentKey |
| offline commit | `windshare/v2 object/offline-commit` | ShareInstance | descriptorKey |

对象领域字面量不含尾 NUL；表中 `|| 0x00` 是统一域终止字节。对象 ciphertext 的逐字节重传幂等；
缓存逐出后允许用新随机 nonce 重 Seal BlockRecord，身份仍由语义 context 确定，`RecordID` 随密封字节变化。

### 3.1 HKDF tree

`K(secret,label,context) = HKDF-SHA256(secret, salt=nil, info=ASCII(label)||0x00||context, 32)`：

```text
descriptorKey  = K(readSecret, "windshare/v2 descriptor", pkHash)
catalogKey     = K(readSecret, "windshare/v2 catalog", ShareInstance)
fileObjectKey  = K(readSecret, "windshare/v2 file-object", ShareInstance || FileID)
revisionKey    = K(fileObjectKey, "windshare/v2 file-revision", FileRevision)
fileSegmentKey = K(revisionKey, "windshare/v2 file-segment", u64_be(segment))
sessionAuthKey = K(readSecret, "windshare/v2 session-auth", ShareInstance)
```

`segment = floor((LocalBlockIndex * chunkSize) / 16 GiB)`，乘法用溢出安全除法实现。每个 AES-GCM key
最多 `2^32` 次 Seal；descriptor/page/revision seal-once 仍计一次，BlockRecord 重 Seal 与多 receiver 扇出
都计入 segment key。达到上限只永久失败对应 revision，不升级为整个 share 失败。

## 4. descriptor、Catalog 与目录事务

### 4.1 ShareDescriptor schema

canonical CBOR map：

| key | 字段 | 合同 |
|---:|---|---|
| 0 | schema | `1` |
| 1 | wireVersion | `2` |
| 2 | suite | `2` |
| 3 | ShareInstance | bstr 16 |
| 4 | SyntheticRoot DirectoryID | bstr 16 |
| 5 | chunkSize | power-of-two `[1 KiB,4 MiB]` |
| 6 | capabilities | u64 bitset |
| 7 | senderPublicKey | bstr 32 |
| 8 | createdAtSeconds | non-negative int ≤ `2^53-1` |
| 9 | pathPolicy | exact `windshare/path/v1-unicode-15.0.0` |

capability bits：0 catalog、1 file-range、2 authenticated-mtime、3 ShareInstance-refresh-reserved、
4 offline-object-reserved、5 multi-relay-delivery-reserved。bit3–5 在对应里程碑前必须为 0，其余位拒绝。
多 relay hint 继续由可重复 `?r=` 表达，未来 bit5 只能声明 delivery 能力，不能改变分享身份。descriptor
不含 root entries、文件/字节/块总数、跨文件地址或 OperationID。同一 ShareInstance 只 Seal 一次并供所有 receiver/relay 重用。

### 4.2 CatalogPage 与 entry

CatalogPage map：0 schema；1 ShareInstance；2 DirectoryID；3 DirectoryGeneration(16B)；
4 PageIndex(u32)；5 Terminal(bool)；6 PreviousDigest(32B)；7 Entries(array)；8 OmittedCount(u64)。

entry 是定长 7 元数组：

```text
[Kind, NodeID, Name, ExpectedSize|null, ModifiedSeconds|null, ModifiedNanos, Precision]
```

Kind directory=`1`、file=`2`；Name 是单 segment canonical 名；目录 size 必为 null，文件 size 为
`0..2^53-1`。mtime 缺失时 `[null,0,0]`；Precision seconds=`1`/milliseconds=`2`/nanoseconds=`3`，
ModifiedSeconds 必须在 `[-(2^53-1),2^53-1]`；`ModifiedNanos` 分别必须为 0、1e6 的倍数或
`0..999999999`。输出仅在值存在且后端声明支持时恢复。
同级名称先按路径策略得到 NFC Name 与 `NFC(fullFold(Name))` collision key；collision key 重复使整个 attempt
失败。成功条目按 Name 原始 UTF-8 bytes 升序，随后才分页；kind、NodeID、OS 枚举顺序与 receiver UI
locale 都不得参与排序。synthetic-root 使用同一 collision/sort 规则。

page 0 的 PreviousDigest 是 32B zero；后续为前一页**完整 sender object bytes**的 SHA-256。PageIndex
从 0 连续；唯一 Terminal page 结束 generation，之后追加拒绝。页面最多 256 entries、完整 sender object
最多 60 KiB。超过一页只按编码大小/entry 上限切分，不受 UI 行数控制。

Unsupported reparse/symlink/irregular child 不跟随且不发 NodeID；扫描仍将其安全名称纳入 collision 检查，
终页只累计 `OmittedCount`。无法规范化或 collision、目录替换/改类型/reparse、枚举前后身份变化均使整个
attempt 失败，不发布 generation。

### 4.3 DirectoryError

map：0 schema；1 ShareInstance；2 DirectoryID；3 ScanAttemptID；4 Code(u16)；5 Retryable(bool)；
6 RetryAfterMilliseconds(uint|null)。永久错误在 share 生命周期逐字节复用；瞬时错误的 retryAfter 范围
250 ms..30 s（默认 1 s），冷却后只有显式请求产生新 attempt。
Retryable=false 必须配 null；Retryable=true 必须配范围内整数。transient-io 必须 retryable；budget 只有在
session/process 容量可释放时可 retryable，share 累计硬上限则永久；cancelled 与其余 code 不建议自动 retry。
每次实际重试使用新 ScanAttemptID。

### 4.4 root 与普通 generation 事务

分享准备的顺序不可交换：

1. 生成 sender key/readSecret/ShareInstance/IDs，打开并验证所选 roots；
2. 只读取每个 selected root 自身的 Lstat/open identity/name/size/mtime，不 `ReadDir`；
3. 在同一 CatalogStore transaction 写 synthetic-root NodeRecord、selected-root NodeRecord、root pages、
   terminal、预算 charge；
4. page/NodeRecord/terminal/spill 全部 flush 后原子 commit；失败清理未提交状态；
5. 从 committed result 构造 descriptor，随后才允许 `REGISTER_INIT`。

selected roots 数量必须在 1..4,096，名称总 UTF-8 bytes ≤1 MiB；每个 root 必须是打开后确认的 regular
file/directory，root symlink/reparse/irregular 直接使准备失败。所有 root 名先执行同级 collision/sort，任何
失败都在 transaction/REGISTER 前结束。

`REGISTERED` 后，后台可为每个目录 root 请求一次直接子项预取；优先级最低、失败静默、停止即取消，
不得递归。普通目录首次 `LIST_CHILDREN` 同样先完整写临时 transaction，再原子发布。多 receiver 同一目录
首次请求 share-scoped singleflight；最后 waiter 离开时取消尚未提交 attempt。

目录身份绑定父 generation 的 NodeRecord（root slot、root-relative locator、kind、object identity baseline），
不绑定 receiver path。已提交页面与 NodeRecord 保留到 share 结束；从内存逐出后从 metadata spill 原样读回，
不得重扫伪造 generation。catalog spill 只能存元数据，不得存文件内容。
process reopen 时只接受 durable commit record 能完整引用且校验 pages/NodeRecords/terminal/budget ledger 的
generation；没有 commit record 的 transaction 全部删除。清理失败可重试但不可见，已提交 generation 不因
清理或内存逐出消失。

## 5. FileRevision、lease 与 file-local content

### 5.1 几何

`ExactSize ∈ [0,2^53-1]`；`BlockCount = ceil(ExactSize/chunkSize)` 只在内存派生。零文件为 0 块；
非空文件末块长度由 size 派生。任何 wire `BlockCount` 字段或一个块跨两个 FileID 必须拒绝。

`BlockRef = (FileID, FileRevision, LocalBlockIndex:u64)`；range 为 canonical、排序、不重叠不相邻的
`[offset,end)` u64 集合。`ReadRange` 只请求相交块；首尾合计冗余小于 `2*chunkSize`。

### 5.2 Revision descriptor 与打开

FileRevisionDescriptor map：0 schema；1 ShareInstance；2 FileID；3 FileRevision；4 ExactSize；
5 ModifiedSeconds|null；6 ModifiedNanos；7 Precision。没有 OperationID、LeaseID 或 BlockCount。

`OPEN_REVISIONS` 一次最多 64 个 FileID，可带每项 initial ranges；response 逐项且保持请求顺序：

```text
success = [FileID, 0, RevisionObjectBytes, LeaseID, LeaseTTLMilliseconds, RenewAfterMilliseconds]
error   = [FileID, 1, RevisionErrorCode, Retryable, RetryAfterMilliseconds|null]
```

success 的 `LeaseTTLMilliseconds/RenewAfterMilliseconds` 固定为 `120000/60000`；其他取值拒绝。
revision error 的 `Retryable=true` 必须携 `1..30000` ms 整数，`Retryable=false` 必须携 `null`。
每个 success 是独立 lease；批量不共享续期/释放/失败。RevisionStore 在发 descriptor 前必须：验证 catalog
private candidate + ExpectedSize、建立稳定 source、完成句柄与 quota admission、注册 drift invalidation。
`VersionCandidate` 是 sender-private provider value：Windows 至少含 volume/file ID、size 与 last-write baseline；
POSIX 等于本节 MutationToken。它必须从父 generation 的已打开对象取得，不能只保存路径；open 时同一
root slot/relative locator 必须重新打开并匹配 candidate。相同 FileID 的并发首次 open 由 share-scoped
singleflight 合并 source/admission，成功请求各发独立 LeaseID；最后 waiter 取消才可撤销尚未发布的建立。

稳定 source 首选排除写入的已打开句柄；否则每次读前后都必须从同一已打开对象读取冻结的
MutationToken。平台合同固定为：Windows 本地 NTFS/ReFS 使用不含 `FILE_SHARE_WRITE` 的 CreateFile
handle，并以 volume/file ID 防路径替换；POSIX 本地 regular file 使用
`device + inode + size + mtime(ns) + ctime(ns)`。POSIX 同 token 的原地改写是已接受残余，见威胁模型；
当前 provider 支持矩阵为 Linux/macOS 本地 regular file，其他 POSIX 在取得同等 stat 字段后即可加入；
缺字段、时间精度不足、network/pseudo filesystem 或未列平台返回 `unsupported_stability`。任何后端都
不得以内容临时文件或完整复制兜底。

lease TTL 120 s；剩余 ≤60 s 才接受 renew；单 lease 总寿命最多 2 h。最后 lease 结束后 revision 保留
120 s resume grace；期间句柄/source/cache identity 不变且可新建 lease。活跃 lease 或 grace 不因新 admission
被驱逐；过期回收后再次打开重新验证 candidate 并产生新 FileRevision，旧 resume 失败。
在线 lease 的 TTL/RenewAfter 全是相对时长，sender 单调时钟为唯一权威；receiver wall clock、Unix 时间与
relay 时间不得参与过期判断。进程正常停止必须先终止相关 operation，再关闭全部 lease/source handle；崩溃时
由 OS 关闭 handle，relay route grace 不保留 revision，恢复后的进程必须重新验证 candidate。

源在读前/中/后变化：永久标记 revision drifted，关闭全部 lease，废弃其 sealed cache，向每个相关 operation
发 sender-signed file error。receiver 中止该 file transaction 并丢弃此 revision 的 DurableRanges；其他文件继续。

### 5.3 BlockRecord

BlockRecord map：0 schema；1 ShareInstance；2 FileID；3 FileRevision；4 LocalBlockIndex；5 Data(bstr)。
Data 长度必须等于由 descriptor 派生的块长。完整 sender object 最大 `MaxChunkSize + 512 = 4,194,816B`。
同 revision 的所有 Data 只能来自 RevisionStore 的同一 admitted object。cache miss 重读仍须在 source 变化
校验内完成，并用新 nonce 重 Seal。

## 6. ProtocolSession transcript、lane 与 OperationEnvelope

### 6.1 sender-authenticated transcript

receiver 从 descriptor 验证 sender key 后生成 `ReceiverNonce(32B)`、X25519 ephemeral key 与诊断用
ReceiverInstanceID。ClientHello canonical binary body 为：

```text
"WS2C" || wire:u8(2) || ShareInstance(16) || ReceiverInstanceID(16) ||
ReceiverNonce(32) || ReceiverX25519Public(32)
```

随后附 `ClientProof = HMAC-SHA256(sessionAuthKey,
"windshare/v2 client-hello\x00" || SHA-256(body))`。sender 在任何 catalog/revision admission 前验证 proof，
并对 `(ShareInstance,ReceiverNonce,receiverPublic)` 的 hash 保留 5 分钟 replay tombstone。

ServerHello body：

```text
"WS2S" || wire:u8(2) || SHA-256(ClientHelloWithProof) || SenderNonce(32) ||
SenderX25519Public(32) || u32_be(initialLaneID) || u32_be(initialLaneEpoch=0)
```

附 Ed25519 signature：`"windshare/v2 server-hello\x00" || SHA-256(ServerHelloBody)`。

```text
transcriptHash    = SHA-256(ClientHelloWithProof || ServerHelloWithSignature)
ProtocolSessionID = first16(SHA-256("windshare/v2 protocol-session\x00" || transcriptHash))
dh                = X25519(localPrivate, peerPublic)
handshakeSecret   = HKDF-SHA256(dh, salt=sessionAuthKey,
                    info="windshare/v2 handshake\x00" || transcriptHash, 32)
receiverToSender  = K(handshakeSecret,"windshare/v2 traffic/receiver-to-sender",transcriptHash)
senderToReceiver  = K(handshakeSecret,"windshare/v2 traffic/sender-to-receiver",transcriptHash)
```

新 ProtocolSession 使旧 OperationID、terminal、lane attach 全部失效。只有用当前 traffic key 认证的
LANE_ATTACH 可以延续 session；RequestedLaneID=0 时分配从未使用的非零逻辑 LaneID，非零时只可重附该
session 已拥有的 LaneID。初始 FrameChannel 使用 LaneEpoch=0，
之后每次 attach 从 ProtocolSession 全局计数器取得前值+1；即使 LaneID 不同也不得复用 epoch。
接收 grant/hello 时只要求 epoch 严格大于**该 LaneID** 上次接受值，允许 gap；不得用跨 lane 的全局
接收 watermark 拒绝乱序到达，因为已签发但丢失的 grant 会烧掉计数值，并发 lane 也可乱序建立。
全局 counter 到 `2^32-1` 后拒绝新 grant 并建立新 ProtocolSession，绝不 wrap。
RelaySessionID 改变不参与以上任何派生。ReceiverInstanceID 虽受 transcript 完整性保护，但不授权、不进
envelope AAD、不进 OutputSession identity。

### 6.2 OperationEnvelope

每个 FrameChannel 只有一个 ProtocolPump 调用 `Recv`，一个 SessionWriter 调用 `Send/SendTerminal`。
业务服务只能向 router/writer 提交消息。

```text
wireVersion:u8(2)
flags:u8                 // bit0 direction: 0 receiver→sender, 1 sender→receiver
reserved:u16(0)
sequence:u64_be
ciphertextLength:u32_be  // 包含 GCM tag
nonce:12B
ciphertextAndTag
```

固定开销 28B（tag 另在 ciphertext 内）；整帧 ≤65,536B，故 canonical plaintext ≤65,492B。
AAD：

```text
"windshare/v2 operation-envelope\x00" || wire || direction ||
ShareInstance || ProtocolSessionID || u32_be(LaneID) || u32_be(LaneEpoch) ||
u64_be(sequence) || u32_be(ciphertextLength)
```

每方向/LaneEpoch sequence 从 0 严格递增；FrameChannel 可靠有序，因此 replay window 固定为下一序号 1 帧，
重复、跳号、回退、跨 epoch 重放为 ProtocolSession error。`2^64-1` 用后关闭该 lane，不得 wrap；每次 Seal
仍使用新随机 nonce。

Ordinary plaintext is the canonical CBOR map `0 MessageKind(u8); 1 OperationID(bstr16|null); 2 Body`.
`OperationID` MUST be fresh within a ProtocolSession and is retained for 30 s after operation terminal.
`SESSION_TERMINAL` MUST use null; every other operation request/response/error/cancel MUST use a non-null ID. The ID is
wire correlation, not local lifecycle authority. A receiver peer attempt issues an opaque local binding/generation before
offer creation; this value is not transmitted. Terminal import, cancellation, and cleanup MUST validate that exact
binding, so delayed work cannot affect another local generation even if an adapter repeats a wire ID.

| kind | value | Body |
|---|---:|---|
| LIST_CHILDREN | 1 | `[DirectoryID, Generation|null, PageIndex]`；一个 final CATALOG_RESULT/ERROR，扫描中可有 progress |
| CATALOG_RESULT | 2 | sender control，恰好一个 page/error object bytes，结束 LIST_CHILDREN |
| OPEN_REVISIONS | 3 | ≤64 项 `[FileID, InitialRanges]` |
| OPEN_RESULTS | 4 | sender control，逐项 success/error，项数/顺序与 request 相同并结束 operation |
| RENEW_LEASE | 5 | `[LeaseID]` |
| RELEASE_LEASE | 6 | `[LeaseID]`；重复 release 幂等 |
| REQUEST_BLOCKS | 7 | `[LeaseID, LocalBlockIndex[]]`，最多 256 indices |
| BLOCK_FRAGMENT | 8 | §7 特殊 binary plaintext |
| CANCEL | 9 | `[ReasonCode]`；重复 cancel 幂等 |
| OPERATION_ERROR | 10 | sender control 或 receiver AEAD error |
| SESSION_TERMINAL | 11 | sender-signed，无 OperationID |
| LANE_ATTACH | 12 | established lane 上请求/返回一次性新 lane grant |
| SCAN_PROGRESS | 13 | sender control `[ScanAttemptID,DiscoveredEntries]`；同 operation 零或多条 |
| OPERATION_COMPLETE | 14 | sender control `[ResultCount]`；结束 release/block operation，renew 禁止 |
| LEASE_RESULT | 15 | sender control `[LeaseID,LeaseTTLMilliseconds,RenewAfterMilliseconds]`；结束 renew |
| PEER_OFFER | 16 | receiver→sender；开启一个 E2E peer negotiation operation |
| PEER_ANSWER | 17 | sender control；同 operation 一个语义答案，精确重放幂等丢弃，不结束 candidate trickle |
| PEER_CANDIDATE | 18 | 双向；绑定 PEER_OFFER 的 path/attempt identity；canonical 精确重放幂等丢弃 |

InitialRanges 是 canonical `[Start,End)` u64 数组：排序、不重叠、不相邻，每文件最多 256 段、每个 OPEN
合计最多 1,024 段。REQUEST_BLOCKS 的 index 唯一且严格递增。一个 operation 恰有一个 final：
CATALOG_RESULT、OPEN_RESULTS、LEASE_RESULT、OPERATION_COMPLETE、OPERATION_ERROR 或被 CANCEL；final 后 30 s 内同
OperationID 的重复 final/fragment 丢弃，冲突内容是 session error。REQUEST_BLOCKS 为每个 index 交付一个
完整 record 后以 `OPERATION_COMPLETE(ResultCount=request count)` 结束，故缺块不能被静默当成功。
RENEW_LEASE 只以 LEASE_RESULT 或 OPERATION_ERROR 结束；RELEASE_LEASE 以 OPERATION_COMPLETE(0) 结束。
PEER_OFFER operation 接受一个 PEER_ANSWER 与任意有界 PEER_CANDIDATE；由 receiver CANCEL、sender
OPERATION_ERROR 或 SessionTerminal 结束。answer 不是 final，candidate 可先于 answer 到达。相同 authenticated
SemanticBody 的 answer 重放不再次交付或推进答案基数；不同的第二个 answer 是 session error。

sender control 的签名外壳固定为 canonical CBOR map：

```text
SignedControl   = {0:1, 1:SemanticBody, 255:Signature(bstr64)}
UnsignedControl = {0:1, 1:SemanticBody}
```

signature 不是 SemanticBody 的字段。验签成功后 session 返回 field 1 的 canonical encoding，因此业务 body
可以是 map 或 array，不会因签名层丢失类型。SemanticBody 固定为：

```text
CATALOG_RESULT     {0:1, 1:ObjectBytes}
OPEN_RESULTS       {0:1, 1:RevisionOpenResult[]}
OPERATION_ERROR    {0:1, 1:Scope(u8), 2:Code(u16), 3:Retryable,
                    4:RetryAfterMilliseconds|null, 5:Message(non-empty NFC UTF-8 <=512B)}
SESSION_TERMINAL   {0:1, 1:SessionCode(u16), 2:Message(non-empty NFC UTF-8 <=512B)}
LANE_ATTACH grant  {0:1, 1:1, 2:LaneID, 3:LaneEpoch, 4:AttachNonce(16B)}
SCAN_PROGRESS      {0:1, 1:ScanAttemptID, 2:DiscoveredEntries(u64)}
OPERATION_COMPLETE {0:1, 1:ResultCount(u32)}
LEASE_RESULT       {0:1, 1:LeaseID, 2:LeaseTTLMilliseconds, 3:RenewAfterMilliseconds}
PEER_ANSWER        SignalBody（§6.3）
PEER_CANDIDATE     SignalBody（§6.3）
```

LANE_ATTACH request Body 为 `{0:1,1:0,2:RequestedLaneID}`；response 与 request 共用非空 OperationID。SCAN_PROGRESS 的
DiscoveredEntries 单调，不能当 page 或最终总数。receiver 发 OPERATION_ERROR 时 Body 使用同一 error
schema 且没有 SignedControl 外壳；它只有 traffic-key AEAD 权威，不能伪装 sender error。
error Scope 固定：1 ProtocolSession、2 directory、3 revision、4 block、5 peer；OPERATION_ERROR 只允许 2–5 且
Code 必须落在对应 §8 范围，scope 1 必须使用 SESSION_TERMINAL/本地 terminal。CANCEL ReasonCode 固定：
1 user、2 superseded、3 output-abort、4 timeout、5 lane-race；receiver→sender CANCEL 无 sender signature，
sender 取消用签名 OPERATION_ERROR(cancelled)，不得发送无签名 CANCEL 冒充 receiver 意图。

sender 发出的非 transport-neutral 控制统一携 Ed25519 signature。签名 preimage：

```text
controlDomain || 0x00 || ShareInstance || ProtocolSessionID || u32(LaneID) || u32(LaneEpoch) ||
direction(1) || u64(sequence) || MessageKind || OperationID-or-16-zero ||
SHA-256(canonical UnsignedControl)
```

operation domain=`windshare/v2 control/operation`；terminal=`.../session-terminal`；lane attach=`.../lane-attach`。
AEAD 合法但 sender signature 错误终止 ProtocolSession；AEAD 无法认证时只产生无 ID 本地 terminal，不向对端
反射 attacker-controlled OperationID。

### 6.3 E2E peer signaling reservation

receiver/browser 是 offerer。三个 connectivity payload 使用 Core Deterministic CBOR 定长数组：

```text
Offer/Answer = [1, PeerPathID:bstr16, AttemptID:bstr16, SDP:tstr]
Candidate    = [1, PeerPathID:bstr16, AttemptID:bstr16, Candidate:tstr,
                SDPMid:tstr|null, SDPMLineIndex:uint16|null,
                UsernameFragment:tstr|null]
```

两种 identity 都必须非零。首个 PEER_OFFER 的 pair 绑定该 operation 内全部 answer/candidate；替换一个字节、
旧 AttemptID 重放或跨 PeerPath 拼接都终止该 negotiation。SDP 必须非空、有效 UTF-8/NFC 且 ≤60 KiB；
Candidate 非空且 ≤4 KiB；SDPMid/UsernameFragment 各 ≤256B。receiver 的 PEER_OFFER/PEER_CANDIDATE Body
直接是 SignalBody。sender 的 PEER_ANSWER/PEER_CANDIDATE 把 SignalBody 放入统一 SignedControl field 1；
验签后 session 只把该 field 的 canonical bytes 交给 connectivity codec。

每个 attempt 只允许 receiver/offerer 创建一条规范的有序可靠 DataChannel。远端额外创建 channel、Open
失败或 lane admission 失败都只终止该 peer path，并按 channel-admission 分类；不得关闭健康
ProtocolSession/relay。

A well-formed sender-signed `OPERATION_ERROR` with peer scope and the exact admitted operation remains attempt-local.
A signature proves sender identity, not message kind, failure scope, or peer binding:
malformed/cross-scope signed failures, conflicting signed answers/finals, continuation-authority violations, and signed
answer/candidate binding mismatches seal a session-unsafe consequence. Invalid, zero, or cross-binding adapter results
are local adapter-contract failures and cannot mint that consequence.

接收方向的每个 OperationID 代际只有一个 continuation authority；active queue 与 30 s tombstone 共享
request kind、peer binding 及 answer/final 指纹。
canonical candidate 必须在 operation/session 队列计费、candidate quota 与 `AddICECandidate` 前去重；
浏览器本地 callback 同样在 outbound event/quota 前由 negotiation 去重，signaling adapter 不再维护第二套 ledger。
精确重放不占预算，不同 candidate 恰好计数一次，且只交付一个 overflow representative 触发 operation-local
candidate-limit。attempt 结束时清除 candidate set；墓碑仍丢弃 binding-compatible late candidate 和精确 answer/final，
但错误 response kind、binding 变化或冲突 answer/final 仍是 session error。
Because transport acceptance does not prove remote routing state, an ambiguous initial `PEER_OFFER` retains its exact
generation-bound replay until operation end; every lane migration queues that replay before dependent
`PEER_CANDIDATE` on the same replacement FIFO, and the remote drops exact request replays idempotently. A migrated
`CANCEL` may arrive first and authoritatively creates the operation's preemptive tombstone.

协商失败以 scope=peer 的 sender-signed `OPERATION_ERROR` 结束该 operation：`0x5001` negotiation、
`0x5002` timeout、`0x5003` candidate-limit、`0x5004` channel-admission。四者均为永久错误，
必须 `Retryable=false` 且 `RetryAfter=null`；健康 relay/lane 与 ProtocolSession 不受影响。

Receiver terminal authority is sealed by the exact-operation workflow and is never inferred from an error class. The
first transition owner/provenance remains fixed, while the proven consequence joins monotonically as
`operation-only < session-unavailable < session-unsafe`. `Terminate` contributes evidence to the same join; publication
waits for any in-flight `Receive` and exact cleanup, then exposes one immutable outcome and text-free trace snapshot.
Bounded residual classification is diagnostic only: unknown or truncated graphs can neither mint nor erase authority.

这些消息始终在 OperationEnvelope/WS2O 内，relay 不见 SDP、candidate 或 peer identity。PeerPathID/AttemptID
不授权内容，也不参与 ProtocolSession、block/cache/output 身份；DataChannel Open 后仍须先取得 lane grant，
完成 WS2A/WS2B，才以 LaneID/LaneEpoch 附加到现有 ProtocolSession。provider、TURN 与路径成本不进本合同。

### 6.4 新 FrameChannel attach

receiver 必须先在已认证 lane 取得上面的 sender-signed grant；grant 30 s 过期且只消费一次。新 channel 的
第一帧不是 OperationEnvelope，而是固定 LaneHello：

```text
"WS2A" || wire:u8(2) || ShareInstance(16) || ProtocolSessionID(16) ||
LaneID(u32) || LaneEpoch(u32) || GrantOperationID(16) || AttachNonce(16) ||
Proof(32)
Proof = HMAC-SHA256(receiverToSender,
        "windshare/v2 lane-hello\x00" || all preceding LaneHello bytes)
```

sender 在任何 queue/reassembly admission 前验证 session、epoch 严格大于该 LaneID 已接受值、未消费 grant、
nonce 与 proof，原子消费
grant，再返回：

```text
body = "WS2B" || wire:u8(2) || SHA-256(LaneHello) || SenderNonce(16)
ack  = body || Ed25519.Sign(senderKey,
       "windshare/v2 lane-accept\x00" || SHA-256(body))
```

通过 HMAC 认证但不能接纳的 candidate 返回固定 signed rejection；HMAC 无效、畸形或未知 key material 只能
silent-close，不能反射 attacker 字段：

```text
body = "WS2N" || wire:u8(2) || code:u8 || reserved:u16(0) ||
       SHA-256(LaneHello)(32) || retryAfterMilliseconds:u32
reject = body || Ed25519.Sign(senderKey,
         "windshare/v2 lane-reject\x00" || SHA-256(body))
```

code 固定为 1 unknown-session、2 stale-epoch、3 grant-expired、4 grant-consumed、5 admission-limited、
6 stopping、7 grant-mismatch；只有 code 5 可携 `0..30000` ms retry，其余必须为 0。WS2A/WS2B/WS2N 长度分别
109/117/108B；所有 reserved、未知 code、尾字节与全零 identity/nonce 均拒绝。

receiver 验签后双方才把 channel 绑定到该 LaneID/Epoch，两个方向 sequence 从 0 开始。错误只关闭候选
channel；grant 30 秒到期，消费或到期后再保留 30 秒有界 tombstone，重复 hello 必须得到 signed
grant-consumed/expired rejection，不能再次绑定。没有仍有效的既有 lane 或预取 grant
时必须建立新 ProtocolSession，不能仅凭 RelaySessionID 复活旧 session。

SessionWriter control queue 上限 256 frames/4 MiB，data queue 1024 frames/64 MiB；每发送最多 8 个 data
frame 必须重查 control。terminal 预占 control slot，接受后丢弃排队 data/拒绝新消息，并经
`SendTerminal` 保证先于 Recv close 可见。数据背压不得停止 pump 接收 CANCEL。

## 7. BLOCK_FRAGMENT 与重组

BLOCK_FRAGMENT 解密后的 plaintext 不用 CBOR，固定为：

```text
codec:u8(1) || kind:u8(8) || flags:u8(bit0=last) || reserved:u8(0) ||
OperationID(16) || RecordID(16) || FragmentIndex:u32 || FragmentCount:u32 ||
TotalLength:u32 || PayloadLength:u32 || Payload
```

头 52B；结合 envelope/GCM，`Payload <= 65,440B`。`FragmentCount ∈ [1,128]`，
`TotalLength <= 4,194,816B`，count 必须等于 `ceil(TotalLength/MaxFragmentPayload)`（末片除外），last
仅能出现在 `index=count-1`。在 envelope AEAD、所有整数和 TotalLength 上限通过前不得分配重组缓冲。

重组 key=`ProtocolSessionID + OperationID + RecordID`。相同 index+bytes 幂等；相同 index 不同 bytes、
两个 last、总长/count 冲突或完整长度不符只终止 block operation。缺片 15 s timeout；CANCEL/完成后保留
30 s tombstone，迟到片丢弃。完整 object 的 SHA-256 前 16B 必须等于 RecordID，再验 sender object signature。

## 8. 错误域与资源预算

### 8.1 error code

| 范围 | code | 语义/作用域 |
|---|---|---|
| session | `0x1001..1008` | auth、replay/sequence、malformed、version、budget、sender signature、illegal terminal、sender stopped；结束 ProtocolSession |
| directory | `0x2001..2008` | stale、permission、collision、too-wide、budget、permanent-io、transient-io、cancelled；跳过该 subtree |
| revision | `0x3001..3008` | stale、not-found、unreadable、unsupported-stability、quota、lease-expired、drift、invalid-lease；只结束 file revision |
| block | `0x4001..4006` | invalid-ref、out-of-range、object-auth、fragment-conflict、timeout、cancelled；只结束 block/range operation |
| peer | `0x5001..5004` | negotiation、timeout、candidate-limit、channel-admission；只结束 peer negotiation operation，全部永久不可重试 |

directory error 必须签名且带 attempt/retry 字段；revision/block error 是 sender control 时必须签名。
`OPERATION_ERROR` 的 `Retryable=true` 必须携带闭区间 `1..30000` ms 的整数延迟；
`Retryable=false` 必须携 `null`。Go/TypeScript 都在统一 control validator 拒绝 0、超上限与永久错误带延迟。
本地输出错误不上 wire。share/process 本地 admission 不足只拒绝新 scan/open/lane，不驱逐活动资源。

| 名称 | 值 |

| active leases session/share/process | 32 / 256 / 1,024 |

预算 charge 在工作/内存/磁盘/句柄取得前原子执行，释放幂等；ProtocolSession 重连可换 lane 但不重置

## 9. Selection、连接分级与 OutputSession

### 9.1 SelectionRules 与 JobOutcome

SelectionRules 是 `Default(select|exclude)` 加 DirectoryID/FileID override。节点取最近祖先 override；
FileID override 最具体。展开状态不参与选择。点击下载时复制冻结规则，后续 UI 修改只影响下一 job。
规则是按 opaque ID 的 map，不接受重复 key；从 synthetic root 向下应用最后一个目录 override，目标
FileID override 最后应用，因此显式文件可重新包含在排除目录下，显式目录可覆盖祖先。尚未发现的后代
继承当时最近目录规则，展开/折叠/搜索绝不修改规则。

CLI `--only` uses a mutually exclusive, bounded `SelectionByCatalogPath` job mode. Its paths are matched only
against authenticated catalog names during the job walk; they are not wire fields or local-filesystem authority.
`SelectionByNodeID` remains the opaque ID-map protocol model. Every explicit selected ID/path target must be found
by a complete failure-free walk before `Small`/`Succeeded`; incomplete branches remain `Unknown`.

TransferJob 对每个已提交 generation 立即排入选中文件并继续递归选中目录。目录失败记录 typed failure、
跳过未知 subtree、继续独立分支。结局只有：无失败且完成=`Succeeded`；至少一个目录/文件失败且其余结算=
`CompletedWithErrors`；用户取消、session terminal 或无隔离输出失败=`Aborted`。

SelectionMeasure 单调记录 discovered file count/bytes 与 discovery 状态：

```text
Large   = fileCount >= 30 || totalBytes >= 8 MiB
Small   = discoveryTerminalSuccess && fileCount < 30 && totalBytes < 8 MiB
Unknown = otherwise (尚未越过阈值，且未完成或存在 directory failure)
```

判定按上表顺序，Large 是吸收态：阈值一旦达到，即使发现尚未终止或之后某目录失败也不回到 Unknown。
空选择完整发现为 Small。目录不计数、空文件计数；整数加法溢出直接 Large。

目录浏览只使用已建立的 relay browsing lane，不启动 SDP/ICE，也不建立 0/8 计时器。preview click 的第一条
同步语句独立记录 `previewT0` 并立即启动 P2P；它没有 output picker，目录发现、revision open 与媒体探测均
不得延后启动，application relay 最迟在 `previewT0+8 s` 入池（P2P 已明确失败时立即入池）。

download click 的第一条同步语句另行记录 `downloadT0`；同一 handler 同步启动 output picker，并在不 await
discovery/picker 的情况下启动 P2P。Small 立即把浏览 relay lane 加入 LaneSet；Large/Unknown 在 P2P 失败时
立即加入，否则最迟 `downloadT0+8 s` 加入 application relay。preview/download 各自拥有计时器和取消状态，
不得复用 picker 完成或另一作业的 t0。TURN 仅保留各自 `0..8 s` 中的插入 hook，本合同不冻结其时刻。

BlockBroker/cache key 固定为 `ShareInstance+FileID+FileRevision+LocalBlockIndex`，不含 consumer/job/lane；
preview、download、reconnect 可合并同一上游读取，但各自保留 cancel/completion/error。调度顺序为
control/terminal > catalog/open/cancel > 显式 preview range > active download > root prefetch；每级有界且低级
仍获公平进度。preview cache 不是 DurableRanges，不能授权输出 resume。

### 9.2 OutputSession 能力与 file transaction

OutputSession 报告：`DurabilityLevel(None=0, ProcessRestart=1, PowerLoss=2)`、random-write、file-failure-
isolation、mtime。接口语义固定：

```text
EnsureDirectory / FinalizeDirectory
BeginFile(binding) -> FileTransaction + VerifiedDurableRanges
FileTransaction.WriteRange / Checkpoint / Commit / Abort
FinishJob(JobOutcome) / AbortJob
```

binding=`BackendID + OutputSessionID + ShareInstance + FileID + FileRevision + ExactSize +
canonical path-or-persistent-handle digest + owned output object identity`。DurableRanges 是 canonical、排序、
不重叠不相邻 `[offset,end)`；只有 BeginFile/Checkpoint 返回值可进入 have-state。

checkpoint 顺序不可交换：

1. 完成目标 range writes；
2. 按声明级别 flush data（ProcessRestart 至少对新进程可见；PowerLoss 使用 file/device sync）；
3. 写完整新 journal temp，包含 schema/binding/data-generation/ranges/checksum；
4. flush journal；PowerLoss 再原子 replace 并 sync parent directory/等价平台边界；
5. 重新打开/验证 journal 与 output identity 后才返回 DurableRanges。

崩溃在任一步只允许恢复上一次已验证 checkpoint。BeginFile 验 schema/backend/session/handle-or-path/size/
FileID/revision/owned object；不符只废弃该文件状态。journal 校验和防意外损坏；同 OS 账户恶意篡改是非目标。
Commit 必须先执行覆盖最终完整 ranges 的同级 checkpoint，再 exclusive/atomic publish 最终文件、恢复受信
mtime，并重新验证最终 identity；在此之前文件仍是未提交 transaction。Abort 只删除该 OutputSession 精确
owned 的 staging/journal，不能按名称猜测或覆盖既有对象。

直接 single-file/ZIP stream 报 None 且不声明 file isolation。single-file stream 首字节输出后失败升级
`AbortJob`。ZIP 以 **member start** 为不可回滚边界：某成员的 local-file-header 任一字节输出前，该文件失败可
跳过并记录到 `CompletedWithErrors` 清单；header 或该成员任一后续字节一旦输出，该成员失败就必须
`AbortJob`，不得继续写 central directory 冒充有效完成。空文件也以 local-file-header 首字节视为 started。
FSA/OPFS 只有在持久 handle/存储、重授权、reopen 身份和上述 checkpoint 实测全部成立时才可报
ProcessRestart；否则必须报 None。当前 Web 后端均不得声称 PowerLoss。CLI osfs 可在 file+directory sync
均成立的平台报 PowerLoss，否则降为 ProcessRestart。receiver staging 是输出能力，不违反 sender 禁止副本。
Web 标准 FSA/OPFS 不提供受信 mtime 设置，均报告 mtime=false；CLI osfs 只有 root-confined 平台调用成功时
报告 true。FSA/OPFS staging 可报告 random-write/file-failure-isolation，direct stream/ZIP 两项均为 false。

Web OPFS 默认 job/process staging admission 为 8/16 GiB，并额外保留 `max(512 MiB, quota*10%)`；不足时
选择 FSA/stream 或 typed quota error。输出同时打开 file transaction 默认上限 32。

## 10. 当前在线生命周期与 M4 离线预留

当前生产只实现 `LiveOnly`：

```text
Preparing -> LiveOnly -> Stopping -> Stopped
```

- Preparing 失败从未 REGISTER；LiveOnly 的服务源是在线 sender。
- 显式 Stop/退出拒绝新 join，发送 sender-signed SESSION_TERMINAL，取消 scan/revision/lanes，完成
  `WS2X -> WS2Q(purpose=2) -> WS2V -> WS2Y`，并在 STOPPED 前持久化永久 stopped tombstone；不得进入
  60 s grace。
- 意外网络/进程消失才保留 relay route 60 s；期间 join 返回 starting/not-found，不把旧 sender object
  冒充活动 sender，并立即结束/墓碑化既有 RelaySessionID。恢复仍需 purpose=1 新 challenge、resume token 与
  私钥 proof；显式停止的 instance 永不走 resume。

当前没有生产 offline uploader、inventory commit 或寄存交付路径。M4 若实现，可把状态机扩展为
`LiveOnly -> OfflineUploading -> OfflineCommitted`，但必须满足：未提交 staging/inventory 对 receiver 不可见；
commit 前 Stop 取消上传并清理 staging；只有 server 验证完整 inventory 并原子返回 `COMMIT_ACK` 后才能进入
OfflineCommitted。寄存服务只能交付原始 transport-neutral sender objects，不能保存/重放 ProtocolSession
envelope，也不能获得 readSecret/sender private key。

M4 预留 OfflineCommit map：0 schema；1 ShareInstance；2 InventoryRoot(32B)；3 ExpiresAtSeconds；
4 DeliveryIdentity(32B)；5 ObjectCount(u64)；6 TotalCipherBytes(u64)。InventoryRoot 是按 object digest
字节序排序后的二叉 Merkle root（叶=`SHA-256(0x00||objectDigest)`，节点=`SHA-256(0x01||left||right)`，
奇数复制末叶）。inventory 计数/字节/root 覆盖本次提交的 descriptor/page/error/revision/block 对象，不含
OfflineCommit 自身；空 inventory 拒绝。它由 sender object 域签名；M4 只能提交列出的 sender object 与
inventory，不能保存 session envelope。

### 10.1 可执行证据

跨运行时合同向量由 `core/internal/protocolcontract` 生成，TS 独立重建/验证。至少覆盖：suite/link/shareId、
HKDF、descriptor/page/directory-error/revision/block/offline sender object、purpose-bound registration/resume/stop
binary control 与 proof、WS2O、WS2A/B/N、transcript/direction keys、operation error/terminal/lane attach envelope、最大 BlockRecord 的
65 片重组、PeerPathID/AttemptID 信令、hostile canonical CBOR、错误/预算/selection/output/lifecycle 状态。
修改任一 identity axis 必须失败。

平台 source/output/catalog 验收覆盖：Windows write-exclusion/path
replacement、POSIX token/unsupported backend fail-closed、transaction commit/crash cleanup、fragment
budget、checkpoint crash cuts、stream abort escalation 与 OPFS reopen。FSA 在重授权/identity spike 完成前
只能报告 None；当前 Web 后端均不声明 PowerLoss。所有生产变更必须继续以这些向量、状态模型与验收索引为门。
