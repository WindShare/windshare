# WindShare 协议规范

> 状态:**当前已实现协议的权威规范**。未来范围与验收门见 [`后续计划.md`](后续计划.md)；
> 协议变更落地时必须同步更新本文、威胁模型与跨实现测试向量。

## 1. 链接与能力模型(§6.3)

### 1.1 链接格式

```
https://<前端域>/<shareId>?r=<中转主机>#<base64url(suiteByte ‖ readSecret)>
                 │           │             └── fragment,浏览器不发送(能力令牌)
                 │           └── 可选:第三方中转提示(非秘密,明文);缺省 = 同源/官方中转
                 └── 路径 = shareId,发给中转(定位会话)
```

- `readSecret`:**16B / 128-bit** 随机(`crypto/rand`)。fragment = `suiteByte(1) ‖ readSecret(16)`
  → base64url 无填充 ≈ **23 字符**。
- `suiteByte`:套件/版本位。`0x01` = {AES-256-GCM, HKDF-SHA256, 128-bit readSecret, 随机 nonce}。
  签名档 `0x02` 见 [`后续计划.md` §3.2](后续计划.md#32-签名套件-suite-0x02)；
  密码档/256-bit 档/ChaCha 档皆为新 suite 号。
- `shareId`:**9B / 72-bit** 客户端随机,base64url,**纯路由句柄,不进任何密钥派生**;
  中转遇活跃碰撞则拒绝、客户端重生成。
- `?r=` **按多值设计**(`?r=a&r=b`):M1 仅实现单值,但解析自始按列表处理。
- **分离密钥(split-key,可选;默认单链)**:发送侧去掉 fragment,输出"裸链接 + 密钥串"
  两段经不同渠道分发;接收侧(网页输入框 / CLI `--key`)手动补入(§6.10)。

### 1.2 密钥派生(HKDF-SHA256)

salt 一律为空,label 取精确 ASCII 字面字节、无结尾 NUL:

```
manifestKey = HKDF(readSecret,            "windshare/v1 manifest", 32)
streamKey   = HKDF(readSecret,            "windshare/v1 stream",   32)
segKey(s)   = HKDF(streamKey, "windshare/v1 seg" ‖ u32_be(s),      32)
```

`readSecret` 是唯一秘密;keys 的每分享唯一性由 readSecret 随机唯一保证(故 salt 留空);
`shareId` 不入密钥。读写分离:host 另持绝不进链接的 `writeSecret`(M1 占位,M4 实装)。

### 1.3 AEAD 统一不变量

> **每一次 GCM 加密都用一枚全新随机 12B nonce,并把 nonce 前置到密文一起上线。**
> AAD 编码"域 + 位置"。统一规则消除一切"确定性 nonce 复用"的可能。

- **清单**:`out = nonce(12, 随机) ‖ AES-256-GCM-Seal(manifestKey, nonce, CBOR(manifest), aad=suiteByte)`。
  **每分享仅 Seal 一次**,进程生命周期内复用同一字节串——GCM tag 即清单指纹,
  续传 journal 与 suite 0x02 的 manifestTag 都锚定它。
- **块 i**:

  ```
  seg      = i / chunksPerSeg          // chunksPerSeg = SegmentBytes / chunkSize
  nonce    = 12 随机字节               // 每次 Seal 重新生成
  aad      = suiteByte ‖ u64_be(i)     // 域分隔 + 全局块号(位置绑定)
  blockCT  = nonce ‖ AES-256-GCM-Seal(segKey(seg), nonce, plaintext_i, aad)
                                       // 块密文 = 12B nonce + 明文 + 16B tag
  ```

  线传输时 `blockCT` 整体按 `MaxBlockPayload` 切帧,nonce 自然位于首帧,数据面帧布局无 nonce 字段。

**安全论证(摘)**:随机 96-bit nonce 生日界按 **Seal 次数**计(非块位置数);每 `segKey`
维护 Seal 计数器,达 `MaxSealsPerSegKey = 2³²`(NIST SP 800-38D 随机 IV 上限)即中止分享,
碰撞概率封顶 ≈ 2⁻³²。**块密文允许 Seal-once 字节复用**:重放同一 `nonce‖ct‖tag` 是密码学
空操作;缓存逐出后以新随机 nonce 重 Seal 同样安全。≤16 GiB 分享只有 1 段;寻址始终用
全局块号,分段纯属内部加密细节。

**接收端校验顺序(必须)**:重组块 `i` 全部帧 → 取首 12B nonce →
**GCM 解密并验证 tag(AAD=suiteByte‖u64_be(i))** → 落盘前路径校验 → 流式落盘。

**漂移处理**:随机 nonce 使"分享期间文件被改"密码学安全;`osfs.Source` 读后复核
`size/mtime`(读→复核→交付),变更即中止,负责一致性而非安全。

### 1.4 内容几何、选择与续传身份

- 打包流仅由认证清单的 entries 数组顺序与文件 size 前缀和派生。目录和空文件不占流;
  `offset`/`streamLen` 不是清单字段。
- `chunkSize` 必须是 `[1 KiB,4 MiB]` 内的 2 次幂。`streamLen ≤ 2⁴⁸`
  (256 TiB),`chunkCount = ceil(streamLen/chunkSize) ≤ 2²⁶`;一位/块的稠密状态上限
  为 8 MiB。接收端必须在分配 bitfield、请求状态、重组缓冲前验证这些界。
- 块需求以规范化的不可变 `ChunkSet` 表示:成员是已排序、互不相交且不相邻的半开
  区间 `[first,end)`。协议实现不得为了表达完整分享而预展开 eager 块号数组。
- selection 只编译一次为 `TransferPlan`:selected entries、selected bytes、`ChunkSet`、
  plan-local have-state、写入范围与 finalization 共用同一解释。selector `null` 表示全选,
  `[]` 表示空选;精确文件只选该文件,显式或隐式目录选子树,未知 selector 失败。
- `PlanID = SHA-256("windshare/v1 transfer-plan\x00" ‖ repeated(
  u64_be(path UTF-8 byte length) ‖ path UTF-8 bytes))`,其中 selected canonical paths
  按 UTF-8 字节排序。它标识选择的语义集合,不受 selector 顺序或重复影响。
- journal v2 同时绑定完整 sealed-manifest 指纹与 `PlanID`,记录 plan-local have-state
  及事务确切创建的 canonical file paths。选择变化不得复用旧 have-state;只有 journal
  记录的 owned path 可 reopen,文件名中的短指纹仅用于 namespace,不作为身份依据。

## 2. 线协议(§6.7)

### 2.1 信令面(经中转 WS,JSON;中转仅转发)

| 方向 | 消息 | 说明 |
|---|---|---|
| 发送端→中转 | `register {shareId, resumeTokenHash}` + 清单二进制帧 | 注册分享并上传加密清单(blob 紧随 register 走二进制帧);`resumeTokenHash = SHA-256(resumeToken)`,断线宽限期重注册凭 token 原像(§6.8) |
| 发送端↔中转 | `keepalive` | 维持在线(`KeepaliveInterval`) |
| 接收端→中转 | `join {shareId}` | 加入分享;`not_found` 时短窗指数退避重试 |
| 中转→接收端 | `manifest {sessionId}` + 清单二进制帧 / `not_found` | **唯一**的清单获取路径(无 HTTP 端点);`sessionId` 由中转在 join 时分配,后续 `signal`/`bye`/转发帧均携带 |
| 双向(经中转转发) | `signal {sessionId, kind, payload}` | WebRTC 协商(offer/answer/candidate) |
| 双向 | `bye {sessionId}` | 结束会话 |

- **WS 端点路径 = `/v1/ws/<shareId>`(M1 前向要求)**:消息内 `shareId` 须与路径一致
  (不一致即拒)。`v1` 是**线协议版本位**,覆盖信令 JSON 与数据面帧布局(`suiteByte`
  只版本化加密套件);health/config 通告支持的版本列表,错配即明确拒绝。shareId 入
  路径为区域内 L7 按 shareId 一致性哈希分流留位(§6.15)。
- **中转端点提示必须跨运行时唯一**:只接受绝对 `http`/`https`/`ws`/`wss` URL；
  authority 限 ASCII DNS LDH、规范 IPv4 或带方括号 IPv6。`xn--` 保留到协议拥有版本化
  IDNA 实现后再启用，避免 Go 与浏览器的 Unicode 表选择不同主机。首尾空白、parser
  auto-repair、dot segment 与两端序列化不同的原始 component spelling 一律在拨号前拒绝。
- **信令 JSON 有双重固定上限**:UTF-8 线文本不超过 64 KiB，且含顶层对象和忽略字段在内
  最多嵌套 64 层 object/array。该结构上限属于 v1 线协议，不继承 Go/JavaScript parser
  各自可变的递归深度。
- **清单 blob 走二进制 WS 帧**:清单帧以类型前缀与转发帧区分、紧随所属 JSON 消息
  (同连接 WS 有序);中转原样存储与回放、零解析。
- **控制流按 session 隔离**:`signal`/`bye`/session error/terminal 与普通转发数据使用
  有界 per-session control/data lanes;连接级消息单独排队。一个 session 溢出只能终止
  该 session,不能阻塞或关闭同一 WS 上的其他会话。

### 2.2 数据面(DataChannel 或中转转发,定长二进制,小端,传输无关)

> 术语:**明文块(chunk)**按全局块号寻址,是加密/校验/续传单位;其 AEAD 输出为
> **密封块**(块密文 `nonce‖ct‖tag`);线上传输单位是**传输帧**(≤ `MaxFrameSize`)。

| 帧 | 字段(固定小端布局) | 说明 |
|---|---|---|
| `REQUEST` | `type:u8, n:u32, indices:[u64×n]` | 请求若干块 |
| `BLOCK` | `type:u8, index:u64, seq:u32, flags:u8(last 位), len:u32, payload` | 整帧 `≤ MaxFrameSize`;`payload ≤ MaxBlockPayload = 65,518B` |
| `ERROR` | `type:u8, code:u16, msglen:u16, msg` | 会话终止错误;UTF-8 文本受帧上限约束 |

- 块密文 = `nonce(12) ‖ ct ‖ tag` 作为整体按 `MaxBlockPayload` 切帧(`seq` 递增、末帧置
  `last`);接收端按 `index` 重组全部帧后再解密。
- 数据面为 Go↔TS 双向编解码,以黄金向量逐字节校验(`testvectors/`)。
- 中转转发仅包裹 `sessionId` 做路由,不解析内层;`REQUEST` 非加密,块号访问模式对
  中转可见(已列入威胁模型已知泄漏)。
- `FrameChannel.SendTerminal(ctx, ERROR)` 是唯一带最终帧结束通道的操作。返回 nil 表示
  transport 已接受终止帧并保证它先于 `Recv` 关闭对端可见;普通 `Send(ERROR)` 后立即
  `Close` 不满足协议。终止一经接受,该 session 后续输入/输出一律丢弃。

relay WS 的外层二进制信封如下;`inner` 对 relay 完全 opaque:

| type | 布局 | 生命周期语义 |
|---|---|---|
| `0x01` manifest | `0x01 ‖ sealedManifest` | 紧随 register/manifest JSON |
| `0x02` forward | `0x02 ‖ sessionId(8B) ‖ inner` | 普通 session 帧 |
| `0x03` terminal-forward | `0x03 ‖ sessionId(8B) ‖ inner ERROR` | 接受后优先交付,写完成后 tombstone |

普通与终止转发信封的包裹头都固定为 9B。`MaxFrameSize=65,536B` 约束**内层整帧**,
故中转 WS 上的 routed binary message 上限为 `65,536+9=65,545B`;清单信封另受
`MaxManifestSize+1B` 约束。

terminal 会预占该 session 的控制容量并丢弃尚未发送的普通流量;已开始的单次写无法撤销,
但 terminal 必须成为下一次写。tombstone 阻止迟到的同 SID 帧重建 session;未知或已
tombstone 的 SID 只能得到临时 session 终止响应,不能借用连接级优先 lane。

## 3. 路径安全与跨平台规范化(§6.13,M1 必做)

路径策略标识固定为 **`windshare/path/v1-unicode-15.0.0`**。Go 与 TS 必须使用
Unicode 15.0.0 NFC 与 locale-independent full case fold;依赖升级不得静默改变策略。

**清单内路径规范(canonical)**:相对路径、`/` 分隔、有效 UTF-8、**NFC 归一化**、
无前导 `/`、盘符/UNC、`.`/`..`/空段、Unicode Cc/Cf(含 bidi/零宽格式字符)及
Windows 非法字符(`< > : " | ? * \`)。

**发送端(构建清单时)**:
- 逐路径规范化(含 NFC);无法安全表达者拒绝并报错。
- 符号链接与一切 reparse point(junction 等):M1 不跟随、不打包,跳过并告警。
- 重复及树冲突检测:`CollisionKey = NFC(fullFold(path))`;完整路径、祖先 segment 的
  折叠别名及 file-as-ancestor 均拒绝。

**接收端(落盘前,纵深防御,不信任清单)**:
- **清单结构校验(解密后、下载前)**:应用 §1.4 的完整几何界与相同路径策略。
- Sink 构造时打开并持有 `os.Root`;mkdir/open/write/mtime 全部使用 root-relative API。
  词法检查只提供明确错误与纵深,真实 containment 不依赖字符串 `safeJoin`;预植或竞态
  替换的 symlink/junction/reparse point 不得把操作导向根外。
- **Windows 保留名与非法字符**:`CON/PRN/AUX/NUL/COM1–9/LPT1–9`、
  `COM¹/²/³`、`LPT¹/²/³`、`CONIN$`/`CONOUT$`(含扩展名/大小写别名)、结尾空格或点
  及 `< > : " | ? * \` 全部拒绝;`:` 尤其不能成为 NTFS ADS。
- **路径长度**:按目标平台组件与总路径界预检,并保留底层 cause;超限稳定分类为
  `ErrPathTooLong`。
- **工具保留名**:仅输出根首段在 full fold 后以 `.wsresume` 开头时拒绝;嵌套同名
  不与 journal 冲突。
- **已存在同名文件**:没有全局 resume 模式。未 owned 的路径用原子
  `O_CREATE|O_EXCL`;`OwnershipLedger.RecordCreated` 持久化成功后写入才可成功。已 owned
  但缺失的路径报错,不得静默重建或把目录整体升级为可覆盖。
