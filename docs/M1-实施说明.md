# M1 实施说明(跨 Agent 接口契约)

> 目的:M1 各任务由多个 Agent 并行实现,本文钉死包边界、公开签名与常量归属,
> 避免集成期返工。**权威以 [`执行计划.md`](执行计划.md) 为准**(尤其 §6.3–§6.7、§8、
> 附录 B);本文只做"精确、可直接照抄"的落地契约。工程规约见根 `AGENTS.md`。

## 1. 模块与导入方向(§6.2)

- 模块①:`github.com/windshare/windshare/core`(纯核心);模块②:根
  `github.com/windshare/windshare`(CLI/transport/relay)。
- **root 可 import core;core 永不 import root**(也永不 import transport/relay 任何包)。
- **`relay/protocol` 纯类型零依赖**(仅 stdlib):`transport/relay` 客户端与 relay 服务端
  共同 import,不得反向依赖任何实现包。
- core 当前允许的外部直接依赖:`golang.org/x/text`(仅 `manifest`/`osfs`)、
  `fxamacker/cbor`(仅 `manifest`);新增依赖按合理性论证引入(AGENTS.md/§0③)。
- core 包间依赖:`chunk`/`manifest` 可 import `link`(取 suite 常量);`share` 组合
  `link`/`layout`/`chunk`/`manifest`/`internal/keyderiv`;`session` 不依赖加密包
  (它只见密文字节与块号)。
- **双模块发版**:根 go.mod 以版本 require core、**禁 replace**;M0 阶段为占位
  pseudo-version(`v0.0.0-00010101000000-000000000000`),本地构建一律经 `go.work`。
  发版两步:① 打 `core/vX.Y.Z` tag;② 根 go.mod 升 require 后再打根 tag。CI 的
  `GOWORK=off` job 在 core 首次发布前 `continue-on-error`(ci.yml 内有注释)。

## 2. `core/internal/keyderiv`(§6.3)

密钥派生**只在此包**发生(manifest/chunk 只接受派生好的 key 参数),由 `core/share`
门面接线。HKDF-SHA256,salt 一律为空,label 为精确 ASCII 字面字节:

```go
func ManifestKey(readSecret []byte) []byte              // HKDF(readSecret, "windshare/v1 manifest", 32)
func StreamKey(readSecret []byte) []byte                // HKDF(readSecret, "windshare/v1 stream",   32)
func SegKey(streamKey []byte, seg uint32) []byte        // HKDF(streamKey, "windshare/v1 seg"‖u32_be(seg), 32)
```

## 3. `core/link`(§6.3)

```go
type Link struct {
    Suite      byte     // suite 常量的语义家在本包:SuiteAESGCM byte = 0x01
    ReadSecret []byte   // 16B;链接中唯一秘密
    ShareID    string   // 9B 随机的 base64url;纯路由句柄,不入密钥
    Relays     []string // ?r= 多值,自始按列表解析(M1 仅用首个)
}
```

- URL 编解码:fragment = base64url 无填充(`suiteByte‖readSecret`);解析容错见 §6.10
  (纯密钥串 / 带 `#` / 误粘完整链接)。
- split-key:提供"完整链接 ↔ 裸链接 + 密钥串"的拆分与合并纯函数。

## 4. `core/layout`(§6.4,B14)

- 规范序:按规范化相对 path 的 UTF-8 字节序排序(排序不承担安全职责)。
- 几何全派生:按 entries 数组顺序对 size 做前缀和(目录与 size=0 文件不占流),
  `offset`/`streamLen` 不入清单;校验 `size ≥ 0`、前缀和 ≤ `MaxStreamBytes`。
- 双向映射:块号 → `[]FileRange`(path/off/n);文件集合 → 覆盖块区间(选择性下载)。
- 块几何:块 `i` 覆盖 `[i·chunkSize, min((i+1)·chunkSize, streamLen))`;chunkSize 须为 2 的幂。

## 5. `core/chunk`(§6.3,B1–B3/B12/B13)

```go
func NewCodec(streamKey []byte, chunkSize int64, rng io.Reader) (*Codec, error)
func (c *Codec) Seal(i uint64, plaintext []byte) ([]byte, error) // 输出 nonce‖ct‖tag
func (c *Codec) Open(i uint64, blockCT []byte) ([]byte, error)
```

- AAD = `suiteByte ‖ u64_be(i)`(i 为全局块号)。
- segKey 按 `seg = i / (SegmentBytes/chunkSize)` 选取,内部经 keyderiv 派生并缓存
  (注:Codec 持 streamKey,segKey 派生经 share 门面注入或直调 keyderiv——同 core 模块内,
  以 share 接线为准)。
- nonce:每次 Seal 从注入的 `rng` 取全新 12B,随密文前置输出。
- 每 segKey 维护 Seal 计数,达 `MaxSealsPerSegKey` 即返回错误中止分享(B12)。
- Open 按 suiteByte 分派解析、不硬编码密文尾部长度(为 0x02 的 `‖sig(64)` 留位,B13)。

## 6. `core/manifest`(§6.4,B4/B5/B14/B15)

```go
func Seal(manifestKey []byte, m *Manifest, rng io.Reader) ([]byte, error) // nonce‖cbor_gcm;aad=suiteByte
func Open(manifestKey []byte, sealed []byte) (*Manifest, error)
```

- **接受 `manifestKey []byte` 参数**——本包不做任何派生。
- CBOR:fxamacker/cbor,编码 Core Deterministic;解码严格模式拒非 canonical;
  解码先宽容探测 `v`,未知版本报"清单版本不支持,请升级"(B15)。
- 清单字段:`v`、`chunkSize`、`entries[]{path,size,mtime,isDir}`;**无 offset/streamLen**
  (B14);mtime = Unix epoch 毫秒(int64)。
- 每分享仅 Seal 一次、字节复用;GCM tag 即清单指纹(续传 journal 锚定)。
- 路径 canonical(NFC、非法字符、折叠碰撞)构建期在此校验(§6.13,x/text 仅此与 osfs)。

## 7. `core/session`(§6.6–§6.7)

接口**逐字**如下(消费侧定义;transport/* 与 osfs 在外层实现):

```go
// ── 传输层:可靠有序的双向"帧通道"。WebRTC DataChannel 与 relay WS 各实现一个。
type FrameChannel interface {
    Send(ctx context.Context, f Frame) error  // 背压:缓冲满则阻塞/返回错误
    Recv() <-chan Frame                        // 入站帧流
    State() ChannelState                       // Connecting / Open / Closed —— 供热切换感知
    Close() error
}

// ── 发送端:按块读明文(消费侧 = 发送会话)。读后复核该块仍匹配快照(size/mtime),变更即中止。
type BlockStore interface {
    ReadBlock(index uint64) (plaintext []byte, err error)
    BlockCount() uint64
}

// ── 接收端:落地已解密、已校验的明文(消费侧 = 接收会话)。落盘前路径校验见 §6.13。
type Sink interface {
    WriteBlock(index uint64, plaintext []byte) error
    Have() Bitfield            // 已持有块,用于断点续传
}
```

- **数据面帧二进制编解码归此包**(REQUEST/BLOCK/ERROR,固定小端布局,§6.7/协议规范 §2.2)。
- **帧类型字节(M0 钉死,金标向量随后锁定)**:`REQUEST=0x01`、`BLOCK=0x02`、`ERROR=0x03`;
  `BLOCK.flags` 的 last 位 = bit0。ERROR 的 code 值域 M1 实现时在此包定义具名常量。
- 线常量在此定义(root 侧 import 使用):`MaxFrameSize`、`InFlightWindow`、
  `P2PConnectTimeout`、`SmallShareBytes`。
- 调度器:需求集 = 选中块 − Have();每通道请求队列/在途/超时重试/源评分/帧重组/背压;
  **有序交付模式**(Sink 声明有序约束 → 按块号升序消费 + 有界重排缓冲 + 队头重试最高
  优先);热切换 = FrameChannel 入池/降权。

`core/share` 门面(Sharer/Receiver/FileSource/FileSink)签名照抄 §6.6 代码块;
`Options` 携带 chunkSize、可注入固定 readSecret 与随机源、shareId(B11)。

## 9. 测试与向量(§7,T0.3)

- **确定性命脉**:各包测试用注入的固定 RNG(喂 nonce)与固定 readSecret;严禁在测试
  路径里落 `crypto/rand` 直调(实现经 `Options`/`NewCodec` 的 rng 注入)。
- 黄金向量:`testvectors/`(信封 schema 与编码约定见其 README);Go 读取骨架 =
  `core/internal/testvec`,TS 读取骨架 = `web/test/vectors.ts`;实际向量由 T1.7
  `-update` 生成并提交。
- **web 工具链(硬性约束)**:React **19**(`react`/`react-dom` ^19);包管理器一律
  **pnpm**(`pnpm install` / `pnpm build` / `pnpm vitest run`),不用 npm/yarn;锁文件
  为 `web/pnpm-lock.yaml` 并入库,pnpm 版本钉在 `web/package.json` 的
  `packageManager` 字段(CI 经 pnpm/action-setup 读取,本地可 corepack enable)。
- CI:两 Go 模块**分别** vet/test/覆盖率 ≥70%(`.github/workflows/ci.yml`);web 侧
  pnpm install + build + vitest;`GOWORK=off` 构建校验守发版不变量。
- relay 侧未钉死项(T2.1 实现时在 `relay/protocol` 定义并回写本文):信令 JSON 字段的
  精确 schema、清单二进制帧类型前缀、转发帧 sessionId 包裹布局、ERROR code 值域。
