## 1. 模块与导入方向(§6.2)

- 模块①:`github.com/windshare/windshare/core`(纯核心);模块②:根
  `github.com/windshare/windshare`(CLI/transport/relay/connectivity/e2e)。
- **root 可 import core;core 永不 import root**(也永不 import transport/relay 任何包)。
- **`relay/protocol` 纯类型零依赖**(仅 stdlib):`transport/relay` 客户端与 relay 服务端
  共同 import,不得反向依赖任何实现包。
- core 当前允许的外部直接依赖:`golang.org/x/text`(仅 `manifest`/`osfs`)、
  `fxamacker/cbor`(仅 `manifest`);新增依赖按合理性论证引入(AGENTS.md/§0③)。
- core 包间依赖:`chunk`/`manifest` 可 import `link`(取 suite 常量);`share` 组合
  `link`/`layout`/`chunk`/`manifest`/`internal/keyderiv`;`session` 不依赖加密包
  (它只见密文字节与块号)。
- **双模块发版**:根 go.mod 以版本 require core、**禁 replace**;core 发布前为占位
  pseudo-version,本地构建一律经 `go.work`。发版两步:① 打 `core/vX.Y.Z` tag;
  ② 根 go.mod 升 require 后再打根 tag。CI 拆两个 job(E1):`gowork-off-core`
  与 `gowork-off-root` 均为硬性——G-EXT 于 2026-07-13 解决(`core/v0.1.0`
  发布于 `github.com/WindShare/windshare`,根 require 升 `v0.1.0`),root
  步骤的 `continue-on-error` 例外已移除。

## 2. `core/internal/keyderiv`(§6.3)

密钥派生**只在此包**发生(manifest/chunk 只接受派生好的 key 参数),由 `core/share`
门面接线。HKDF-SHA256,salt 一律为空,label 为精确 ASCII 字面字节:

```go
func ManifestKey(readSecret []byte) []byte              // HKDF(readSecret, "windshare/v1 manifest", 32)
func StreamKey(readSecret []byte) []byte                // HKDF(readSecret, "windshare/v1 stream",   32)
func SegKey(streamKey []byte, seg uint32) []byte        // HKDF(streamKey, "windshare/v1 seg"‖u32_be(seg), 32)
```

## 3. `core/link`(§6.3)

```go
type Link struct {
    Suite      byte     // suite 常量的语义家在本包:SuiteAESGCM byte = 0x01
    ReadSecret []byte   // 16B;链接中唯一秘密
    ShareID    string   // 9B 随机的 base64url;纯路由句柄,不入密钥
    Relays     []string // ?r= 多值,自始按列表解析(M1 仅用首个)
}
```

- URL 编解码:fragment = base64url 无填充(`suiteByte‖readSecret`);解析容错见 §6.10
  (纯密钥串 / 带 `#` / 误粘完整链接)。
- split-key:提供"完整链接 ↔ 裸链接 + 密钥串"的拆分与合并纯函数。

## 4. `core/layout`(§6.4,B14)

- 规范序:按规范化相对 path 的 UTF-8 字节序排序(排序不承担安全职责)。
- 几何全派生:按 entries 数组顺序对 size 做前缀和(目录与 size=0 文件不占流),
  `offset`/`streamLen` 不入清单;校验 `size ≥ 0`、前缀和 ≤ `MaxStreamBytes`。
- 几何资源界是协议的一部分:`MinChunkSize=1 KiB`、`MaxChunkSize=4 MiB`、
  `MaxChunkCount=2²⁶`、`MaxChunkStateBytes=8 MiB`、`MaxStreamBytes=2⁴⁸`
  (256 TiB)。必须在创建 bitfield、请求状态或块缓冲前完成校验。
- 块 `i` 覆盖 `[i·chunkSize, min((i+1)·chunkSize, streamLen))`;chunkSize
  必须为上述范围内的 2 次幂,块数用商/余数推导以避免加法溢出。
- `ChunkRange` 一律为半开区间 `[First,End)`;`ChunkSet` 是不可变、已排序、互不
  相交且不相邻的规范区间集。选择、并集、差集与迭代按区间数工作,不得把完整分享
  预展开为 eager 的逐块块号 slice。
- `Layout.Select` 一次产生认证清单顺序的 entry indices 与 `ChunkSet`:精确文件只选
  该文件,显式或隐式目录选子树,未知 selector 失败。

## 5. `core/chunk`(§6.3,B1–B3/B12/B13)

```go
func NewCodec(streamKey []byte, chunkSize int64, rng io.Reader) (*Codec, error)
func (c *Codec) Seal(i uint64, plaintext []byte) ([]byte, error) // 输出 nonce‖ct‖tag
func (c *Codec) Open(i uint64, blockCT []byte) ([]byte, error)
```

- AAD = `suiteByte ‖ u64_be(i)`(i 为全局块号)。
- segKey 按 `seg = i / (SegmentBytes/chunkSize)` 选取,内部经 keyderiv 派生并缓存
  (注:Codec 持 streamKey,segKey 派生经 share 门面注入或直调 keyderiv——同 core 模块内,
  以 share 接线为准)。
- nonce:每次 Seal 从注入的 `rng` 取全新 12B,随密文前置输出。
- 每 segKey 维护 Seal 计数,达 `MaxSealsPerSegKey` 即返回错误中止分享(B12)。
- Open 按 suiteByte 分派解析、不硬编码密文尾部长度(为 0x02 的 `‖sig(64)` 留位,B13)。

## 6. `core/manifest`(§6.4,B4/B5/B14/B15)

```go
func Seal(manifestKey []byte, m *Manifest, rng io.Reader) ([]byte, error) // nonce‖cbor_gcm;aad=suiteByte
func Open(manifestKey []byte, sealed []byte) (*Manifest, error)
```

- **接受 `manifestKey []byte` 参数**——本包不做任何派生。
- CBOR:fxamacker/cbor,编码 Core Deterministic;解码严格模式拒非 canonical;
  解码先宽容探测 `v`,未知版本报 `manifest: unsupported manifest version; upgrade required`(B15)。
- 清单字段:`v`、`chunkSize`、`entries[]{path,size,mtime,isDir}`;**无 offset/streamLen**
  (B14);mtime = Unix epoch 毫秒(int64)。
- 每分享仅 Seal 一次、字节复用;GCM tag 即清单指纹(续传 journal 锚定)。
- 路径 canonical(NFC、非法字符、折叠碰撞)构建期在此校验(§6.13,x/text 仅此与 osfs)。

### 6.1 `core/manifest.PathPolicy` 与 `core/osfs`(§6.13)

- `PathPolicyVersion = "windshare/path/v1-unicode-15.0.0"`;NFC、Unicode full
  case fold、Cc/Cf、Windows 保留名与 `.wsresume` 折叠前缀是一条不可拆的跨实现契约。
- `CurrentPathPolicy()` 返回不可变策略值,提供 `Canonicalize`、`Validate` 与
  `CollisionKey`;现有 `CanonicalPath`/`ValidatePath` 亦委托同一策略。
- 接收 Sink 在构造时 `os.OpenRoot`,之后 mkdir/open/write/mtime 全经 root-relative
  API;词法检查只负责产品错误与纵深,不再承担真实 containment。Sink 持有 OS
  root handle,具体调用方须在接收事务结束时调用 `Close`。
- 落盘所有权是精确能力,没有全局 resume/overwrite 开关:

```go
type OwnershipLedger interface {
    Owns(path string) bool
    RecordCreated(path string) error // 持久化成功后 WriteRange 才可返回成功
}
type SinkOptions struct {
    Ownership OwnershipLedger // nil = 新的进程内事务
}
```

未拥有路径首次打开固定用 `O_CREATE|O_EXCL`;只有 ledger 明确拥有的 canonical
file path 可 reopen。新建成功但 ownership 持久化失败时宁可留下安全孤儿并中止,
不得把全目录升级为可覆盖。

## 7. `core/session`(§6.6–§6.7)

接口**逐字**如下(消费侧定义;transport/* 与 osfs 在外层实现):

```go
// ── 传输层:可靠有序的双向"帧通道"。WebRTC DataChannel 与 relay WS 各实现一个。
type FrameChannel interface {
    Send(ctx context.Context, f Frame) error  // 背压:缓冲满则阻塞/返回错误
    SendTerminal(ctx context.Context, f Frame) error // 接受后必须先交付终止帧,再关闭 Recv
    Recv() <-chan Frame                        // 入站帧流
    State() ChannelState                       // Connecting / Open / Closed —— 供热切换感知
    Close() error
}

// ── 发送端:按块读明文(消费侧 = 发送会话)。读后复核该块仍匹配快照(size/mtime),变更即中止。
type BlockStore interface {
    ReadBlock(index uint64) (plaintext []byte, err error)
    BlockCount() uint64
}

// ── 接收端:落地已解密、已校验的明文(消费侧 = 接收会话)。落盘前路径校验见 §6.13。
type Sink interface {
    WriteBlock(index uint64, plaintext []byte) error
    Have() Bitfield            // 已持有块,用于断点续传
    DeliveryOrder() DeliveryOrder // DeliveryAnyOrder(随机写)/ DeliveryAscending(严格升序)
}
```

- **数据面帧二进制编解码归此包**(REQUEST/BLOCK/ERROR,固定小端布局,§6.7/协议规范 §2.2)。
- **帧类型字节(M0 钉死,金标向量随后锁定)**:`REQUEST=0x01`、`BLOCK=0x02`、`ERROR=0x03`;
  `BLOCK.flags` 的 last 位 = bit0。`ERROR` 是会话终止帧,必须走 `SendTerminal`,不能按
  普通 `Send` 后立即 `Close`;发送失败同时保留原始域错误与 `ErrTerminalDelivery`。
- 线常量在此定义(root 侧 import 使用):`MaxFrameSize`、`MaxBlockPayload`、
  `InFlightWindow`。**8 MiB/10 s 连接策略不在本包**:归接收端连接编排层所有
  (`web/src/connectivity` 的 `SMALL_SHARE_BYTES`/`P2P_CONNECT_TIMEOUT_MS`;合同 11)。
- 调度器:需求集 = `TransferPlan.Chunks().Subtract(Have())`;每通道请求队列/在途/超时
  重试/源评分/帧重组/背压;
  **有序交付模式**(`Sink.DeliveryOrder()` 声明有序约束,会话构造期消费 → 按块号升序
  消费 + 有界重排缓冲 + 队头重试最高优先);热切换 = FrameChannel 入池/降权。

### 7.1 `core/share.TransferPlan` 与 journal v2

- `Receiver.Plan(selectors)` 是选择的唯一入口:`nil` 表示完整选择,非 nil 的空 slice
  表示不选择任何内容;旧 receiver 级 `ChunksFor`/`Sink`/`Finalize(paths)` 不保留。
- `TransferPlan` 不可变地拥有 selected entries、selected bytes、`ChunkSet`、plan-local
  have-state、写入与无块条目的收尾物化。边界块整体认证,只把选中 file ranges 写盘。
- `PlanID = SHA-256("windshare/v1 transfer-plan\x00" ‖ repeated(
  u64_be(path UTF-8 byte length) ‖ path UTF-8 bytes))`;路径按 UTF-8 字节排序。
  selector 顺序/重复不影响结果,不同选择必须得到不同 ID。
- CLI journal v2 同时绑定完整 sealed-manifest 指纹与 `PlanID`,并持久化 plan-local
  have-state和该事务确切创建的 canonical file paths。旧选择的 have-state不得授权新
  选择的共享边界块;只有 recorded owned path 可通过 `OwnershipLedger` reopen。

`core/share` 门面还组合 `Sharer`/`Receiver`/`FileSource`/`FileSink`;`Options` 携带
chunkSize、可注入固定 readSecret 与随机源、shareId(B11)。

## 10. relay 线协议钉死项(T2.1 回写,权威实现 = `relay/protocol`)

- **WS 二进制帧类型前缀**:清单帧 = `0x01 ‖ sealedManifest`;普通转发帧 =
  `0x02 ‖ sessionId(8B raw) ‖ 内层数据面帧`;终止转发帧 =
  `0x03 ‖ sessionId(8B raw) ‖ opaque inner ERROR`。`SessionIDBytes = 8`;sessionId 在
  信令 JSON 中以 base64url 无填充表示,外层 payload 在向量中用标准 padded base64。
  两种 routed envelope 均增加 9B 包裹头;`MaxFrameSize=65,536B` 是内层整帧上限,
  因此 relay WS 上的 routed binary message 上限是 65,545B。
- relay 与客户端按 session 维护有界 control/data lanes。终止一经接受便预占控制容量、
  丢弃该 session 尚未发送的普通流量并在写完成后 tombstone;后续同 SID 流量丢弃,
  不得复活 session。连接级消息不借用 session lane,单 session 溢出不关闭其他会话。
- **信令 error code 为字符串值域**:`protocol_error / share_id_mismatch /
  share_id_collision / resume_rejected / manifest_too_large /
  manifest_budget_exceeded / rate_limited / frame_too_large / session_overflow /
  unknown_session / sender_gone`。
- **`registered` ack**:中转对 register 成功回 `registered` 确认;发送端**收到 ack
  后才打印链接**(不许 sleep 猜测)。
- **接收端断连**:中转向发送端合成 `bye{sessionId}`,发送端据此拆除对应发送会话。
- `MaxSignalingMessageBytes = 64 KiB`;`GET /v1/config` 响应 =
  `protocol.ServerConfig{protocolVersions, limits}`。
- 信令 JSON 消息集以 `relay/protocol` 的 Encode/Decode(双向校验)为准,
  `transport/relay`(T3.1)与 web(T5.2)均须经其对齐,不得自造 schema。
- `relay/signaling.DefaultMaxManifestSize` 直接复用 `core/manifest.MaxManifestSize`
  (单一权威;root import core,见 `relay/signaling/hub.go`)。

## 11. 连接编排层与资源准入(M1c/T3.3、B2 回写)

- **Go `connectivity`(根模块)**:在传输与会话之上组合按会话隔离的信令投影与
  `FrameChannel` 工厂——拥有 WebRTC 协商(offer/answer/ICE)、share 级发送端扇出
  (每接收端独立会话,慢接收端不得阻塞兄弟)、语义化 fatal 分类与指纹门禁的接收端
  恢复(rejoin 供给最新信令路由,健康的既有 P2P 数据路存活)。`transport/relay`、
  `transport/webrtc` 适配器保持零回退/零扇出/零调度策略。
- **`web/src/connectivity`**:用户手势前零 REQUEST/offer/candidate/ICE;选中字节
  ≤ `SMALL_SHARE_BYTES` 立即起中转、后台协商 P2P,更大选择最多等
  `P2P_CONNECT_TIMEOUT_MS` 再起中转、P2P 继续;后到的 Open P2P 通道**热加入同一个**
  `ReceiveSession` 调度池(不替换会话、不停用中转聚合);超时/取消/重试由注入时钟
  驱动测试。
- **`relay/admission`**:消费侧 lease 接口承担 register 资源与连接/join 决策,时钟与
  来源身份可注入;lease 熬过发送端重连宽限、分享实际回收才释放一次,拒绝/错误路径
  不泄漏、不重复释放。无效 Origin 配置使启动失败而非静默 deny-all。
